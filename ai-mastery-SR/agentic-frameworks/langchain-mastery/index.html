LangChain Basics

What is LangChain, and how is it used in NLP applications?

LangChain is a framework that helps you build LLM-powered applications in a structured way.

Instead of writing everything from scratch (prompt building, calling the LLM, handling outputs, adding retrieval, adding tools), LangChain gives you building blocks that you can connect like a pipeline.

Where it’s used in NLP apps:

  • Summarization: take long text → return a short summary
  • Question Answering: user asks → model answers
  • RAG (Retrieval-Augmented Generation): user asks → retrieve relevant docs → model answers using docs
  • Agents: model decides which tool to use (search, database, calculator) and then answers
  • Structured extraction: take messy text → extract JSON (name, email, date, etc.)

Simple mental model:

  • Prompt = instructions + variables
  • Model = produces output
  • Parser = converts output into the format you want
  • Chain (pipeline) = connects multiple steps together

What are the core components of LangChain?

Here are the most important pieces you’ll use often:

  1. Models (LLMs / Chat Models)
    • The “brain” that generates text.
    • Example: a chat model that answers questions.
  2. Prompts (PromptTemplate / ChatPromptTemplate)
    • A reusable template for instructions.
    • Supports variables like {text}, {topic}, etc.
  3. Output Parsers
    • Convert model output into the type you want:
      • plain string
      • JSON/dict
      • a validated structure
  4. Chains / Runnables
    • The pipeline that connects prompt → model → parser.
    • Example: summarization chain.
  5. Document Loaders + Text Splitters
    • Load text from PDFs/web/files and split into chunks.
  6. Embeddings + Vector Stores + Retrievers
    • Used for RAG to store and fetch relevant chunks.
  7. Agents + Tools
    • Used when the model needs to take actions (search, calculator, DB query).
  8. Callbacks
    • Used for logging/debugging/monitoring timing and inputs/outputs.

How do you configure an LLM in LangChain for text generation?

To configure an LLM in LangChain for text generation, you mainly need to:

  1. Choose a model

  2. Set parameters (like temperature, max tokens)

  3. Call the model with a prompt

Let’s understand this step by step in very simple terms.


Think of an LLM like a smart writing assistant.

Configuring it means:

  • Choosing which assistant to use

  • Deciding how creative it should be

  • Telling it how long the answer can be


First, you install and import the required packages.

Example (using OpenAI in LangChain):

from langchain_openai import ChatOpenAI

Now we configure the model.

llm = ChatOpenAI(
    model="gpt-4o-mini",
    temperature=0.7,
    max_tokens=200
)

Let’s understand each part.

model="gpt-4o-mini"
This selects which LLM you want to use.

Different models have different:

  • Cost

  • Speed

  • Intelligence level


temperature=0.7

This controls creativity.

  • 0.0 → Very predictable, factual

  • 1.0 → More creative, more random

Example:

If you ask:
“Write a tagline for a coffee shop”

Low temperature →
"Fresh coffee served daily."

High temperature →
"Wake up your soul with every sip."


max_tokens=200

This controls the maximum length of the output.

If you set it to 50, the model gives shorter answers.

If you set it to 500, it can write longer responses.

This also affects cost.

More tokens = more money.


Now let’s generate text.

response = llm.invoke("Write a short story about a robot learning emotions.")
print(response.content)

What happens here?

  • You pass a prompt.

  • The model generates text.

  • The output is stored in response.content.

That’s basic text generation.


Now let’s do it properly using a PromptTemplate.

This is useful when building real applications.

from langchain.prompts import PromptTemplate

template = PromptTemplate.from_template(

)

prompt = template.format(
    tone="funny",
    topic="machine learning"
)

response = llm.invoke(prompt)
print(response.content)

Why use PromptTemplate?

Because it lets you:

  • Reuse structured prompts

  • Insert variables dynamically

  • Build scalable applications


Now let’s look at a slightly more structured approach using LCEL (LangChain Expression Language).

prompt = ChatPromptTemplate.from_template(
    "Explain {topic} in simple terms."

Here’s what this means:

prompt | llm

This creates a chain.

Data flows like:

Input → Prompt → LLM → Output

This is clean and production-friendly.


Now let’s summarize the main configuration parameters you should know.

  1. model
    Selects which LLM to use.

  2. temperature
    Controls randomness and creativity.

  3. max_tokens
    Limits response length.

  4. top_p (optional)
    Controls diversity using probability sampling.

  5. frequency_penalty / presence_penalty
    Reduces repetition.

These are hyperparameters that control generation behavior.

How do you save and load a LangChain chain for reuse?

How do you save and load a LangChain chain for reuse?

Saving and loading a LangChain chain means:

You build it once
Then reuse it later
Without rebuilding everything again

Think of it like saving a trained workflow.

Instead of rewriting your logic every time, you store it and reload it when needed.


First, understand what a chain is.

A chain connects components like:

Input → Prompt → LLM → Output

For example:

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini")

prompt = ChatPromptTemplate.from_template(
    "Explain {topic} in simple terms."
)

chain = prompt | llm

Now we want to save this chain.


LangChain provides built-in serialization methods.

Most modern LangChain objects support:

  • .save()

  • load_chain()

Let’s see how.


Saving a chain

chain.save("my_chain.json")

What this does:

  • Converts the chain configuration into JSON

  • Stores model settings, prompt structure, and components

  • Saves it to a file

Important:

This saves the structure, not the actual LLM weights.

The model is not copied.
Only the configuration is stored.

Think of it like saving a recipe, not the food.


Loading a chain

from langchain.chains import load_chain

loaded_chain = load_chain("my_chain.json")

Now you can use it:

response = loaded_chain.invoke({"topic": "AI"})
print(response.content)

It works the same way as before.


Now let’s understand something important.

Modern LangChain (LCEL style using |) is based on runnables.

For those, you can use .to_json() and .from_json().

Example:

import json

# Save
with open("chain.json", "w") as f:
    json.dump(chain.to_json(), f)

# Load
from langchain_core.runnables import RunnableSerializable

with open("chain.json") as f:
    data = json.load(f)

loaded_chain = RunnableSerializable.from_json(data)

This works for LCEL pipelines.


When should you save a chain?

  • When deploying to production

  • When sharing workflows with a team

  • When versioning your AI pipelines

  • When reusing the same structured workflow


Now let’s understand limitations.

  1. API keys are NOT saved
    You must set them again in the environment.

  2. Custom Python logic may not serialize automatically
    You may need to rebuild certain parts.

  3. Model weights are not stored
    Only configuration is saved.

Write a function to create a simple LangChain LLMChain for text summarization.

Modern LangChain encourages building pipelines like: prompt → model → parser.

from langchain.chat_models import init_chat_model
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

def create_summarization_chain():
    model = init_chat_model("openai:gpt-4.1-mini", temperature=0.2)

    prompt = ChatPromptTemplate.from_messages([
        ("system", "You are a helpful assistant. Summarize clearly in 5 bullet points."),
        ("human", "{text}")
    ])

    chain = prompt | model | StrOutputParser()
    return chain

Code explanation:

  • ChatPromptTemplate.from_messages(...) builds a chat-style prompt:
    • system message sets rules
    • human message contains the variable {text}
  • StrOutputParser() ensures the final output is returned as a plain string.
  • prompt | model | parser is a pipeline:
    1. format prompt
    2. call model
    3. parse output
  • The function returns a reusable chain.

How you would use it:

chain = create_summarization_chain()
summary = chain.invoke({"text": "Very long article text here..."})
print(summary)

How do you handle API rate limits in LangChain?

API rate limits mean the provider only allows a certain number of requests per minute (or tokens per minute). If you exceed that, you’ll get errors like 429 Too Many Requests.

In LangChain, the safest approach is to retry with backoff, and also control concurrency so you don’t spam the API.

A common pattern is:

  • Catch rate limit errors

  • Wait a bit (backoff)

  • Retry a few times

  • If it still fails, return a graceful error or fallback model

Here’s a simple retry approach using tenacity (very common in Python):



from tenacity import retry, wait_exponential, stop_after_attempt, retry_if_exception_type

# Replace with the provider's rate limit exception if available
class RateLimitError(Exception):
    pass

@retry(
    wait=wait_exponential(multiplier=1, min=1, max=20),
    stop=stop_after_attempt(5),
    retry=retry_if_exception_type(RateLimitError),
)
def safe_invoke(chain_or_llm, inp):
    return chain_or_llm.invoke(inp)

What this does:

  • If a rate limit error happens, it retries.

  • It waits longer each time (1s → 2s → 4s … up to 20s).

  • It stops after 5 attempts.

You also reduce rate limit issues by:

  • Using smaller max_tokens

  • Batching calls (if supported)

  • Limiting parallel requests (for example, don’t run 200 calls at once

How do you use LangChain to switch between different LLM providers?

The easiest way is to use a factory function that returns the right LangChain model based on a config value.

That way, your chain stays the same, and only the provider changes.



from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic

def get_llm(provider: str):
    provider = provider.lower()

    if provider == "openai":
        return ChatOpenAI(model="gpt-4o-mini", temperature=0.2)

    if provider == "anthropic":
        return ChatAnthropic(model="claude-3-5-sonnet-latest", temperature=0.2)

    raise ValueError(f"Unknown provider: {provider}")

Now your code can do:



llm = get_llm("openai")      # or "anthropic"
response = llm.invoke("Explain transformers simply.")
print(response.content)

Why this is good:

  • You don’t rewrite chains

  • You can A/B test providers

  • You can fallback when one provider is down or rate-limited

Implement a custom LangChain LLM wrapper for a local model.

If you have a local model (like a Hugging Face model running on your machine), you can wrap it so LangChain can use it like any other LLM.

Below is a simple wrapper using transformers text-generation pipeline.



from typing import Any, List, Optional
from langchain_core.language_models.llms import LLM

class LocalHFLLM(LLM):
    def __init__(self, pipeline, max_new_tokens: int = 200, temperature: float = 0.7):
        super().__init__()
        self.pipeline = pipeline
        self.max_new_tokens = max_new_tokens
        self.temperature = temperature

    @property
    def _llm_type(self) -> str:
        return "local_hf_pipeline"

    def _call(self, prompt: str, stop: Optional[List[str]] = None, **kwargs: Any) -> str:
        out = self.pipeline(
            prompt,
            max_new_tokens=self.max_new_tokens,
            temperature=self.temperature,
            do_sample=self.temperature > 0,
        )[0]["generated_text"]

        # Keep it simple: return only the continuation after the prompt
        text = out[len(prompt):]

        # Basic stop handling
        if stop:
            for s in stop:
                if s in text:
                    text = text.split(s)[0]
        return text

Usage:



from transformers import pipeline

gen = pipeline("text-generation", model="gpt2")  # example local model
llm = LocalHFLLM(gen)

print(llm.invoke("Write a short line about AI: "))

What this gives you:

  • Your local model behaves like a LangChain LLM

  • You can plug it into chains, tools, and prompts

Write a function to validate LangChain LLM outputs.

A common validation approach is:

  • force the model to output JSON
  • parse JSON
  • check required keys exist
import json
from typing import Any, Dict, List

def validate_json_output(output_text: str, required_keys: List[str]) -> Dict[str, Any]:
    data = json.loads(output_text)  # throws error if JSON is invalid

    missing = [k for k in required_keys if k not in data]
    if missing:
        raise ValueError(f"Missing required keys: {missing}")

    return data

Code explanation:

  • json.loads(...) converts text → Python dict.
  • If output isn’t valid JSON, it fails immediately (good for strict systems).
  • We check if the required keys exist; if not, raise an error.
  • If everything is OK, return the parsed dict.

def translate_text(text: str, target_language: str) -> str:

Implement a LangChain callback for logging execution time. 
import time
from typing import Any, Dict, Optional
from langchain_core.callbacks.base import BaseCallbackHandler

class TimingCallback(BaseCallbackHandler):
    def __init__(self):
        self.start_time: Optional[float] = None

    def on_chain_start(self, serialized: Dict[str, Any], inputs: Dict[str, Any], **kwargs: Any) -> None:
        self.start_time = time.perf_counter()

    def on_chain_end(self, outputs: Dict[str, Any], **kwargs: Any) -> None:
        if self.start_time is None:
            return
        elapsed = time.perf_counter() - self.start_time
        print(f"Chain execution time: {elapsed:.3f} seconds")

Code explanation:

  • BaseCallbackHandler is the base class for callbacks.
  • on_chain_start(...) runs when a chain begins.
    • we store the start time
  • on_chain_end(...) runs when the chain finishes.
    • we subtract and print elapsed time
  • time.perf_counter() is good for precise timing.

How to attach it (conceptually):

  • You pass the callback handler to the model/chain config when invoking, so it logs timing during execution.

What is a chain in LangChain, and how is it used in NLP?

A chain in LangChain is a workflow (pipeline) that connects multiple steps to solve an NLP task end-to-end.

A chain typically includes:

  • Prompt (what you ask the model + placeholders)
  • Model (LLM/chat model that generates text)
  • Parser (converts raw output into a clean format like string/JSON)
  • Optional extras: retriever, tools, memory, validators

How chains help in NLP:

  • Summarization: text → prompt → model → summary
  • Question answering: context + question → prompt → model → answer
  • RAG: question → retrieve docs → prompt with docs → model → grounded answer
  • Extraction: messy text → prompt → model → structured JSON

Beginner-friendly analogy:

A chain is like a cooking recipe:

  1. gather ingredients (inputs)
  2. follow steps (prompt + model calls)
  3. plate the dish nicely (parser/formatting)

How do you create a sequential chain in LangChain?

A sequential chain means: Step 1 output becomes Step 2 input, then Step 3, and so on.

Here’s an example: outline → expand into paragraph

from langchain.chat_models import init_chat_model
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser

def create_sequential_chain():
    model = init_chat_model("openai:gpt-4.1-mini", temperature=0.2)
    parser = StrOutputParser()

    # Step 1: Create an outline
    outline_chain = (
        PromptTemplate.from_template("Create a 3-bullet outline on: {topic}")
        | model
        | parser
    )

    # Step 2: Expand the outline into a paragraph
    expand_chain = (
        PromptTemplate.from_template("Expand this outline into a short paragraph:\\n\\n{outline}")
        | model
        | parser
    )

    # Combine: output of outline_chain becomes input to expand_chain
    chain = outline_chain | (lambda outline: {"outline": outline}) | expand_chain
    return chain

Code explanation (simple):

  • outline_chain produces a string outline.
  • expand_chain expects an input variable called {outline}.
  • (lambda outline: {"outline": outline}) converts the string into the dict format the next prompt expects.
  • The | operator connects them in order.

Usage:

chain = create_sequential_chain()
print(chain.invoke({"topic": "RAG for beginners"}))

What is the difference between LLMChain and SequentialChain?

What is the difference between LLMChain and SequentialChain?

To understand the difference, first understand what a “chain” means in LangChain.

A chain is simply:

Input → Some processing → Output

Now let’s look at both types clearly.


LLMChain

An LLMChain is the simplest type of chain.

It connects:

Prompt → LLM → Output

That’s it.

It is used when you only need one LLM call.

For example:

You give a topic → It generates an explanation.

Here’s a simple example:

from langchain.chains import LLMChain
from langchain_openai import ChatOpenAI
from langchain.prompts import PromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini")

prompt = PromptTemplate(
    input_variables=["topic"],
    template="Explain {topic} in simple terms."
)

chain = LLMChain(llm=llm, prompt=prompt)

result = chain.run("neural networks")
print(result)

What happens here?

  • Input: "neural networks"

  • Prompt formats it

  • LLM generates output

  • Final answer is returned

Only one LLM call.

Simple.


Now let’s look at SequentialChain.


SequentialChain

A SequentialChain connects multiple chains in sequence.

It means:

Output of Chain 1 → Input of Chain 2 → Input of Chain 3 → Final Output

So instead of one step, you have multiple steps.

Example:

Step 1: Generate a story idea
Step 2: Expand it into a paragraph
Step 3: Summarize the paragraph

Each step may call an LLM.

Here’s a simplified example:

from langchain.chains import LLMChain, SequentialChain
from langchain_openai import ChatOpenAI
from langchain.prompts import PromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini")

prompt1 = PromptTemplate(
    input_variables=["topic"],
    template="Generate a short story idea about {topic}."
)

chain1 = LLMChain(llm=llm, prompt=prompt1, output_key="idea")

prompt2 = PromptTemplate(
    input_variables=["idea"],
    template="Write a short paragraph based on this idea: {idea}"
)

chain2 = LLMChain(llm=llm, prompt=prompt2, output_key="story")

overall_chain = SequentialChain(
    chains=[chain1, chain2],
    input_variables=["topic"],
    output_variables=["story"]
)

result = overall_chain.run("space exploration")
print(result)

What happens here?

  1. First LLM generates an idea.

  2. That idea becomes input to the second LLM.

  3. Second LLM writes the full story.

  4. Final output is returned.

This is multi-step processing.


Key Differences

LLMChain:

  • Single prompt

  • Single LLM call

  • One-step process

  • Simple tasks

SequentialChain:

  • Multiple chains

  • Multiple LLM calls

  • Multi-step logic

  • Complex workflows


Think about it like this:

LLMChain = One question → One answer

SequentialChain = Step-by-step reasoning pipeline

If you only need one transformation, use LLMChain.

If your task needs multiple transformations, use SequentialChain.


In practical applications:

Use LLMChain for:

  • Text generation

  • Summarization

  • Simple classification

Use SequentialChain for:

  • Multi-step content generation

  • Data cleaning → reasoning → formatting

  • Complex AI workflows

How do you pass inputs to a LangChain chain?

How do you pass inputs to a LangChain chain?

Passing inputs to a LangChain chain means giving it the data it needs to generate a response.

In simple terms:

You create a chain with placeholders.
Then you provide values for those placeholders when calling the chain.


First, understand that most chains expect input as a dictionary.

The keys of the dictionary must match the variable names in the prompt.

Let’s see a basic example.

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini")

prompt = ChatPromptTemplate.from_template(
    "Explain {topic} in simple terms."
)

chain = prompt | llm

Here, the prompt contains {topic}.

That means the chain expects an input variable named "topic".

Now we pass input like this:

response = chain.invoke({"topic": "machine learning"})
print(response.content)

Important:

The key "topic" must match the placeholder {topic}.

If it does not match, the chain will raise an error.


Now let’s look at multiple inputs.

prompt = ChatPromptTemplate.from_template(
    "Explain {topic} in a {tone} tone."
)

chain = prompt | llm

response = chain.invoke({
    "topic": "neural networks",
    "tone": "funny"
})

print(response.content)

Here we pass two values:

  • topic

  • tone

Both must be provided.


There are different ways to pass inputs depending on the situation.

1. Using .invoke() (Most Common)

chain.invoke({"topic": "AI"})

This is recommended for modern LangChain (LCEL).


2. Using .run() (Older style)

If using LLMChain:

result = chain.run("AI")

This works only when there is one input variable.

If there are multiple inputs, you must use a dictionary:

result = chain.run({"topic": "AI", "tone": "simple"})

3. Batch inputs

You can process multiple inputs at once:

responses = chain.batch([
    {"topic": "AI"},
    {"topic": "blockchain"},
    {"topic": "cloud computing"}
])

This is useful for:

  • Bulk processing

  • Reducing latency

  • Cost optimization


4. Streaming input

If streaming is enabled:

for chunk in chain.stream({"topic": "AI"}):
    print(chunk.content, end="")

This returns tokens as they are generated.


Now let’s understand the key rule clearly.

The prompt defines required input variables.
You must pass those variables as a dictionary when calling the chain.

Input variables → Dictionary keys
Values → Dictionary values

What is the role of output parsers in LangChain chains?

Output parsers in LangChain are used to convert the LLM’s raw text into a clean, structured format that your program can reliably use.

Write a function to chain text generation and parsing.

This is a very common production requirement:

  • Generate output in a known format (usually JSON)

  • Parse it into a Python object

  • Fail fast if parsing is invalid

Below is a function that:

  • Forces JSON output

  • Parses it into a Python dict



import json
from typing import Any, Dict, List
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

class ParseError(Exception):
    pass

def build_generation_with_json_parsing_chain(
    required_keys: List[str],
    model: str = "gpt-4o-mini",
):
    llm = ChatOpenAI(model=model, temperature=0.0, max_tokens=300)

    prompt = ChatPromptTemplate.from_template(
        "Return ONLY valid JSON.\n"
        "JSON must contain keys: {keys}\n\n"
        "Task: {task}\n"
        "Input: {text}\n"
    )

    def parse_json(output) -> Dict[str, Any]:
        raw = output.content if hasattr(output, "content") else str(output)
        try:
            data = json.loads(raw)
        except Exception as e:
            raise ParseError(f"Invalid JSON output: {e}\nRaw:\n{raw}")

        missing = [k for k in required_keys if k not in data]
        if missing:
            raise ParseError(f"Missing keys: {missing}\nRaw:\n{raw}")

        return data

    chain = prompt | llm
    return chain, parse_json

Usage:



chain, parser = build_generation_with_json_parsing_chain(["summary", "sentiment"])

out = chain.invoke({
    "keys": ["summary", "sentiment"],
    "task": "Summarize and label sentiment as positive/negative/neutral",
    "text": "I love LangChain because it makes LLM apps easier."
})

data = parser(out)
print(data)

This pattern is strong because:

  • You control the output format

  • You avoid fragile string handling

  • You make your pipeline safer for production

  • retries (exponential backoff)

  • fallback model/provider

How do you implement a chain with dynamic routing in LangChain?

Dynamic routing means: based on the input, you choose which sub-chain should run.

This is useful when you have different “skills” or workflows, like:

  • If the user asks a math question → use a math-focused prompt

  • If the user asks for writing → use a writing-focused prompt

  • If the user asks for coding → use a code-focused prompt

In LangChain (LCEL), the clean way is to:

  1. Create a router that decides a route label (like "math", "code", "general")

  2. Use RunnableBranch (or a similar branching mechanism) to execute the right chain

Here’s a simple, practical implementation.



from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnableLambda, RunnableBranch

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.2)

# 1) Define specialized prompts/chains
math_chain = ChatPromptTemplate.from_template(
    "You are a math assistant. Solve step-by-step:\n{query}"
) | llm

code_chain = ChatPromptTemplate.from_template(
    "You are a coding assistant. Provide Python code and a short explanation:\n{query}"
) | llm

general_chain = ChatPromptTemplate.from_template(
    "You are a helpful assistant. Answer clearly:\n{query}"
) | llm

# 2) Router function (simple rule-based routing)
def route_fn(inputs: dict) -> str:
    q = inputs["query"].lower()
    if any(k in q for k in ["derive", "integral", "probability", "solve", "equation"]):
        return "math"
    if any(k in q for k in ["python", "java", "bug", "code", "function", "api"]):
        return "code"
    return "general"

router = RunnableLambda(lambda x: {**x, "route": route_fn(x)})

# 3) Branch based on route
routed_chain = router | RunnableBranch(
    (lambda x: x["route"] == "math", math_chain),
    (lambda x: x["route"] == "code", code_chain),
    general_chain,  # default
)

# Usage
resp = routed_chain.invoke({"query": "Write a Python function to reverse a string."})
print(resp.content)

Why this works well:

  • Routing logic is separated from generation logic

  • You can add more routes without changing the rest

  • You can swap rule-based routing with an LLM-based router later

If you want LLM-based routing, you typically do:

  • A “classifier prompt” that outputs a route label (math/code/general)

  • Then branch using the label

Implement a chain to handle multi-step reasoning in LangChain.

Multi-step reasoning means you break a task into steps, and each step feeds the next step.

A common pattern is:

  1. Create a plan or outline

  2. Execute the plan

  3. Produce a final formatted answer

In LangChain LCEL, you can do this by composing multiple prompts.



from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnableLambda

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.2)

plan_prompt = ChatPromptTemplate.from_template(
    "Create a short step-by-step plan to solve this task:\n{task}\nReturn only the plan."
)

solve_prompt = ChatPromptTemplate.from_template(
    "Use this plan to solve the task.\n\nTask: {task}\nPlan:\n{plan}\n\nSolution:"
)

final_prompt = ChatPromptTemplate.from_template(
    "Rewrite the solution into a concise final answer with bullet points if helpful.\n\n{solution}"
)

plan_chain = plan_prompt | llm
solve_chain = solve_prompt | llm
final_chain = final_prompt | llm

# Glue: take outputs from one step and pass them to the next step
multi_step_chain = (
    {
        "task": lambda x: x["task"],
        "plan": plan_chain,  # plan_chain receives {"task": ...}
    }
    | RunnableLambda(lambda x: {"task": x["task"], "plan": x["plan"].content})
    | solve_chain
    | RunnableLambda(lambda x: {"solution": x.content})
    | final_chain
)

resp = multi_step_chain.invoke({"task": "Explain how overfitting happens and how to prevent it."})
print(resp.content)

What’s happening:

  • Step 1 generates a plan (plan_chain)

  • Step 2 uses that plan to generate a solution (solve_chain)

  • Step 3 formats it into a final answer (final_chain)

Important note:

  • For sensitive tasks (like factual answers), multi-step reasoning is usually paired with retrieval (RAG) so the reasoning is grounded in sources.

What is a LangChain agent, and how is it used in NLP?

A LangChain agent is an LLM-driven component that can decide what to do next instead of following a fixed pipeline.

A normal chain is like:

  • prompt → LLM → output (fixed steps)

An agent is like:

  • think → “I need a tool” → use tool → look at result → think again → answer

Agents are used in NLP when tasks require:

  • searching (web / docs / vector DB)
  • calculations
  • database queries
  • multi-step reasoning
  • tool orchestration (use 2–3 tools before final answer)

Example (beginner-friendly):

User asks: “Summarize the latest policy update from our wiki and list 3 key changes.”

An agent can:

  1. search wiki tool
  2. fetch policy page
  3. summarize
  4. generate final bullet points

How do you create a tool in LangChain for an agent?

A tool is simply a function with:

  • a clear name
  • a clear input
  • a clear output
  • a description telling the agent when to use it

Here’s a simple custom tool:

from langchain_core.tools import tool

@tool
def word_count(text: str) -> str:
    """Count words in the given text and return the count."""
    return str(len(text.split()))

Code explanation:

  • @tool wraps your function so the agent can call it.
  • The docstring is important because the agent reads it to decide whether to use the tool.
  • Tools should return simple outputs (string/JSON) to keep agent behavior predictable.

What is the ReAct framework in LangChain agents?

ReAct means Reason + Act.

In ReAct-style agents, the model alternates between:

  • Reasoning: deciding what to do
  • Acting: calling a tool
  • Observing: reading the tool result
  • repeating until it can answer

Example flow:

  • Thought: “I need the current USD→INR rate.”
  • Action: call currency tool
  • Observation: “1 USD = 83.1 INR”
  • Thought: “Now I can compute the conversion.”
  • Final Answer: …

ReAct makes agents more reliable because they don’t “guess”—they fetch facts using tools.

What is the role of the agent executor in LangChain?

What is the role of the Agent Executor in LangChain?

The Agent Executor in LangChain is responsible for running an agent step-by-step until a final answer is produced.

In simple words:

The agent decides what to do.
The Agent Executor actually does it.


Let’s break this down clearly.

In LangChain, an agent is an LLM-powered decision maker.

It looks at a user question and decides:

  • Should I call a tool?

  • Which tool should I call?

  • What input should I pass?

  • Should I answer directly?

But the agent itself does not execute tools.

That’s where the Agent Executor comes in.


Think of it like this:

  • The agent = the brain (decision-making)

  • The tools = hands (do actions)

  • The Agent Executor = the manager who coordinates everything

The executor:

  1. Sends the user input to the agent.

  2. Gets back a decision (for example: “Call calculator tool”).

  3. Executes the tool.

  4. Sends the tool result back to the agent.

  5. Repeats this loop until the agent gives a final answer.

This loop is very important.

Without it, the agent cannot actually use tools.


Let’s see a simple flow.

User asks:
“What is the square root of 144 plus 10?”

Step 1:
Agent thinks → “I should use the calculator tool.”

Step 2:
Agent Executor runs the calculator tool with input “sqrt(144)”.

Step 3:
Tool returns “12”.

Step 4:
Executor sends result back to agent.

Step 5:
Agent thinks → “12 + 10 = 22. Final answer: 22.”

Then the executor stops.


Technically, the Agent Executor:

  • Manages the reasoning loop

  • Executes tools

  • Tracks intermediate steps

  • Handles errors

  • Stops when a final answer is produced

  • Prevents infinite loops (via max iterations)


Here’s a simplified example:

from langchain.agents import initialize_agent
from langchain_openai import ChatOpenAI
from langchain.tools import Tool

llm = ChatOpenAI(model="gpt-4o-mini")

def multiply(a: int, b: int) -> int:
    return a * b

tools = [
    Tool(
        name="Multiplier",
        func=lambda x: str(eval(x)),
        description="Multiply numbers"
    )
]

agent_executor = initialize_agent(
    tools=tools,
    llm=llm,
    agent="zero-shot-react-description",
    verbose=True
)

result = agent_executor.run("What is 5 * 6?")
print(result)

Here:

  • The agent decides to use the “Multiplier” tool.

  • The Agent Executor actually runs the tool.

  • It feeds the result back to the agent.

  • It returns the final answer.


Why is Agent Executor important?

Because agents are not single LLM calls.

They are multi-step reasoning systems.

The executor enables:

  • Tool usage

  • Multi-step reasoning

  • External API calls

  • Database queries

  • Controlled stopping conditions

Without the executor, the agent would just output text saying:
“I should call a calculator.”

But it would never actually call it.


In production systems, the Agent Executor also helps with:

  • Limiting maximum steps

  • Logging intermediate reasoning

  • Handling tool errors

  • Monitoring performance

How do you handle tool failures in LangChain agents?

Tool failures happen due to:

  • network issues
  • timeouts
  • invalid tool inputs
  • API errors
  • empty results

Best practices:

  • wrap tool logic in try/except
  • return a clean error message instead of crashing
  • let the agent decide the next step (“try again”, “use alternative tool”, “respond with limitation”)

Example:

from langchain_core.tools import tool

@tool
def safe_tool(input_text: str) -> str:
    """A tool that demonstrates safe error handling."""
    try:
        if not input_text.strip():
            return "ERROR: Empty input provided."
        return input_text.upper()
    except Exception as e:
        return f"ERROR: Tool failed with {type(e).__name__}: {e}"

Why this helps:

Agents behave better when tools return predictable error strings instead of throwing exceptions.

from langchain.chat_models import init_chat_model
from langchain_core.tools import tool
from langchain.agents import create_react_agent, AgentExecutor

@tool
def word_count(text: str) -> str:
    """Return the number of words in the text."""
    return str(len(text.split()))

@tool
def make_bullets(text: str) -> str:
    """Convert the text into 3 short bullet points."""
    lines = [s.strip() for s in text.split(".") if s.strip()]
    bullets = lines[:3]
    return "\\n".join([f"- {b}" for b in bullets]) if bullets else "- (no content)"

def create_agent_with_tools():
    model = init_chat_model("openai:gpt-4.1-mini", temperature=0.2)
    tools = [word_count, make_bullets]

    agent = create_react_agent(llm=model, tools=tools)
    executor = AgentExecutor(agent=agent, tools=tools, verbose=True)
    return executor

Code explanation:

  • We define tools using @tool.
  • create_react_agent(...) creates an agent that can reason and call tools.
  • AgentExecutor(...) actually runs the loop and returns a final answer.
  • verbose=True prints reasoning/tool steps (useful for learning and debugging).

How do you implement a LangChain agent with memory?

An agent with memory means the agent can remember what was said earlier and use it in later steps.

In LangChain, the clean way to do this is:

  • Store chat history (messages)

  • Automatically inject that history into the agent each time

  • Keep histories separate per user/session

The most common modern pattern is to use:

  • An agent executor (the thing that runs the agent)

  • RunnableWithMessageHistory (to add memory)


Here is a practical implementation using an OpenAI chat model and an agent that can use tools.

from langchain_openai import ChatOpenAI
from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_core.tools import tool
from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# 1) LLM
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.2)

# 2) Tools
@tool
def add(a: int, b: int) -> int:
    """Add two integers."""
    return a + b

tools = [add]

# 3) Agent prompt includes a placeholder for memory
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant. Use tools when needed."),
    MessagesPlaceholder("history"),
    ("human", "{input}"),
    MessagesPlaceholder("agent_scratchpad"),
])

# 4) Create agent and executor
agent = create_tool_calling_agent(llm, tools, prompt)
agent_executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 5) Memory store (per session/user)
_store = {}

def get_history(session_id: str):
    if session_id not in _store:
        _store[session_id] = InMemoryChatMessageHistory()
    return _store[session_id]

# 6) Wrap executor with message history
agent_with_memory = RunnableWithMessageHistory(
    agent_executor,
    get_history,
    input_messages_key="input",
    history_messages_key="history",
)

Memory in LangChain is a mechanism to store and reuse past information (usually chat history or extracted facts) so your application can respond with continuity.

In NLP apps, memory is used to:

  • keep a conversation coherent across turns (“as I said earlier…”)
  • remember user preferences (“use bullet points”, “my name is X”)

How do you add memory to a LangChain chain?

How do you add memory to a LangChain chain?

Adding memory to a LangChain chain means the chain can use previous conversation turns when generating the next response.

So instead of answering like it’s the first message every time, it can remember what was said earlier.


The clean modern way to do this in LangChain is:

  1. Put a MessagesPlaceholder in your prompt for chat history

  2. Store messages in a message history object

  3. Wrap your chain with RunnableWithMessageHistory so history is automatically added and updated


Here is a simple working example.

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

# 1) Model
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.2)

# 2) Prompt with a history placeholder
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    MessagesPlaceholder("history"),
    ("human", "{input}")
])

# 3) Base chain
chain = prompt | llm

# 4) Store chat history per session
_store = {}

def get_history(session_id: str):
    if session_id not in _store:
        _store[session_id] = InMemoryChatMessageHistory()
    return _store[session_id]

# 5) Wrap chain with message history (memory)
chain_with_memory = RunnableWithMessageHistory(
    chain,
    get_history,
    input_messages_key="input",
    history_messages_key="history",
)

# ---- Usage ----
resp1 = chain_with_memory.invoke(
    {"input": "My name is Sana. Remember it."},
    config={"configurable": {"session_id": "user-1"}}
)
print(resp1.content)

resp2 = chain_with_memory.invoke(
    {"input": "What is my name?"},
    config={"configurable": {"session_id": "user-1"}}
)
print(resp2.content)

What’s happening here?

  • MessagesPlaceholder("history") is where old messages will be inserted.

  • InMemoryChatMessageHistory stores the conversation.

  • RunnableWithMessageHistory automatically:

    • Loads the old messages for that session

    • Adds them into the prompt

    • Saves the new messages after the response


Important notes:

  • You must pass a session_id so different users don’t share memory.

  • In-memory history resets when your program restarts.

  • In production, you replace in-memory storage with a persistent store (like Redis or a database).

What is ConversationBufferMemory in LangChain?

What is ConversationBufferMemory in LangChain?

ConversationBufferMemory is a memory class in LangChain that stores the entire conversation history and sends it back to the model every time the chain runs.

In simple words:

It remembers everything that has been said so far.

And it keeps adding new messages to that memory.


Let’s understand why this is needed.

By default, an LLM does not remember previous messages.

Every call is independent.

If you say:

"My name is Rahul."

And then later ask:

"What is my name?"

The model will not know unless you send the earlier message again.

ConversationBufferMemory solves this problem.

It automatically stores previous messages and includes them in the next prompt.


How does it work?

It keeps a running “buffer” of the conversation like this:

User: Hello
AI: Hi there
User: My name is Rahul
AI: Nice to meet you
User: What is my name?

Before answering the last question, the memory sends the entire conversation to the LLM.

So the LLM sees the full history.


Here’s a simple example using classic LangChain chains.

from langchain_openai import ChatOpenAI
from langchain.chains import ConversationChain
from langchain.memory import ConversationBufferMemory

llm = ChatOpenAI(model="gpt-4o-mini")

memory = ConversationBufferMemory()

conversation = ConversationChain(
    llm=llm,
    memory=memory,
    verbose=True
)

conversation.predict(input="My name is Asha.")
conversation.predict(input="What is my name?")

What happens here?

  • The first input is stored in memory.

  • The second input is sent along with the previous conversation.

  • The model can now answer correctly.


Important characteristics:

  1. It stores the full conversation (no summarization).

  2. It keeps growing as conversation grows.

  3. It may increase token usage over time.

That last point is important.

If the conversation becomes very long, the prompt becomes large.

Large prompts:

  • Cost more

  • Slow down responses

  • May exceed context window limits


Because of that, LangChain also provides other memory types like:

  • ConversationBufferWindowMemory (stores last N messages)

  • ConversationSummaryMemory (summarizes old messages)

  • ConversationSummaryBufferMemory (combines both)

But ConversationBufferMemory is the simplest.

It stores everything.


When should you use it?

  • Small conversations

  • Prototypes

  • Demos

  • Short support chats

When should you avoid it?

  • Very long conversations

  • Large-scale production without token control

Write a function to store conversation history in a database.

A practical production pattern is:

  • Store conversation messages per session_id

  • Save both user and assistant messages

  • Fetch last N messages when needed

Below is a simple SQLite implementation (easy to run anywhere).



import sqlite3
from typing import List, Tuple, Optional

def init_chat_db(db_path: str = "chat_history.sqlite") -> None:
    conn = sqlite3.connect(db_path)
    cur = conn.cursor()
    cur.execute("""
        CREATE TABLE IF NOT EXISTS messages (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            session_id TEXT NOT NULL,
            role TEXT NOT NULL,
            content TEXT NOT NULL,
            created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
        )
    """)
    conn.commit()
    conn.close()

def save_message(session_id: str, role: str, content: str, db_path: str = "chat_history.sqlite") -> None:
    conn = sqlite3.connect(db_path)
    cur = conn.cursor()
    cur.execute(
        "INSERT INTO messages (session_id, role, content) VALUES (?, ?, ?)",
        (session_id, role, content)
    )
    conn.commit()
    conn.close()

def load_messages(session_id: str, limit: int = 20, db_path: str = "chat_history.sqlite") -> List[Tuple[str, str]]:
    """
    Returns a list of (role, content), most recent last (chronological order).
    """
    conn = sqlite3.connect(db_path)
    cur = conn.cursor()
    cur.execute("""
        SELECT role, content
        FROM messages
        WHERE session_id = ?
        ORDER BY id DESC
        LIMIT ?
    """, (session_id, limit))
    rows = cur.fetchall()
    conn.close()
    return list(reversed(rows))

Implement a function to merge multiple memory contexts.

Sometimes you have multiple memory sources, like:

  • Summary memory (high-level chat summary)

  • Entity memory (facts about people)

  • Vector memory (retrieved relevant old messages)

Merging means combining them into one context string that you inject into the prompt.

Here’s a simple merge function that is safe and predictable.



from typing import Dict, Any, Optional

def merge_memory_contexts(
    *contexts: Optional[str],
    separator: str = "\n\n",
    max_chars: int = 8000
) -> str:
    """
    Merges multiple memory strings into one context block.
    - Removes empty contexts
    - Adds clear separators
    - Truncates to max_chars to prevent runaway prompt size
    """
    parts = [c.strip() for c in contexts if c and c.strip()]
    merged = separator.join(parts)

    if len(merged) > max_chars:
        merged = merged[-max_chars:]  # keep the most recent tail
    return merged

How do you handle memory overflow in LangChain?

Memory overflow happens when conversation history becomes too large and:

  • Increases cost

  • Increases latency

  • Exceeds the model context window

The best ways to handle it are:

  1. Window-based memory

    • Keep last N turns only

    • Example: ConversationBufferWindowMemory(k=5)

  2. Token-limited memory

  4. Vector-store memory

  5. Truncation as a last resort

    • Hard cut old messages when needed (least smart option)

In production, a strong approach is:

  • Summary memory + token limit + vector retrieval for deep recall

Write a function to implement a custom memory type in LangChain.

A custom memory type is useful when you want to store memory in your own format.

To create custom memory, you implement two main things:

  • load_memory_variables() → what memory returns into the prompt

  • save_context() → how new user/AI messages are stored

Here’s a small custom memory that keeps only the last N user messages.



from typing import Dict, Any, List
from langchain.memory.chat_memory import BaseChatMemory

class LastNUserMessagesMemory(BaseChatMemory):
    def __init__(self, k: int = 5, memory_key: str = "history"):
        super().__init__()
        self.k = k
        self.memory_key = memory_key
        self._user_messages: List[str] = []

    @property
    def memory_variables(self) -> List[str]:
        return [self.memory_key]

    def load_memory_variables(self, inputs: Dict[str, Any]) -> Dict[str, str]:
        text = "\n".join(self._user_messages[-self.k:])
        return {self.memory_key: text}

    def save_context(self, inputs: Dict[str, Any], outputs: Dict[str, Any]) -> None:
        # ConversationChain uses "input" as the user field in many setups
        user_text = inputs.get("input") or inputs.get("text") or ""
        if user_text:
            self._user_messages.append(str(user_text))

    def clear(self) -> None:
        self._user_messages.clear()

Vector-store memory means:

  • Store each message as an embedding (a numeric vector)

  • Later, retrieve the most relevant past messages for the current query

  • Inject only those into the prompt

This is useful because you can store a long conversation but only fetch what matters.

Typical flow:

  1. Save messages into a vector store (like FAISS, Chroma, Pinecone)

  2. On a new user query, do similarity search

  3. Use retrieved messages as “memory context”

from langchain_openai import OpenAIEmbeddings

    vs = FAISS.from_documents([Document(page_content="")], embeddings)
    return vs

def add_to_vector_memory(vs, session_id: str, role: str, content: str):
    vs.add_documents([
        Document(
            page_content=f"{role}: {content}",
            metadata={"session_id": session_id, "role": role}
        )
    ])

def retrieve_vector_memory(vs, session_id: str, query: str, k: int = 4) -> str:
    docs = vs.similarity_search(query, k=k)

How do you optimize memory for long conversations in LangChain?

Long conversations cause three big problems:

  • Cost grows (more input tokens every turn)

  • Latency grows (more text to process)

  • Context window overflow (model can’t fit it all)

The best optimizations are:

  1. Use summary memory

    • Summarize older messages into a short summary

    • Keep only recent messages verbatim

  2. Use token-limited memory

    • Keep memory under a token budget

    • Automatically drop the oldest turns

  3. Use vector-store memory

    • Store everything in a vector database

    • Retrieve only relevant past messages for the current query

  4. Keep memory structured

    • Store “facts and preferences” separately from chat logs

    • Facts are small but very valuable

  5. Trim aggressively

    • Don’t store repeated greetings and filler text

    • Keep only what helps future turns

A strong production pattern is:

  • Recent turns (last N) + summary of older turns + retrieved relevant memories (vector store)

That keeps conversations coherent without blowing up cost and latency.

What is retrieval-augmented generation (RAG) in LangChain?

Retrieval-Augmented Generation (RAG) is a pattern where an LLM does not answer only from its internal training knowledge. Instead, it:

  1. retrieves relevant documents (or chunks) from an external knowledge source (usually a vector store), then
  2. generates the final answer using that retrieved text as context.

In LangChain, a typical RAG flow looks like:

  • Load documents → split into chunks → embed chunks → store in vector DB
  • User asks a question → embed question → retrieve top-k chunks → feed chunks + question to LLM → answer

Why this is useful:

  • improves factual accuracy for your private data
  • reduces hallucinations (when done properly)
  • allows your app to answer from updated documents without retraining the model

Write a function to create a LangChain RAG pipeline.

Below is a clean RAG pipeline function using modern LangChain (LCEL style).

It does:

  • Create embeddings

  • Build a vector store from documents

  • Retrieve relevant docs for a question

  • Pass docs + question to an LLM

  • Return a grounded answer



from typing import List
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_core.documents import Document
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough

def build_rag_pipeline(
    texts: List[str],
    model: str = "gpt-4o-mini",
    k: int = 4,
):
    # 1) Turn input texts into Documents
    docs = [Document(page_content=t) for t in texts]

    # 2) Build embeddings + vector store
    embeddings = OpenAIEmbeddings()
    vectorstore = FAISS.from_documents(docs, embeddings)
    retriever = vectorstore.as_retriever(search_kwargs={"k": k})

    # 3) Prompt: force the model to use the retrieved context
    prompt = ChatPromptTemplate.from_template(
        "Answer the question using ONLY the context below.\n"
        "If the answer is not in the context, say: \"I don't know.\"\n\n"
        "Context:\n{context}\n\n"
        "Question: {question}\n"
        "Answer:"
    )

    llm = ChatOpenAI(model=model, temperature=0.0, max_tokens=300)

    # 4) Helper to format retrieved docs into a single context string
    def format_docs(retrieved_docs: List[Document]) -> str:
        return "\n\n".join(d.page_content for d in retrieved_docs)

    # 5) RAG chain (retrieve -> format -> generate)
    rag_chain = (
        {
            "context": retriever | format_docs,
            "question": RunnablePassthrough(),
        }
        | prompt
        | llm
    )

    return rag_chain, vectorstore

Usage:



rag_chain, vs = build_rag_pipeline([
    "RAG stands for Retrieval-Augmented Generation.",
    "In RAG, the system retrieves relevant documents and provides them as context to the LLM.",
    "Embeddings enable semantic search by representing text as vectors."
])

resp = rag_chain.invoke("What is RAG and why are embeddings important?")
print(resp.content)

This is a complete, practical RAG setup:

  • Retrieval reduces hallucinations

  • Embeddings make retrieval work by meaning

  • The prompt keeps the answer grounded in the provided context

How do you use LangChain to implement semantic search?

Semantic search means you search by meaning, not just by exact keywords.

So if a user searches “password recovery”, you can still find a document that says “reset your account credentials”, even if it doesn’t contain the exact words “password recovery”.

In LangChain, semantic search is typically built using:

  • Embeddings (turn text into vectors/numbers)

  • A vector store (stores vectors and lets you search them)

  • A retriever (a standard interface LangChain uses to query the vector store)


Here’s the simplest working flow:

  1. Convert documents into embeddings

  2. Store them in a vector store (FAISS / Chroma / Pinecone, etc.)

  3. Convert user query into an embedding

  4. Run similarity search to fetch top-k most relevant documents


A simple semantic search function (FAISS example)

from typing import List
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_core.documents import Document

def build_semantic_search_index(texts: List[str]):
    embeddings = OpenAIEmbeddings()
    docs = [Document(page_content=t) for t in texts]
    vectorstore = FAISS.from_documents(docs, embeddings)
    return vectorstore

def semantic_search(vectorstore, query: str, k: int = 4) -> List[str]:
    docs = vectorstore.similarity_search(query, k=k)
    return [d.page_content for d in docs]

Usage:

vs = build_semantic_search_index([
    "To reset your password, go to settings and click 'Forgot password'.",
    "LangChain is used to build LLM applications with chains and tools.",
    "RAG retrieves documents and provides them to the LLM as context."
])

results = semantic_search(vs, "How do I recover my account password?", k=2)
for r in results:
    print("-", r)

What’s happening:

  • Documents are stored as vectors

  • The query becomes a vector

  • FAISS finds the closest vectors (most similar meaning)

Implement a function to combine multiple vector stores in LangChain.

Combining multiple vector stores usually means: you want to search across separate indexes (for example, one for product docs, one for policies, one for tickets) and return the best results overall.

In LangChain, the cleanest way is to convert each vector store into a retriever, then combine them using a MergerRetriever.

That gives you one “super retriever” that queries all of them.

Here’s a practical function.

from typing import List
from langchain_core.retrievers import BaseRetriever
from langchain.retrievers.merger_retriever import MergerRetriever

def combine_vectorstores_as_retriever(
    vectorstores: List[object],
    k: int = 4
) -> BaseRetriever:
    """
    Combines multiple LangChain vector stores into a single retriever.
    Each vector store is queried, and results are merged into one list.

    vectorstores: list of vector store objects that support .as_retriever()
    k: number of results to fetch per store
    """
    retrievers = [vs.as_retriever(search_kwargs={"k": k}) for vs in vectorstores]
    return MergerRetriever(retrievers=retrievers)

Example usage

# vs1 and vs2 are vector stores (FAISS, Chroma, etc.)
combined_retriever = combine_vectorstores_as_retriever([vs1, vs2], k=3)

docs = combined_retriever.invoke("How do I reset my password?")
for d in docs:
    print(d.page_content)

Important note about ranking

MergerRetriever merges results from all stores, but the merged list may not be globally “best-ranked” across stores, because each store returns its own top-k.

If you want better ranking, a common production approach is:

  • retrieve top-k from each store

  • then rerank all retrieved documents using an LLM or a cross-encoder model

But for most applications, merging is already very useful and simple

How do you optimize vector store retrieval in LangChain?

How do you optimize vector store retrieval in LangChain?

Optimizing vector store retrieval means improving two things at the same time:

  • Quality: you fetch the most relevant chunks

  • Speed/Cost: you fetch them quickly and don’t retrieve unnecessary text

In LangChain, most retrieval problems come from bad chunking, weak search settings, or too much irrelevant context.


1) Chunk your documents correctly

Chunking is the #1 retrieval quality lever.

If chunks are too big:

  • Retrieval returns large blocks with a lot of noise

  • Prompts become expensive

If chunks are too small:

  • You lose context

  • Answers become incomplete

A good starting point for many text docs is:

  • chunk_size: 500–1000 characters

  • chunk_overlap: 50–150 characters

Example:

from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=120
)
chunks = splitter.split_text(long_text)

2) Use metadata and filtering

Metadata reduces search space and improves precision.

Store metadata like:

  • user_id / tenant_id

  • document type

  • product/category

  • date

Then filter retrieval.

Many vector stores support metadata filters (exact syntax depends on store).

Conceptually:

  • “Search only in category=policies”

  • “Search only in user_id=123 docs”

This gives faster and cleaner results.


3) Tune k (top results) and fetch_k

k is how many documents you finally return.

If k is too low:

  • you miss relevant context

If k is too high:

  • you add noise and increase prompt cost

A common pattern is:

  • fetch more candidates (fetch_k)

  • return fewer final docs (k)

This is especially helpful with MMR.


4) Use MMR (Max Marginal Relevance) to reduce duplicates

Similarity search often returns near-duplicate chunks.

MMR improves diversity, so you get different parts of the document set.

In LangChain, use:

  • search_type="mmr"

  • set fetch_k higher than k

Example:

retriever = vectorstore.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 4, "fetch_k": 20, "lambda_mult": 0.5}
)

What this does:

  • Gets 20 candidates

  • Selects 4 diverse results

  • Reduces repeated context


5) Re-rank retrieved results (big quality boost)

If you need high accuracy, do this:

  1. Retrieve top 20 with embeddings

  2. Re-rank with a stronger model

Re-ranking can be done using:

  • an LLM scoring prompt

  • a cross-encoder model (common in search systems)

This often improves retrieval quality more than changing embedding models.


6) Reduce context before sending to the LLM

Even after retrieval, you can compress what you send.

Two common ways:

  • Summarize each chunk into key points

  • Extract only relevant sentences

This reduces tokens → cheaper + faster.

In LangChain, this is often called context compression.


7) Cache retrieval results

If the same query repeats, caching saves time.

You can cache:

  • query → doc ids

  • query embedding → results

Even simple caching helps in production.


8) Use the right vector store and index settings

Performance depends on the store:

  • FAISS is fast locally

  • Pinecone/Weaviate/Qdrant scale well

  • HNSW indexes are fast for large datasets

For big collections:

  • Use ANN (approximate nearest neighbor) indexes (like HNSW)

  • Don’t do brute-force search


9) Evaluate retrieval, don’t guess

Good retrieval is measured by tests like:

  • “Did we retrieve the chunk that contains the answer?”

  • precision@k, recall@k

A simple way:

  • Prepare 20–50 real user queries

  • Check if retrieved docs contain the answer

  • Tune chunking + k + MMR + filters


A practical optimized retriever function

Here’s a clean function that applies the most useful settings:

from typing import Optional, Dict, Any

def build_optimized_retriever(
    vectorstore,
    k: int = 4,
    fetch_k: int = 20,
    use_mmr: bool = True,
    filters: Optional[Dict[str, Any]] = None,
):
    """
    Builds a retriever with sensible defaults:
    - MMR for diversity
    - fetch_k > k for better candidate selection
    - optional metadata filtering
    """
    search_type = "mmr" if use_mmr else "similarity"
    search_kwargs = {"k": k}

    if use_mmr:
        search_kwargs.update({"fetch_k": fetch_k, "lambda_mult": 0.5})

    if filters:
        search_kwargs["filter"] = filters  # supported by many vector stores

    return vectorstore.as_retriever(search_type=search_type, search_kwargs=search_kwargs)

    k=4,
    fetch_k=25,
    use_mmr=True,
    filters={"category": "policies"}
)

def compress_docs(docs, question):

How do you debug a LangChain chain that fails?

When a chain fails, it’s usually one of these: wrong inputs, prompt variable mismatch, tool/model errors, parsing issues, or retrieval returning junk. A clean debugging workflow:

  1. Reproduce with a single input (don’t test with many variables first).
  2. Check prompt variables match keys in your input dict.
    • If prompt has {question} you must pass {"question": "..."}.
  3. Print the formatted prompt (or at least the resolved inputs) before calling the model.
  4. Run in steps instead of running the full chain:
    • prompt formatting → model call → output parsing
  5. If you use RAG:
    • inspect retrieved docs (print them) before calling the LLM
  6. If you parse JSON:
    • log the raw model output and only then parse
  7. Turn on verbose tracing/logging to see where it breaks.

Most failures become obvious when you look at:

  • the exact prompt sent
  • the exact documents retrieved
  • the exact raw output returned

Write a function to retry LangChain chain execution on failure. 
import time
import random

def retry_chain_invoke(chain, inputs, max_retries=3, base_delay=1.0):
    last_error = None

    for attempt in range(1, max_retries + 1):
        try:
            return chain.invoke(inputs)
        except Exception as e:
            last_error = e

            # Exponential backoff with jitter
            delay = base_delay * (2 ** (attempt - 1))
            delay = delay + random.uniform(0, 0.25 * delay)

            if attempt == max_retries:
                break

            time.sleep(delay)

    raise RuntimeError(f"Chain failed after {max_retries} retries: {last_error}")

Code explanation:

  • attempt loop tries to invoke the chain.
  • if it fails, we wait (backoff) and try again.
  • jitter avoids many clients retrying at the same moment.
  • after max retries, it raises a clear error.

Write a function to optimize LangChain memory usage.

A practical memory optimization is: keep only the last N messages and optionally keep a summary for older messages. Below is a simple “trim history” function that works even if your history is stored as a list of message dicts.

def optimize_memory(history, max_messages=12):
    """
    Trims conversation history to the last `max_messages` items.

    Expected formats:
    - history as a list of strings
    - history as a list of dicts like {"role": "user"/"assistant", "content": "..."}
    """
    if history is None:
        return history

    if not isinstance(history, list):
        # If it's already a string (like a big history blob), you can optionally truncate it here.
        return history

    # Keep only the most recent messages
    trimmed = history[-max_messages:]

    return trimmed

Code explanation:

  • If history is a list, we keep only the last max_messages.
  • This prevents long chats from growing endlessly and overflowing the model context window.
  • This function is intentionally simple; you can extend it by:
    • summarizing older messages into one summary string
    • storing older turns in a database or vector store for long-term memory

Example usage:

inputs["history"] = optimize_memory(inputs.get("history"), max_messages=10)
result = chain.invoke(inputs)
