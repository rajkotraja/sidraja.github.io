What is Corrective RAG (CRAG), and how does it improve retrieval accuracy?

Definition

Corrective RAG (CRAG) is an enhanced version of Retrieval-Augmented Generation (RAG) where the system not only retrieves external documents but also evaluates and corrects the quality of retrieved information before using it for generation.

It adds a feedback or verification step to reduce errors from irrelevant, outdated, or low-quality retrieved content.

Why it is used

Standard RAG can still fail when:

  • Retrieved documents are irrelevant

  • Sources are noisy or contradictory

  • Retrieval step misses key information
    CRAG improves reliability by actively correcting or filtering retrieval results before passing them to the language model.

How it works (step-by-step)

  1. User query is received
    The system analyzes the input question.

  2. Initial retrieval (like standard RAG)
    A retriever fetches top-k relevant documents from a knowledge base or search index.

  3. Relevance evaluation / correction step

    • Retrieved documents are scored or verified for relevance

    • Irrelevant or low-confidence documents are removed or down-weighted

    • Some systems may re-query or refine the search if results are weak

  4. Optional query refinement
    If retrieval quality is poor, the system reformulates the query and retrieves again.

  5. Final context selection
    Only the most relevant and reliable documents are passed to the LLM.

  6. Answer generation
    The model generates the response using corrected, high-quality context.

Key components

  • Retriever (search engine or vector database)

  • Relevance scorer or critic model

  • Query rewriting module (optional)

  • Filtering / reranking system

  • Generator (LLM)

How CRAG improves retrieval accuracy

1. Removes irrelevant documents

Filtering ensures only high-quality context is used.

2. Reduces hallucinations

Better context reduces the model’s need to guess or invent information.

3. Improves ranking quality

Reranking ensures the most useful documents appear first.

4. Handles poor retrieval cases

If initial retrieval is weak, the system can retry or refine queries.

5. Enhances robustness

Works better in noisy or ambiguous search scenarios.

Advantages

  • More accurate and grounded responses

  • Reduced hallucination compared to standard RAG

  • Better handling of ambiguous queries

  • Improved trustworthiness in enterprise use cases

Disadvantages

  • Higher latency due to extra evaluation step

  • More computational cost

  • Increased system complexity

  • Requires additional scoring or verification models

Simple real-world example

Imagine asking a research assistant a question.

  • In standard RAG, the assistant quickly collects documents and answers based on whatever they find.

  • In CRAG, the assistant first checks whether those documents are actually relevant, discards weak sources, and only then writes the answer.

So instead of “just searching,” it adds a “fact-checking filter” before answering.

Short interview-ready answer

Corrective RAG (CRAG) is an improved version of Retrieval-Augmented Generation that adds a correction or verification step after retrieval. It evaluates and filters retrieved documents for relevance and quality, and may refine queries if needed. This improves retrieval accuracy, reduces irrelevant context, and leads to more reliable and less hallucinated model outputs.

How does RAG Fusion improve recall over single-query retrieval?

Definition

RAG Fusion is an advanced Retrieval-Augmented Generation (RAG) technique that improves retrieval quality by generating multiple query variations, retrieving documents for each, and then fusing (combining and reranking) the results into a stronger final context set.

Why it is used

Single-query retrieval often misses relevant documents because a user’s question can be phrased in many ways. RAG Fusion reduces this “query mismatch” problem and improves recall (finding more relevant information).

How it works (step-by-step)

  1. Original query input
    The user asks a question.

  2. Query expansion
    The system generates multiple semantically similar or reformulated queries, such as:

    • paraphrases

    • keyword-focused versions

    • expanded contextual versions

  3. Parallel retrieval
    Each query is sent to the retriever (vector DB or search engine), producing multiple result sets.

  4. Result fusion
    All retrieved documents are merged.

  5. Ranking / scoring (fusion step)

    • Documents are re-ranked based on frequency, relevance scores, or reciprocal rank fusion (RRF)

    • Highly consistent documents across queries get higher priority

  6. Final context selection
    Top-ranked documents are selected for the LLM to generate the answer.

  7. Generation step
    The LLM uses the fused context to produce the final response.

Key components

  • Query rewriting / expansion module

  • Retriever (vector database or search system)

  • Fusion algorithm (e.g., Reciprocal Rank Fusion)

  • Reranker or scoring model

  • LLM generator

How it improves recall

1. Multiple query perspectives

Different query formulations capture different subsets of relevant documents.

2. Reduces query mismatch

Even if one query fails, others may succeed in retrieving missing information.

3. Broader coverage

Combines results from several retrieval paths, increasing total relevant document coverage.

4. Reinforces important results

Documents retrieved consistently across queries are prioritized.

Advantages

  • Higher recall than single-query retrieval

  • Better robustness to vague or ambiguous queries

  • Reduces dependency on perfect query formulation

  • Improves answer completeness

Disadvantages

  • Higher latency due to multiple retrieval calls

  • Increased compute and storage usage

  • Possible noise from irrelevant expanded queries

  • Requires careful fusion strategy tuning

Simple real-world example

If you ask a librarian one question in a single way, they might miss some books. But if you ask the same question in different ways like:

  • “AI in healthcare”

  • “medical applications of artificial intelligence”

  • “how AI is used in hospitals”

and then combine all book recommendations, you are more likely to find all relevant books. That is exactly what RAG Fusion does.

Short interview-ready answer

RAG Fusion improves recall by generating multiple variations of a user query, retrieving documents for each, and then fusing and reranking the results. This increases coverage by capturing relevant documents that a single-query retrieval might miss, making the final context more complete and improving overall response quality.

What is Modular RAG, and how does it differ from basic/advanced RAG?

Definition

Modular RAG is a Retrieval-Augmented Generation (RAG) architecture where the system is built as a set of interchangeable components (modules)—such as retrieval, query rewriting, reranking, memory, and generation—that can be independently optimized, replaced, or extended.

Instead of one fixed pipeline, Modular RAG is a flexible system design for building retrieval-based LLM applications.

Why it is used

As RAG systems become more complex, a single fixed pipeline becomes hard to scale or improve. Modular RAG allows developers to:

  • Swap components easily (e.g., different retrievers or rerankers)

  • Add new capabilities without redesigning the whole system

  • Optimize each stage independently for better performance

How it works (step-by-step)

A Modular RAG system typically includes interchangeable modules:

  1. Query processing module

    • Cleans or rewrites the user query

    • May expand or decompose questions

  2. Retrieval module

    • Fetches relevant documents (vector search, BM25, hybrid search, etc.)

  3. Reranking module

    • Reorders retrieved documents based on relevance scoring

  4. Context assembly module

    • Selects and formats the best chunks for the LLM

  5. Generation module

    • LLM generates the final answer

  6. (Optional) Memory / feedback module

    • Stores past interactions or improves future retrieval

Each module can be independently upgraded or replaced.

Key idea

“RAG as a system of plug-and-play components instead of a fixed pipeline.”

How it differs from basic RAG

Basic RAG

  • Simple pipeline:
    Query → Retrieve → Generate

  • Single retriever

  • No reranking or query refinement

  • Limited flexibility

Limitations of basic RAG

  • Lower recall and precision

  • Sensitive to query phrasing

  • Hard to improve individual stages independently

How it differs from advanced RAG

Advanced RAG

  • Adds improvements like:

    • reranking

    • query rewriting

    • multi-query retrieval (e.g., RAG Fusion)

  • Still often built as a tightly coupled pipeline

Limitation of advanced RAG

  • Improvements are often added as extensions, not cleanly separable modules

  • Harder to swap or reuse components across systems

Modular RAG vs Basic/Advanced RAG

Flexibility

  • Basic RAG: Low

  • Advanced RAG: Medium

  • Modular RAG: High

Architecture

  • Basic: Fixed pipeline

  • Advanced: Enhanced pipeline

  • Modular: Component-based system

Customization

  • Basic: Limited

  • Advanced: Partial

  • Modular: Fully customizable

Scalability

  • Basic: Limited

  • Advanced: Better

  • Modular: Highly scalable and extensible

Advantages of Modular RAG

  • Easy to upgrade individual components

  • Better experimentation and A/B testing

  • Supports hybrid retrieval strategies

  • Scales well for enterprise systems

  • Encourages reusable architecture

Disadvantages

  • More complex system design

  • Higher engineering overhead

  • Requires coordination between modules

  • Debugging can be harder due to multiple moving parts

Simple real-world example

Think of a restaurant:

  • Basic RAG: One chef does everything—prepares ingredients, cooks, and serves.

  • Advanced RAG: Still one kitchen, but with some helpers.

  • Modular RAG: Separate specialists—one for prep, one for cooking, one for plating, and one for quality check. Each can be improved or replaced independently.

Short interview-ready answer

Modular RAG is a retrieval-augmented generation architecture where the system is built using independent, interchangeable components like query processing, retrieval, reranking, and generation. Unlike basic RAG, which uses a simple retrieve-and-generate pipeline, and advanced RAG, which adds improvements within a fixed pipeline, Modular RAG allows full flexibility to swap and optimize each module independently, making it more scalable and adaptable.

What is ColBERT, and how does late interaction improve retrieval?

Definition

ColBERT (Contextualized Late Interaction over BERT) is a neural retrieval model that improves search accuracy by representing both queries and documents as token-level embeddings and performing a “late interaction” matching step between them.

Unlike traditional dense retrieval that compresses a whole document into a single vector, ColBERT keeps richer information at the token level.

Why it is used

Standard retrieval methods often lose detail because they compress entire texts into one embedding. This makes it harder to capture fine-grained relevance (especially for long documents). ColBERT addresses this by preserving detailed token interactions while still remaining efficient.

How it works (step-by-step)

  1. Encode query and documents separately

    • Both query and document are passed through a transformer (like BERT).

    • Instead of one vector, each word/token gets its own embedding.

  2. Token-level representation storage

    • Documents are stored as a matrix of embeddings (not a single vector).

    • Query is also represented as multiple token embeddings.

  3. Late interaction step

    • For each query token, the model computes similarity with all document tokens.

    • For each query token, it takes the maximum similarity score over document tokens.

  4. Score aggregation

    • These max scores are summed (or aggregated) to produce the final relevance score.

  5. Ranking

    • Documents are ranked based on these interaction scores.

Key idea

Instead of compressing meaning early, ColBERT delays matching until after encoding—hence “late interaction.”

How late interaction improves retrieval

1. Preserves fine-grained meaning

  • Each word in a query can match specific words in a document

  • Avoids loss of detail from single-vector embeddings

2. Better handling of long documents

  • Important parts of a document are not diluted in a single embedding

  • Even small relevant phrases can strongly influence ranking

3. Higher retrieval accuracy

  • Captures exact semantic matches at token level

  • Improves precision and recall compared to single-vector models

4. More expressive than dense retrieval

  • Dense retrieval = one vector per text

  • ColBERT = many vectors per text with interaction

Trade-offs / disadvantages

  • Higher storage cost (stores token embeddings for documents)

  • More compute during scoring (token-to-token comparisons)

  • More complex indexing compared to standard vector search

Simple real-world example

Imagine searching for a document about “deep learning optimization techniques.”

  • A single-vector model might miss it if the document talks about “neural network training improvements” indirectly.

  • ColBERT can still match important token-level overlaps like “learning,” “optimization,” and “training,” even if phrased differently across sentences.

Short interview-ready answer

ColBERT is a neural retrieval model that uses token-level embeddings for both queries and documents and applies a late interaction mechanism to compute relevance. Instead of compressing texts into a single vector, it compares query tokens with document tokens and aggregates maximum similarities. This improves retrieval accuracy by preserving fine-grained semantic matching while still being efficient for large-scale search systems.

How does FLARE decide when to fetch external information?

Definition

FLARE (Forward-Looking Active Retrieval Augmented Generation) is a RAG-style approach where the model dynamically decides during generation whether it needs to retrieve external information, based on its confidence in the next token or continuation.

Instead of retrieving once at the start, FLARE performs adaptive, step-by-step retrieval triggers.

Why it is used

Traditional RAG retrieves context before generation, which can be inefficient or unnecessary if the model already knows the answer. FLARE improves efficiency and accuracy by retrieving only when the model is uncertain or likely to hallucinate.

How it works (step-by-step)

  1. Initial generation starts without retrieval
    The model begins producing the answer using its internal knowledge.

  2. Forward-looking prediction
    At each step (or span), the model predicts upcoming tokens and estimates confidence.

  3. Uncertainty detection
    The system checks signals such as:

    • Low probability for next tokens

    • High entropy (uncertainty in predictions)

    • Ambiguous or knowledge-heavy phrases emerging

  4. Retrieval trigger decision
    If uncertainty crosses a threshold, FLARE decides:

    • “I need external knowledge here”

  5. Query construction
    The partially generated text is converted into a search query.

  6. External retrieval
    Documents are fetched from a knowledge base or search engine.

  7. Context injection and regeneration
    Retrieved information is added, and the model continues generation with improved grounding.

  8. Repeat dynamically
    This process can occur multiple times during generation.

Key components

  • Language model with confidence estimation

  • Uncertainty scoring mechanism

  • Retrieval system (search engine or vector database)

  • Query generation module

  • Iterative generation loop

How FLARE decides when to fetch information

1. Low confidence in token prediction

If the model is unsure about what comes next, retrieval is triggered.

2. High uncertainty spans

Long ambiguous segments (e.g., factual or technical claims) increase retrieval likelihood.

3. Knowledge-sensitive phrases

Triggers often occur when the model starts generating:

  • dates, facts, names

  • technical explanations

  • rare or domain-specific content

4. Forward prediction risk

If future tokens are predicted to likely require factual grounding, retrieval is initiated proactively.

Advantages

  • Reduces unnecessary retrieval calls

  • Improves factual accuracy during generation

  • Reduces hallucinations dynamically

  • More efficient than always-on RAG

  • Better handling of multi-step reasoning

Disadvantages

  • More complex system design

  • Requires good uncertainty estimation (not trivial)

  • Slight latency overhead due to on-the-fly retrieval

  • May over-trigger or under-trigger if thresholds are poorly tuned

Simple real-world example

Imagine writing an essay:

  • You start writing from memory.

  • When you reach a section where you're unsure about a fact (like a historical date), you pause and look it up.

  • Then you continue writing with correct information.

FLARE does exactly this—but automatically and continuously during text generation.

Short interview-ready answer

FLARE decides when to fetch external information by monitoring the model’s uncertainty during generation. If the model shows low confidence or high uncertainty in predicting the next tokens, it triggers retrieval, constructs a query from the partial output, and fetches relevant documents. This allows retrieval to happen dynamically only when needed, improving efficiency and factual accuracy.

What is Cache-Augmented Generation, and when is it better than RAG?

Definition

Cache-Augmented Generation (CAG) is a retrieval approach where instead of repeatedly searching external databases for every query, the system uses a precomputed cache of embeddings, documents, or model responses to quickly supply relevant context to the LLM.

The “cache” acts as a fast-access memory layer between the model and knowledge source.

Why it is used

Traditional RAG systems can be slow and expensive because they perform retrieval at query time. CAG reduces latency and cost by reusing previously computed or frequently accessed information.

How it works (step-by-step)

  1. Knowledge is preprocessed and stored in a cache

    • Documents are embedded and indexed

    • Or frequent query–response pairs are stored

  2. User query arrives
    The system first checks the cache instead of immediately querying a retriever.

  3. Cache lookup

    • If a close match is found, relevant context is retrieved instantly

    • Matching can be based on embeddings or semantic similarity

  4. Cache hit or miss decision

    • Cache hit → use cached context directly

    • Cache miss → fall back to full retrieval system (like RAG)

  5. LLM generation
    The model generates the final response using cached or retrieved context.

Key components

  • Embedding-based cache or key-value store

  • Similarity search mechanism

  • Optional fallback retriever (RAG system)

  • LLM generator

How CAG differs from RAG

RAG (Retrieval-Augmented Generation)

  • Always performs live retrieval at query time

  • Fetches fresh documents from external sources

  • Better for dynamic or constantly changing data

CAG (Cache-Augmented Generation)

  • Uses pre-stored or frequently reused information

  • Avoids repeated retrieval calls

  • Optimized for speed and cost efficiency

When CAG is better than RAG

1. Repetitive queries

If many users ask similar questions (e.g., FAQs), caching avoids redundant retrieval.

2. Low-latency requirements

CAG is faster because it avoids real-time search.

3. Stable knowledge domains

Works well when information does not change frequently (e.g., product documentation, help centers).

4. High-cost retrieval systems

Reduces API calls or vector DB queries, lowering operational cost.

5. Edge or offline systems

Useful when real-time retrieval is expensive or not available.

When RAG is better than CAG

  • Rapidly changing data (news, stock prices)

  • Long-tail or rare queries not seen before

  • Need for up-to-date or real-time information

  • Environments where caching is insufficient

Advantages of CAG

  • Very low latency

  • Reduced compute and retrieval cost

  • Efficient for repeated queries

  • Simple serving pipeline once cache is built

Disadvantages

  • Stale information if cache is outdated

  • Limited coverage for unseen queries

  • Requires cache maintenance and updates

  • Less effective for dynamic knowledge

Simple real-world example

Think of asking a helpdesk:

  • RAG: The agent searches the entire knowledge base every time you ask a question.

  • CAG: The agent keeps frequently asked answers on a desk for instant access and only searches deeper when needed.

Short interview-ready answer

Cache-Augmented Generation (CAG) is a retrieval approach where frequently used or precomputed knowledge is stored in a cache and reused during inference instead of performing live retrieval each time. It is better than RAG in scenarios with repetitive queries, stable knowledge, and low-latency requirements, as it reduces retrieval cost and improves response speed, but it is less effective for dynamic or long-tail queries.

How does adaptive retrieval decide if external data is needed?

Definition

Adaptive retrieval is a Retrieval-Augmented Generation (RAG) strategy where the system dynamically decides whether to fetch external information or rely on the model’s internal knowledge, based on the difficulty and uncertainty of the query during inference.

Why it is used

Not every question needs external retrieval. Simple questions can be answered directly by the model, while factual, complex, or up-to-date questions require external data. Adaptive retrieval reduces unnecessary retrieval calls, improving efficiency and latency.

How it works (step-by-step)

  1. Query analysis begins
    The system examines the user input to estimate complexity and knowledge requirements.

  2. Internal model attempt (optional first pass)
    The model may try to generate an answer using its own parameters.

  3. Uncertainty estimation
    The system evaluates whether the model is confident using signals like:

    • Low token probability (low confidence)

    • High entropy in predictions

    • Ambiguous or incomplete reasoning paths

    • Detection of factual or time-sensitive content

  4. Retrieval decision module
    A learned classifier or heuristic decides:

    • “Use internal knowledge only” OR

    • “Trigger external retrieval”

  5. External retrieval (if needed)
    If triggered, the system fetches relevant documents from a search engine or vector database.

  6. Final answer generation
    The LLM generates the response using either:

    • internal knowledge, or

    • retrieved context + internal knowledge

Key components

  • Query complexity classifier

  • Uncertainty estimation (confidence scoring)

  • Retrieval trigger policy (rule-based or learned)

  • Retriever (vector DB / search engine)

  • LLM generator

How it decides if external data is needed

1. Model confidence signals

If the model is unsure about generating the next tokens, retrieval is triggered.

2. Query type detection

  • Factual / knowledge-heavy → likely retrieval

  • Conversational / simple reasoning → no retrieval

3. Temporal awareness

If the query involves:

  • “latest”, “today”, “current” → retrieval required

4. Complexity scoring

Long, multi-hop reasoning questions are more likely to require retrieval.

5. Learned gating models

Some systems use a small classifier trained to predict whether retrieval will improve answer quality.

Advantages

  • Reduces unnecessary retrieval calls

  • Improves latency and cost efficiency

  • Balances internal knowledge vs external grounding

  • Reduces overload on retrieval systems

Disadvantages

  • Hard to tune confidence thresholds

  • Risk of false negatives (skipping retrieval when needed)

  • Adds system complexity

  • Requires good calibration of uncertainty signals

Simple real-world example

Think of a student answering questions:

  • If asked “What is 2 + 2?”, they answer immediately from memory.

  • If asked “What were the latest inflation rates in India?”, they check the internet first.

Adaptive retrieval works the same way—deciding whether memory is enough or external lookup is needed.

Short interview-ready answer

Adaptive retrieval decides whether external data is needed by evaluating query complexity and the model’s confidence in its own response. If the model shows high uncertainty, detects factual or time-sensitive content, or identifies complex reasoning requirements, it triggers retrieval; otherwise, it relies on internal knowledge. This improves efficiency while maintaining answer accuracy.

What is federated RAG, and how does it query multiple sources?

Definition

Federated RAG (Retrieval-Augmented Generation) is a distributed RAG architecture where the system retrieves information from multiple independent knowledge sources (databases, organizations, or domains) and then combines the results to generate a unified answer.

Instead of relying on a single vector database, federated RAG “federates” retrieval across many sources.

Why it is used

Real-world knowledge is often spread across different systems:

  • Enterprise databases

  • External APIs

  • Private document stores

  • Web sources

Federated RAG allows a single query to access all of them while respecting access control and data separation.

How it works (step-by-step)

  1. User query is received
    The system analyzes the input question.

  2. Query routing (source selection)
    A router module decides which knowledge sources are relevant.
    Example sources:

    • internal company wiki

    • product database

    • external search API

    • domain-specific vector DBs

  3. Parallel query execution
    The same (or reformulated) query is sent to multiple sources simultaneously.

  4. Local retrieval per source
    Each source independently retrieves top-k relevant documents using its own retriever (BM25, embeddings, etc.).

  5. Result aggregation
    Retrieved results from all sources are merged into a single pool.

  6. Cross-source ranking / fusion
    A reranking model or scoring mechanism prioritizes the most relevant documents across all sources.

  7. Context construction
    The best combined set of documents is formatted into a single context.

  8. LLM generation
    The model generates a final response using the unified context.

Key components

  • Query router (decides which sources to use)

  • Multiple retrievers (one per data source)

  • Parallel execution layer

  • Fusion / reranking module

  • Access control system

  • LLM generator

How it queries multiple sources

1. Query decomposition or routing

The system decides:

  • which sources are relevant

  • whether to query all or a subset

2. Parallel retrieval

Each selected source independently processes the query at the same time.

3. Heterogeneous search

Different sources may use:

  • vector search (semantic retrieval)

  • keyword search (BM25)

  • structured database queries (SQL/API calls)

4. Result fusion

Outputs are combined using techniques like:

  • ranking scores

  • reciprocal rank fusion (RRF)

  • learned rerankers

Advantages

  • Access to diverse knowledge systems

  • Better coverage than single-source RAG

  • Scales across organizations or domains

  • Supports hybrid retrieval (structured + unstructured data)

  • More complete and accurate answers

Disadvantages

  • Higher system complexity

  • Increased latency due to multiple queries

  • Requires synchronization and ranking across sources

  • Harder debugging and monitoring

  • Data governance and access control challenges

Simple real-world example

Think of asking a complex business question:

Instead of asking one librarian, federated RAG asks:

  • finance department database

  • product documentation system

  • customer support logs

  • external web sources

Then combines all answers into one unified response.

Short interview-ready answer

Federated RAG is a retrieval-augmented generation approach that queries multiple independent knowledge sources in parallel, such as databases, APIs, or vector stores. A routing system selects relevant sources, each source retrieves documents independently, and the results are then fused and reranked before being passed to the LLM. This enables broader coverage and more accurate responses compared to single-source RAG systems.

How does iterative query refinement improve answers?

Definition

Iterative query refinement is a retrieval strategy where the system repeatedly improves a user’s query based on intermediate results and model feedback, instead of relying on a single fixed query. Each iteration makes the query clearer, more specific, or better aligned with relevant documents.

Why it is used

A single query is often:

  • ambiguous

  • underspecified

  • too broad or too narrow

Iterative refinement helps the system progressively converge toward better retrieval results, improving answer quality in RAG systems.

How it works (step-by-step)

  1. Initial query input
    The user provides a question.

  2. First retrieval attempt
    The system retrieves documents based on the original query.

  3. Analysis of retrieved results
    The system evaluates:

    • relevance of documents

    • missing information

    • ambiguity in the query

    • gaps between query and retrieved content

  4. Query refinement step
    The system rewrites or expands the query using:

    • missing keywords from documents

    • clarified intent

    • decomposed sub-questions

  5. Second retrieval round
    A new search is performed using the refined query.

  6. Iteration continues (if needed)
    Steps 3–5 repeat until:

    • sufficient relevant documents are found, or

    • a stopping condition is met

  7. Final answer generation
    The best retrieved context is used by the LLM to generate the response.

Key components

  • Query rewriting module

  • Feedback loop from retrieval results

  • Relevance evaluation system

  • Retriever (vector DB or search engine)

  • Stopping criteria (max iterations or confidence threshold)

How it improves answers

1. Reduces ambiguity

Refinement clarifies vague queries into more precise ones.

2. Improves recall

Additional iterations help retrieve documents missed initially.

3. Improves precision

Irrelevant terms are removed, making results more focused.

4. Enables multi-hop reasoning

Breaks complex questions into smaller, searchable sub-queries.

5. Aligns query with corpus language

Adjusts wording to match how information is written in documents.

Advantages

  • Higher retrieval accuracy

  • Better handling of complex or vague queries

  • Improved multi-step reasoning

  • More complete context for generation

Disadvantages

  • Higher latency due to multiple retrieval rounds

  • Increased compute cost

  • Risk of over-refinement (drifting away from original intent)

  • Requires good stopping criteria

Simple real-world example

Imagine asking a librarian:

  • First, you ask: “Tell me about AI in healthcare.”

  • The librarian gives some general books.

  • You then refine: “AI applications in medical diagnosis using imaging.”

  • You get more specific books.

  • You refine again if needed until you find exactly what you need.

Each step improves the precision of the search.

Short interview-ready answer

Iterative query refinement improves answers by repeatedly updating the user’s query based on retrieved results and feedback from earlier searches. This process helps reduce ambiguity, improve relevance, and increase recall by progressively aligning the query with useful information in the knowledge base, leading to more accurate and complete final responses.

What role does planning play in PlanRAG?

Definition

PlanRAG (Planning-augmented Retrieval-Augmented Generation) is a RAG framework where the system first creates a structured plan of steps or sub-questions before retrieving information and generating an answer.

In this setup, planning acts as a reasoning layer between the user query and retrieval process.

Why it is used

Complex questions often require multiple pieces of information from different sources. Without planning, retrieval can be:

  • incomplete

  • unfocused

  • inefficient (too many irrelevant searches)

Planning ensures the system knows what to search for and in what order.

How planning works in PlanRAG (step-by-step)

  1. User query is received
    The system analyzes the question.

  2. Plan generation (core step)
    The model breaks the query into a structured plan, such as:

    • sub-questions

    • reasoning steps

    • required knowledge components

    Example:
    “What are the causes and effects of climate change?” →

    • What causes climate change?

    • What are its environmental effects?

    • What are its economic impacts?

  3. Step-wise retrieval guided by the plan
    Each sub-question triggers targeted retrieval instead of a single broad search.

  4. Intermediate synthesis (optional)
    Retrieved information is summarized per step.

  5. Final reasoning and generation
    The model combines all intermediate results into a coherent final answer.

Key role of planning in PlanRAG

1. Decomposes complex queries

Turns one difficult question into smaller, solvable tasks.

2. Guides retrieval strategy

Ensures each retrieval call is purposeful and focused.

3. Reduces irrelevant search results

Avoids broad or noisy retrieval by narrowing intent per step.

InstructRAG is a Retrieval-Augmented Generation approach where the model follows explicit instruction-guided reasoning steps (often multi-step or decomposed prompts) before and during retrieval, instead of directly answering a query in one pass.

Why it is used

Standard RAG may:

  • retrieve irrelevant documents

  • miss intermediate reasoning steps

  • fail on multi-hop or complex questions

    The system interprets the user query and identifies:

    • intent

    • required reasoning type

    • missing information

    • sub-questions

    • intermediate reasoning goals

    • facts to verify

    All intermediate reasoning outputs are combined into a coherent final response.

Instead of:

Query → Reasoning steps → Targeted retrieval → Step-wise reasoning → Final answer

  • More accurate and structured responses

InstructRAG improves outputs by introducing structured reasoning steps before and during retrieval. When should knowledge graphs be used instead of RAG?

Definition

A Knowledge Graph (KG) is a structured representation of information where entities (nodes) and their relationships (edges) are explicitly defined.
RAG (Retrieval-Augmented Generation) retrieves unstructured or semi-structured text and uses an LLM to generate answers from it.

Why this comparison matters

Both are ways to “give external knowledge” to LLMs, but they work very differently:

  • RAG → text-centric, flexible, unstructured retrieval

  • Knowledge Graphs → structured, relationship-centric reasoning

When knowledge graphs should be used instead of RAG

1. When relationships matter more than text

Use KG when questions depend on explicit relationships between entities.

Example:

  • “Who is the parent company of X?”

  • “What is the relationship between A and B?”

KGs are naturally built for this.

2. When data is highly structured and stable

Knowledge graphs work best when:

  • schema is well-defined

  • entities and relations are stable

  • updates are controlled

Examples:

  • medical ontologies

  • company hierarchies

  • product catalogs

  • scientific taxonomies

3. When exact reasoning or traversal is needed

KGs enable multi-hop logical queries, such as:

  • A → B → C relationships

  • shortest path reasoning

  • constraint-based queries

RAG struggles to guarantee structured reasoning consistency.

4. When interpretability is critical

Knowledge graphs provide:

  • explicit paths of reasoning

  • traceable relationships

  • explainable answers

This is important in domains like:

  • healthcare

  • finance

  • legal systems

5. When hallucination risk must be minimized

Since KG queries are deterministic, they:

  • return structured facts

  • avoid generative guesswork

RAG, by contrast, still relies on LLM generation, which can hallucinate.

When RAG is better than knowledge graphs

RAG is preferred when:

  • data is unstructured (documents, PDFs, web pages)

  • information changes frequently

  • semantic understanding of long text is needed

  • building a KG is too expensive or complex

Key differences

Data format

  • KG: structured triples (entity–relation–entity)

  • RAG: unstructured text chunks

Reasoning style

  • KG: logical, graph traversal

  • RAG: semantic similarity + generation

Flexibility

  • KG: limited to predefined schema

  • RAG: highly flexible across domains

Maintenance

  • KG: expensive to build and maintain

  • RAG: easier to scale with new data

Simple real-world example

  • Knowledge Graph use case:
    “Is Company A a subsidiary of Company B?”
    → KG directly stores this relationship.

  • RAG use case:
    “Summarize what Company A does based on its reports and articles.”
    → RAG retrieves text and generates a summary.

Short interview-ready answer

Knowledge graphs should be used instead of RAG when the task requires structured, relationship-based reasoning over well-defined and stable data. They are ideal for scenarios needing precise entity relationships, multi-hop logical queries, and explainable results, such as in healthcare, finance, or ontologies. In contrast, RAG is better suited for unstructured, dynamic, or text-heavy knowledge sources.

When is RAG unnecessary, and what are simpler alternatives?

Definition

RAG (Retrieval-Augmented Generation) is used to bring external knowledge into an LLM at inference time. It is unnecessary when the model can already answer correctly using internal knowledge, reasoning, or simpler systems without retrieval.

When RAG is unnecessary

1. Pure reasoning or computation tasks

If the problem does not require external knowledge, RAG adds no value.
Examples: math problems, logic puzzles, code debugging.

2. Information already known by the model

If the question is about stable, general knowledge that the model was trained on, retrieval is redundant.
Examples: basic science concepts, general definitions.

3. Small or fixed knowledge base

If the dataset is small enough to fit in prompts or rules, retrieval systems are overkill.

4. Structured queries with exact answers

If data is stored in structured systems like SQL databases or APIs, direct querying is better than retrieval.

5. Simple or template-based tasks

Tasks like formatting text, classification, or rewriting do not need external knowledge.

6. Strict latency or cost constraints

RAG adds retrieval time and infrastructure cost, so it is unnecessary for lightweight or real-time systems.

Simpler alternatives to RAG

1. Prompt engineering

Use carefully designed prompts with examples instead of retrieval.

2. Fine-tuning

Train the model on domain-specific data so it already knows the required patterns.

3. Tool or API calls

Use external tools directly such as calculators, databases, or weather APIs instead of document retrieval.

4. In-context learning

Provide relevant information directly inside the prompt when the knowledge is small.

5. Structured queries (SQL or key-value systems)

Use direct database queries when data is structured and exact.

Simple real-world example

If you ask “What is 2 + 2?”, using RAG to retrieve documents is unnecessary because the answer comes directly from reasoning. But if you ask “What are the latest policy updates in a company document?”, then RAG is useful because the information is external and dynamic.

Short interview-ready answer

RAG is unnecessary when the task can be solved using internal model knowledge, reasoning, or structured systems without needing external retrieval. In such cases, simpler alternatives like prompt engineering, fine-tuning, tool or API calls, in-context learning, or direct database queries are more efficient and cost-effective.

Pros and cons of tightly vs loosely coupled retriever-generator systems?

Definition

In RAG systems, the retriever and generator can be designed in two ways: tightly coupled (end-to-end or jointly optimized) or loosely coupled (independent modules connected via an interface).

Tightly coupled systems

Definition

Retriever and generator are jointly trained or strongly integrated, often sharing optimization objectives or gradients.

Pros

  • Better end-to-end optimization for final answer quality

  • Retriever is directly trained to help generation task

  • Reduced mismatch between retrieved content and LLM usage

  • Can improve relevance of retrieved documents for specific tasks

  • More coordinated reasoning between retrieval and generation

Cons

  • Hard to debug because components are interdependent

  • Less modular and harder to replace individual parts

  • Expensive and complex to train end-to-end

  • Poor flexibility for changing retrievers or models

  • Harder to scale across multiple domains or systems

Loosely coupled systems

Definition

Retriever and generator are independent modules connected via a pipeline (retrieve first, then generate).

Pros

  • Easy to design, implement, and scale

  • Modular components can be swapped independently

  • Easier debugging and monitoring of each stage

  • Flexible across different domains and use cases

  • Supports multiple retrievers or generators easily

Cons

  • No joint optimization between retrieval and generation

  • Possible mismatch between retrieved context and model needs

  • Retriever may return irrelevant or suboptimal results

  • Limited feedback loop for improving retrieval quality

  • Performance depends heavily on retriever quality

Key comparison

Tightly coupled

  • Optimized together

  • High performance potential

  • Low flexibility

  • High complexity

Loosely coupled

  • Independent components

  • Easy to build and maintain

  • Highly flexible

  • Lower joint optimization quality

Simple real-world example

Tightly coupled system is like a chef and farmer working as one team, planning crops specifically for the chef’s recipes.
Loosely coupled system is like a chef buying ingredients from a market and cooking independently of how they were grown.

Short interview-ready answer

Tightly coupled retriever-generator systems jointly optimize retrieval and generation, leading to better task-specific performance but increased complexity and reduced modularity. Loosely coupled systems keep retrieval and generation independent, making them easier to build, scale, and maintain, but they may suffer from suboptimal alignment between retrieved content and generation needs.

What is proposition-based chunking, and when is it better than traditional chunking?

Definition

Proposition-based chunking is a text-splitting method where content is divided into meaningful semantic units (propositions) instead of fixed sizes like characters, words, or sentences.
A proposition is a single idea or factual statement that can stand alone in meaning.

Traditional chunking splits text mechanically (by length or sentences), while proposition-based chunking splits based on meaning.

Why it is used

It is used in NLP systems (especially RAG pipelines and LLM-based search systems) to improve retrieval accuracy by ensuring each chunk contains a complete, meaningful idea instead of partial or mixed information.

How it works (step-by-step)

  1. Input text is analyzed using an NLP model or rules.

  2. The text is broken into individual propositions (atomic ideas).

  3. Each proposition is checked to ensure it is self-contained and meaningful.

  4. These propositions are stored as separate chunks.

  5. During retrieval, only the most relevant propositions are returned to the model.

Key components

  • Proposition extraction (semantic splitting)

  • NLP/LLM-based parsing

  • Context preservation within each unit

  • Embedding generation per proposition

  • Retrieval system (for search or RAG)

Advantages

  • Higher retrieval accuracy

  • Better context preservation

  • Reduces irrelevant or noisy text in chunks

  • Improves LLM answer quality

  • Works well for complex documents (research papers, legal text)

Disadvantages

  • More computationally expensive

  • Requires NLP/LLM preprocessing

  • Harder to implement than simple chunking

  • May over-split text if not tuned properly

Simple real-world example

Traditional chunking:
A paragraph is split every 200 words, so one chunk may contain multiple unrelated ideas.

Proposition-based chunking:
A paragraph like:
“The Earth orbits the Sun. It takes 365 days to complete one revolution.”
is split into:

  • The Earth orbits the Sun

  • The Earth takes 365 days to orbit the Sun

Each chunk is a complete idea.

When it is better than traditional chunking

  • When dealing with complex or dense documents (research papers, legal docs, technical manuals)

  • When high retrieval accuracy is important (RAG systems, search engines)

  • When sentences contain multiple ideas that need separation

  • When context clarity matters more than processing speed

It is not ideal when:

  • You need fast, lightweight preprocessing

  • Data is already simple and well-structured

  • Resources are limited

Short interview-ready answer

Proposition-based chunking is a semantic text-splitting technique where content is divided into individual meaningful ideas called propositions, rather than fixed-size or sentence-based chunks. It is commonly used in NLP and RAG systems to improve retrieval quality by ensuring each chunk contains a complete and self-contained idea. It works by analyzing text, extracting atomic statements, and storing each as a separate chunk for embedding and retrieval. It is better than traditional chunking when dealing with complex or information-dense documents because it improves context accuracy and reduces noise, though it is more computationally expensive.

  1. Input text or conversation history is collected

  2. Irrelevant or redundant information is identified

  3. Key facts, entities, and relationships are extracted

  5. The compressed context is passed to the model instead of full raw text

Key components

  • Information filtering (removing noise)

  • Summarization or abstraction

  • Relevance scoring (what matters most)

  • Memory management (keeping only useful history)

  • Token optimization for LLM input

Advantages

  • Reduces token usage and cost

  • Improves response speed

  • Risk of losing important details

When it becomes especially important

  • Real-time AI assistants with limited context windows

Context compression is important as prompt sizes grow because large inputs exceed token limits, increase cost, and introduce irrelevant noise that reduces model accuracy. How does hierarchical retrieval (summary → chunk → sentence) work?

Definition

Hierarchical retrieval is a multi-stage information retrieval strategy where data is searched in layers of increasing granularity: first at the summary level, then at the chunk level, and finally at the sentence level. It is commonly used in RAG (Retrieval-Augmented Generation) systems to efficiently find precise answers from large documents.

Why it is used

It helps balance:

  • Speed (avoid scanning entire documents in detail)

  • Accuracy (zoom into only relevant sections)

  • Scalability (handle large knowledge bases efficiently)

Instead of searching everything at once, the system progressively narrows down the search space.

How it works (step-by-step)

1. Summary-level retrieval

  • The system first searches high-level summaries of documents

  • These summaries represent the main idea of each document or section

  • Irrelevant documents are filtered out early

2. Chunk-level retrieval

  • From selected summaries, the system moves to related document chunks

  • Chunks are medium-sized text blocks (e.g., paragraphs)

  • Only the most relevant chunks are kept

3. Sentence-level retrieval

  • Within the chosen chunks, the system retrieves specific sentences

  • This is the most fine-grained level

  • Final answer is built using precise, relevant sentences

Key components

  • Document summarization layer

  • Chunking system (paragraph-level segmentation)

  • Embedding-based similarity search

  • Ranking mechanism at each stage

  • Query refinement across levels

Advantages

  • Highly efficient for large datasets

  • Reduces unnecessary computation

  • Improves retrieval precision

  • Scales well for enterprise knowledge bases

  • Minimizes noise in final outputs

Disadvantages

  • More system complexity

  • Extra preprocessing required (summaries + chunks)

  • Risk of losing detail in early filtering stages

  • Higher engineering overhead compared to flat retrieval

Simple real-world example

If you search “refund policy for international orders”:

  • Summary level: selects documents about “shopping policies”

  • Chunk level: narrows to “refund and return section”

  • Sentence level: finds exact rule like:
    “International orders can be refunded within 15 days of delivery.”

When it is useful

  • Large enterprise document systems

  • Legal or compliance databases

  • Research knowledge bases

  • Customer support systems with huge FAQs

  • RAG systems with thousands of documents

Short interview-ready answer

Hierarchical retrieval is a multi-stage search approach used in RAG systems where information is retrieved progressively from summary to chunk to sentence level. First, document summaries are searched to filter relevant sources. Then, relevant chunks are selected from those documents. Finally, the most precise sentences are retrieved from those chunks. This method improves efficiency and accuracy by reducing the search space step-by-step, making it ideal for large-scale document retrieval systems.

What is sentence window retrieval, and how does it differ from parent-child chunking?

Definition

Sentence window retrieval is a retrieval technique where, instead of returning only a single matched sentence, the system also retrieves a fixed “window” of surrounding sentences (before and after it). This helps preserve local context so the answer is more complete and meaningful.

It is commonly used in RAG systems to avoid losing important context that exists outside a single sentence.

Why it is used

Single sentences often lack enough context to be correctly interpreted. Sentence window retrieval solves this by:

  • Preserving nearby context

  • Improving answer coherence

  • Reducing misinterpretation of isolated sentences

How it works (step-by-step)

  1. The document is split into individual sentences

  2. Each sentence is embedded and indexed

  3. When a query matches a sentence, the system identifies the best-matching sentence

  4. Instead of returning only that sentence, it retrieves a fixed window (e.g., ±2 sentences) around it

  5. The expanded context is passed to the LLM for answering

Key components

  • Sentence-level indexing

  • Embedding-based similarity search

  • Window size parameter (e.g., 1–3 surrounding sentences)

  • Context expansion logic

Advantages

  • Preserves local context around key facts

  • Reduces ambiguity from isolated sentences

  • Simple and lightweight compared to complex hierarchies

  • Improves answer quality in short-form retrieval

Disadvantages

  • May include irrelevant nearby sentences

  • Fixed window size may not always match semantic boundaries

  • Less efficient than more structured retrieval methods for large documents

Simple real-world example

If a document says:

  1. “The company offers a 30-day refund policy.”

  2. “Refunds are processed within 5–7 business days.”

  3. “Shipping fees are non-refundable.”

If sentence 2 is retrieved, sentence window retrieval might also include sentences 1 and 3 to provide full policy context.

How does indexing hypothetical questions improve retrieval?

Definition

Indexing hypothetical questions is a retrieval strategy where, instead of only storing raw document text, the system also generates possible user-like questions (hypotheticals) for each document or chunk and indexes those questions along with the content.

This helps the system match user queries more effectively, even when the query is phrased differently from the original text.

Why it is used

Traditional keyword or embedding search often fails when:

  • The user’s question is phrased differently from the document

  • The document is descriptive but not question-shaped

  • The query is abstract or intent-based

Hypothetical question indexing improves semantic alignment between user intent and stored knowledge.

How it works (step-by-step)

  1. The document or chunk is analyzed

  2. An LLM generates multiple possible questions a user might ask about that content

    • These are called hypothetical questions

  3. Both the original content and generated questions are embedded

  4. All embeddings are stored in the index

  5. At query time, the user’s question is matched against both:

    • Original text embeddings

    • Hypothetical question embeddings

  6. The best-matching content is retrieved

Key components

  • LLM-based question generation

  • Dual indexing (text + hypothetical questions)

  • Embedding similarity search

  • Query-to-intent matching

  • Retrieval reranking

Advantages

  • Better semantic matching than keyword search

  • Handles paraphrased or indirect queries well

  • Improves recall in RAG systems

  • Bridges gap between document language and user language

  • Useful for unstructured or explanatory content

Disadvantages

  • Increased storage (extra embeddings for questions)

  • More preprocessing cost (LLM generation step)

  • Risk of generating irrelevant or low-quality questions

  • Can introduce noise if poorly tuned

Simple real-world example

Document text:
“Users can reset their password by clicking ‘Forgot Password’ and receiving a reset link via email.”

Hypothetical questions generated:

  • “How do I reset my password?”

  • “What should I do if I forget my login credentials?”

  • “How is password recovery done?”

If a user asks:
“I can’t log in, what do I do?”
The system still retrieves the correct document because the query matches a generated hypothetical question.

When it is useful

  • Customer support knowledge bases

  • FAQ systems

  • RAG systems with descriptive documents

  • Legal or policy documents

  • Education and tutorial content retrieval

Short interview-ready answer

Indexing hypothetical questions improves retrieval by generating possible user queries for each document and storing them alongside the original content. This helps bridge the gap between how information is written and how users actually ask questions. During retrieval, the system matches user queries against both the document text and these synthetic questions, improving semantic alignment and recall. It is especially useful in RAG systems and FAQ-style applications where user phrasing often differs from document wording.

What is late chunking, and why does it preserve more context?

Definition

Late chunking is a text processing strategy where documents are first processed as a whole (or large continuous text), and chunking is performed after embedding or representation learning, rather than splitting the text into chunks at the beginning.

In contrast, traditional (early) chunking splits text first and then processes each chunk separately.

Why it is used

Late chunking is used to preserve global context and reduce the loss of meaning that often happens when documents are split too early. It is especially useful in embedding-based retrieval systems (like RAG pipelines).

How it works (step-by-step)

  1. The full document is passed into an embedding model or encoder

  2. The model processes the entire context together, capturing global relationships

  3. After encoding, the representation is divided into chunks (based on token spans, attention windows, or embeddings)

  4. Each chunk inherits context-aware embeddings derived from the full document

  5. These context-rich chunks are stored for retrieval

  6. At query time, relevant chunks are retrieved and used for generation

  • Chunk boundaries do not break semantic understanding early

  • More complex implementation

  • RAG systems requiring high accuracy

  • Technical documentation

  • Cases where pronoun/reference resolution matters

  • Knowledge bases with dense contextual relationships

Late chunking is a technique where a full document is first processed or embedded as a whole, and only then is it divided into chunks for storage and retrieval. Incremental updating of a RAG (Retrieval-Augmented Generation) index means adding, modifying, or removing documents in the existing vector database or search index without reprocessing the entire dataset from scratch.

  • Reprocessing metadata

2. Identify affected chunks

  • Only the modified documents are re-chunked

  • Unchanged documents remain untouched

3. Generate embeddings for new/updated chunks

  • New chunks are passed through the embedding model

  • Old embeddings are reused if unchanged

4. Update the vector database

  • Insert new embeddings

  • Replace updated embeddings

  • Remove deleted entries

5. Maintain index consistency

  • Metadata is updated (timestamps, IDs, versions)

  • Optional re-ranking indexes are adjusted

Key components

  • Change detection system (file diff, timestamps, CDC pipelines)

  • Chunk-level tracking (unique IDs per chunk)

  • Embedding service

  • Vector database (e.g., FAISS, Pinecone, Weaviate)

  • Metadata/version control layer

Advantages

  • Faster updates (no full rebuild needed)

  • Lower compute cost

  • Near real-time knowledge updates

  • Scales well for large systems

  • Minimal disruption to existing queries

Disadvantages

  • Requires careful version tracking

  • Risk of stale or duplicate embeddings if not managed well

  • Versioned embeddings for rollback safety

When it is useful

  • Large enterprise knowledge bases

  • Live documentation systems (APIs, product docs)

  • News or frequently changing content

  • Chatbots with continuously updated knowledge

  • Real-time RAG applications

Short interview-ready answer

A RAG index can be updated incrementally by detecting changes in the dataset and only reprocessing the affected documents or chunks instead of rebuilding the entire index. New or modified chunks are re-embedded and inserted into the vector database, while unchanged embeddings are reused. Deleted content is removed and metadata is updated accordingly. This approach reduces compute cost and enables faster, near real-time updates while maintaining an up-to-date retrieval system.

What are multi-vector retrieval techniques, and why are they effective?

Definition

Multi-vector retrieval techniques are methods in which a single document (or chunk) is represented by multiple embedding vectors instead of just one. Retrieval is then performed by comparing the query against all (or several) vectors associated with that document.

This contrasts with traditional retrieval, where each chunk has only one vector representation.

Why it is used

A single embedding often cannot capture all aspects of a complex document. Multi-vector retrieval improves recall and accuracy by representing different parts or perspectives of the same text separately.

How it works (step-by-step)

  1. A document is split into multiple meaningful parts (sentences, phrases, or tokens)

  2. Each part is embedded separately into its own vector

  3. These multiple vectors are stored as the representation of the same document

  4. At query time, the query is also embedded into a vector

  5. The system computes similarity between the query vector and all document vectors

  6. Results are aggregated (e.g., max similarity, weighted sum, or voting)

  7. The most relevant document or passage is retrieved

Key components

  • Multiple embeddings per document

  • Fine-grained text segmentation

  • Similarity scoring functions (max pooling, mean pooling, etc.)

  • Vector database support for multi-vector storage

  • Aggregation strategy for final ranking

Why it is effective

Multi-vector retrieval works better because:

  • Captures fine-grained meaning (different vectors represent different ideas)

  • Improves recall (more chances to match a query)

  • Handles complex or multi-topic documents better

  • Reduces information loss from single-vector compression

  • Improves semantic matching for long or dense text

Advantages

  • Higher retrieval accuracy

  • Better handling of multi-topic documents

  • Improved semantic coverage

  • More robust to query variations

  • Works well in dense retrieval systems (like RAG)

Disadvantages

  • Higher storage cost (more vectors per document)

  • Increased computation during search

  • More complex indexing and ranking logic

  • Slower than single-vector retrieval in large-scale systems

Simple real-world example

A document contains:

  • “The phone has a great camera.”

  • “It also has a long-lasting battery.”

  • “The display is OLED and very bright.”

Instead of one embedding, multi-vector retrieval creates separate vectors for each feature.
So if a user searches:
“best phone battery life”
the system matches directly with the battery-related vector, even if other parts are unrelated.

When it is useful

  • Product search systems (e-commerce)

  • Technical documentation retrieval

  • Long research papers

  • Multi-topic enterprise knowledge bases

  • RAG systems requiring high recall and precision

Short interview-ready answer

Multi-vector retrieval techniques represent each document using multiple embedding vectors instead of a single vector, with each vector capturing different parts or aspects of the content. During retrieval, the query is compared against all vectors, and results are aggregated to determine relevance. This improves effectiveness because it preserves fine-grained semantic information, handles multi-topic documents better, and increases recall compared to single-vector retrieval. It is especially useful in RAG systems, product search, and large knowledge bases where documents contain diverse information.

How does Reciprocal Rank Fusion (RRF) combine retrieval results?

Definition

Reciprocal Rank Fusion (RRF) is an algorithm used to combine results from multiple retrieval systems (or ranking methods) into a single final ranked list. It does this by giving higher importance to items that appear near the top across different ranked lists, rather than relying on raw similarity scores.

Why it is used

Different retrieval systems (BM25, vector search, keyword search, etc.) often produce different rankings. RRF helps:

  • Combine strengths of multiple retrievers

  • Avoid dependence on raw score scales (which may not be comparable)

  • Improve overall ranking quality and robustness

How it works (step-by-step)

  1. Run multiple retrieval methods on the same query

    • Example: keyword search, dense vector search, hybrid search

  2. Each method produces a ranked list of results

  3. For each document, assign a score based on its rank in each list

  4. Use the RRF formula to compute a combined score:

\text{RRF}(d) = \sum_{i=1}^{N} \frac{1}{k + r_i(d)}

Where:

  • ( r_i(d) ) = rank of document ( d ) in the ( i^{th} ) system

  • ( k ) = constant (commonly 60) to reduce impact of very top ranks

  • ( N ) = number of retrieval systems

  1. Sum scores across all systems

  2. Sort documents by final RRF score

  3. Return top-ranked results

Key components

  • Multiple retrievers (BM25, dense retrieval, hybrid systems)

  • Rank-based scoring (not raw similarity scores)

  • Fusion function (reciprocal rank formula)

  • Aggregation across systems

Why it is effective

RRF works well because:

  • It is robust to different scoring scales

  • It rewards consistency across retrievers

  • It reduces bias from any single retrieval method

  • It boosts items that appear frequently in top ranks

  • It improves recall and precision together

Advantages

  • Simple and effective to implement

  • No need to normalize scores across systems

  • Works well with heterogeneous retrieval methods

  • Strong performance in real-world search systems

  • Stable even when individual retrievers are noisy

Disadvantages

  • Ignores actual similarity scores (only ranks matter)

  • Sensitive to choice of constant ( k )

  • May not fully exploit high-confidence score differences

  • Requires multiple retrieval systems for best benefit

Simple real-world example

Imagine three search systems return results:

  • System A: Doc X (1st), Doc Y (2nd)

  • System B: Doc Y (1st), Doc Z (2nd)

  • System C: Doc Y (1st), Doc X (3rd)

Even if scores differ, RRF will rank Doc Y highest because it consistently appears at the top across systems.

When it is used

  • Hybrid search systems (keyword + vector search)

  • RAG pipelines combining multiple retrievers

  • Enterprise search engines

  • Recommendation systems combining signals

  • Open-domain QA systems

Short interview-ready answer

Reciprocal Rank Fusion (RRF) is a technique used to combine results from multiple retrieval systems by ranking documents based on their positions rather than raw scores. It assigns each document a score using the reciprocal of its rank in each system and sums these scores across all systems. Documents that appear higher in multiple ranked lists receive higher final scores. RRF is effective because it is simple, does not require score normalization, and improves robustness by combining the strengths of different retrieval methods.

When should retrieved content be summarized before being sent to the model?

Retrieved content is often:

  • Too large for the context window

  • Noisy or repetitive

  • Only partially relevant to the query

  • Spread across multiple sources

Summarization helps keep only the most useful information while staying within token limits.

When you should summarize retrieved content

1. When retrieved data exceeds the context window

If the combined retrieved chunks do not fit into the model’s token limit, summarization is necessary to compress them without losing key meaning.

2. When retrieval returns many loosely relevant results

If top-k retrieval includes several partially relevant documents, summarizing helps merge them into a single coherent context.

3. When documents are long but query is narrow

If the query targets a specific fact (e.g., “refund policy duration”), full documents are unnecessary—only relevant parts should be summarized.

4. When using multi-document retrieval

When results come from multiple sources, summarization helps:

  • Remove duplication

  • Resolve conflicting information

  • Create a unified view

5. When latency or cost is a concern

Summarizing reduces token usage, which:

  • Lowers API cost

  • Speeds up inference

  • Improves system efficiency

6. When improving signal-to-noise ratio

If retrieved chunks contain:

  • Boilerplate text

  • Irrelevant surrounding context

  • Repeated explanations

Summarization improves focus on key facts.

When you should NOT summarize

1. When precision details are critical

For tasks like:

  • Legal clauses

  • Code retrieval

  • Exact numeric values
    Summarization may remove important details.

2. When chunks are already small and clean

If retrieval already returns highly focused sentence-level chunks, summarization may be unnecessary.

3. When reasoning requires full context

Complex reasoning tasks often benefit from full text instead of compressed versions.

  • Retrieval step (vector/BM25/hybrid)

Advantages

  • Fits large retrieved data into limited context

  • Reduces noise and redundancy

  • Enhances response speed and cost efficiency

  • Enables multi-document reasoning

Disadvantages

  • Risk of losing fine-grained details

  • Possible hallucination during summarization

  • Additional computation step

  • May reduce factual precision if poorly designed

Retrieved documents include:

Retrieved content should be summarized before being sent to the model when the context is too large, noisy, or comes from multiple partially relevant sources. What is token budget management, and how do you balance context vs generation?

Definition

Token budget management is the practice of allocating and controlling the total number of tokens used in an LLM request between:

  • Input context tokens (system prompt, user query, retrieved documents, memory)

  • Output tokens (model’s generated response)

Since LLMs have a fixed context window, you must carefully balance how much space is used for context versus how much is left for generation.

Why it is important

Without token budget management:

  • Important context may be truncated

  • Responses may be cut off mid-generation

  • Retrieval results may overwhelm the prompt

  • Costs and latency increase unnecessarily

  • Model performance becomes unstable due to noise or missing information

How it works (step-by-step)

1. Define total token limit

Each model has a maximum context window, for example:

  • 8K, 32K, 128K tokens, etc.

2. Reserve output budget first

A portion is reserved for the response:

  • Example: 2K tokens for output
    This ensures the model can fully answer without truncation.

3. Allocate remaining budget to input

The remaining tokens are divided among:

  • System instructions

  • User query

  • Conversation history

  • Retrieved documents (RAG context)

4. Prioritize input sources

Not all context is equally important. Typical priority:

  1. System prompt (highest priority)

  2. User query

  3. Retrieved relevant chunks

  4. Conversation history

  5. Low-signal or redundant data (lowest priority, often dropped)

5. Apply compression if needed

If input exceeds budget:

  • Summarize long documents

  • Trim chat history

  • Remove redundant retrieval results

  • Use chunk ranking (top-k selection)

6. Dynamic adjustment during runtime

Advanced systems adjust budgets based on:

  • Query complexity

  • Expected answer length

  • Retrieval size

  • Model behavior (e.g., reasoning-heavy vs factual tasks)

Key components

  • Token counter (precise token estimation)

  • Budget allocator (input vs output split)

  • Ranking system (decides what stays or gets dropped)

  • Compression layer (summarization, deduplication)

  • Retrieval limiter (top-k control)

How to balance context vs generation

1. Reserve output tokens first

Always guarantee enough space for the answer.
If not, the model may truncate responses.

2. Use relevance-based context selection

Only include:

  • Top-k most relevant chunks

  • High-similarity embeddings

  • Query-relevant conversation history

3. Compress instead of dropping critical data

Instead of removing important context:

  • Summarize it

  • Extract key facts

  • Convert to structured form

4. Limit retrieval aggressively

  • Reduce top-k results

  • Use reranking before injection

  • Filter low-score chunks early

5. Prefer structured context over raw text

Structured formats (bullet facts, JSON-like summaries) use fewer tokens and improve clarity.

6. Use hierarchical memory

  • Short-term memory: last few turns

  • Long-term memory: summarized history
    This prevents context explosion in conversations.

Advantages

  • Prevents context overflow errors

  • Improves response reliability

  • Reduces cost and latency

  • Ensures complete output generation

  • Improves focus on relevant information

Disadvantages

  • Requires careful tuning

  • Over-compression can lose important details

  • Hard to balance across different tasks

  • Needs continuous monitoring in production systems

Simple real-world example

If a model has a 4,000-token limit:

  • Reserve 1,000 tokens for response

  • 3,000 tokens left for context

If retrieval returns 5,000 tokens of documents:

  • Keep only top 2,000 most relevant tokens

  • Summarize remaining critical parts

  • Drop low-relevance chunks

This ensures the model still has space to generate a full answer.

Short interview-ready answer

Token budget management is the process of allocating the limited token capacity of an LLM between input context and output generation. It ensures that enough space is reserved for the model’s response while prioritizing the most relevant system prompts, queries, and retrieved context. Balancing is achieved through techniques like relevance-based filtering, retrieval limiting, and context compression. This prevents context overflow, reduces noise, and ensures the model produces complete and accurate responses within its token limits.

How would you design a large-scale RAG system for thousands of daily queries?

Definition

A large-scale RAG (Retrieval-Augmented Generation) system is an architecture that combines a retrieval engine (search over a knowledge base) with a language model to answer thousands or more user queries efficiently, accurately, and at low latency.

At scale, the main challenge is not just retrieval quality, but also:

  • Latency control

  • Cost optimization

  • Index freshness

  • Concurrency handling

  • Context management

High-level architecture

A production RAG system typically has these layers:

  • Ingestion pipeline

  • Indexing layer (vector + keyword + metadata)

  • Query router

  • Retrieval layer (hybrid + reranking)

  • Context builder (compression + selection)

  • LLM generation layer

  • Caching + observability layer

How it works (end-to-end design)

1. Data ingestion pipeline

  • Documents are collected from sources (DBs, APIs, files)

  • Cleaned and normalized

  • Split into chunks (often hybrid / semantic chunking)

  • Enriched with metadata (source, timestamp, tags)

  • Embeddings generated and stored

Key goal: keep ingestion async and scalable using queues (Kafka, SQS, etc.).

2. Indexing layer (multi-representation)

At scale, a single index is not enough. Typically:

  • Vector index (semantic search)

  • Keyword index (BM25 / lexical search)

  • Metadata index (filters like domain, time, user group)

This enables hybrid retrieval for better accuracy.

3. Query routing layer

Before retrieval, system decides:

  • Is this a factual query?

  • Does it need retrieval or can LLM answer directly?

  • Which index should be used?

Techniques:

  • Intent classification

  • Query complexity scoring

  • Lightweight LLM router

4. Retrieval layer (hybrid search)

For each query:

  1. Run vector search

  2. Run keyword search

  3. Merge results

Then apply:

  • Reciprocal Rank Fusion (RRF)

  • Metadata filtering

  • Deduplication

This ensures both semantic and exact-match coverage.

5. Reranking layer

Top results (e.g., top 50) are passed to a reranker:

  • Cross-encoder model or LLM

  • Scores relevance more precisely than embeddings

Output is reduced to top 5–10 chunks.

6. Context construction (critical for scale)

Before sending to LLM:

  • Remove redundancy

  • Compress long chunks (summarization)

  • Apply token budget management

  • Optionally use sentence window or parent-child expansion

Goal: maximize signal, minimize tokens.

7. LLM generation layer

Final prompt includes:

  • User query

  • Retrieved context

  • Conversation memory (if multi-turn)

  • System instructions

The model generates the final response.

8. Caching layer (very important at scale)

To handle thousands of queries efficiently:

  • Query-result cache (exact match)

  • Semantic cache (similar queries)

  • Retrieval cache (top-k results reused)

  • Embedding cache

This significantly reduces cost and latency.

9. Observability and monitoring

Production systems track:

  • Retrieval accuracy

  • Latency per stage

  • Token usage

  • Hallucination rate

  • Cache hit ratio

  • User feedback signals

Key scaling strategies

1. Hybrid retrieval instead of pure vector search

Improves recall and robustness at scale.

2. Precomputation everywhere possible

  • Pre-embed documents

  • Precompute summaries

  • Pre-generate hypothetical questions

3. Async ingestion pipeline

Ensures indexing does not block query flow.

4. Aggressive filtering early

Reduce load before expensive steps:

  • metadata filters

  • query routing

  • top-k pruning

5. Distributed infrastructure

  • Sharded vector databases

  • Load-balanced retrieval services

  • Horizontal scaling of APIs

6. Multi-stage retrieval pipeline

Instead of one expensive search:

  • Fast recall stage (cheap retrieval)

  • Precision stage (reranking)

  • Context refinement stage

Advantages

  • Handles high query volume efficiently

  • Scales horizontally with infrastructure

  • Balances accuracy and latency

  • Cost-effective at production scale

  • Flexible across multiple domains

Disadvantages

  • Complex system design

  • Hard to debug retrieval failures

  • Requires careful tuning of each stage

  • Infrastructure cost can grow quickly

  • Risk of stale or inconsistent indexes

Simple real-world example

A customer support RAG system for an e-commerce company:

  • 10,000 queries/day

  • Thousands of product + policy documents

Flow:

  1. Query: “Can I return international orders?”

  2. Router identifies as policy question

  3. Hybrid retrieval finds return policy docs

  4. Reranker selects most relevant section

  5. Context is compressed

  6. LLM generates final answer

Caching ensures repeated queries are answered instantly.

Short interview-ready answer

A large-scale RAG system is designed using a multi-layer architecture that includes ingestion pipelines, hybrid indexing (vector + keyword), query routing, retrieval with reranking, and context compression before generation. To handle thousands of daily queries efficiently, it uses caching, distributed vector databases, async processing, and multi-stage retrieval pipelines. The system balances accuracy and scalability by filtering and ranking results early, compressing context to fit token limits, and optimizing cost and latency through reuse and precomputation.

What is semantic caching, and how does it differ from simple caching?

Definition

Semantic caching is a caching technique where results are stored and retrieved based on meaning similarity rather than exact matches. Instead of looking for identical queries, it finds past queries that are semantically similar and reuses their responses.

Simple caching (or exact-match caching) stores and retrieves results only when the input query exactly matches a previous one.

Why it is used

In LLM and RAG systems:

  • Users ask the same thing in different ways

  • Exact-match caching has low hit rates

  • Many queries are paraphrases of each other

Semantic caching improves efficiency by reusing responses for similar meaning, not just identical text.

How semantic caching works (step-by-step)

  1. User submits a query

  2. Query is converted into an embedding vector

  3. The cache stores embeddings of past queries + their responses

  4. The system compares the new query embedding with cached embeddings

  5. If similarity exceeds a threshold:

    • Cache hit → return stored response

  6. Otherwise:

    • Perform full retrieval + LLM generation

    • Store new query + response in cache

How simple caching works (step-by-step)

  1. User submits a query

  2. System checks if the exact string exists in cache

  3. If exact match is found:

    • Return stored response

  4. If not:

    • Run full pipeline and store result

Key components

Semantic caching

  • Embedding model (for query representation)

  • Vector similarity search (cosine similarity, dot product)

  • Similarity threshold

  • Cache store with embeddings + responses

  • Optional metadata filtering

Simple caching

  • Hash-based key lookup

  • Exact string matching

  • Key-value store (Redis, in-memory cache)

Advantages of semantic caching

  • Much higher cache hit rate

  • Reduces redundant LLM calls

  • Saves cost and latency

  • Handles paraphrased queries

Disadvantages

  • Requires embedding computation

  • More complex infrastructure

  • Risk of incorrect matches if similarity threshold is poorly tuned

  • Needs careful cache eviction and freshness control

Simple real-world example

User A asks:
“What is semantic search?”

Later, User B asks:
“Explain meaning-based search in NLP”

Simple caching:

  • Cache miss (different text)

Semantic caching:

  • Cache hit (same meaning)

  • Returns previous answer instantly

When it is useful

  • LLM-based chatbots

  • RAG systems with repeated queries

  • Customer support systems

  • Enterprise knowledge assistants

  • High-traffic APIs where many queries are paraphrased

Short interview-ready answer

Semantic caching is a caching technique that retrieves previously generated responses based on semantic similarity between queries rather than exact text matches. It uses embeddings to compare the meaning of a new query with stored queries and returns a cached response if similarity exceeds a threshold. Unlike simple caching, which only works for identical queries, semantic caching handles paraphrased inputs, significantly improving cache hit rates, reducing LLM calls, and lowering latency in large-scale RAG and conversational systems.

What is the cold start problem in RAG, and how can it be solved early on?

Definition

The cold start problem in Retrieval-Augmented Generation (RAG) happens when the system has very little or no existing data, embeddings, user queries, or feedback history to effectively retrieve relevant documents for generating accurate answers.

In simple terms, the system is “new” and does not yet know what to retrieve or how to retrieve it well.

Why it is used / Why it happens

RAG systems rely on:

  • A knowledge base (documents)

  • Chunked and embedded data

  • A vector database for retrieval

  • Query–document relevance patterns

At the beginning:

  • There is limited or no indexed data

  • Embeddings may be poorly optimized

  • No user interaction history exists

  • Retrieval quality is weak

This leads to poor or irrelevant responses.

How it works (step-by-step in cold start phase)

  1. User asks a query

  2. System tries to retrieve relevant documents

  3. Few or no relevant chunks are found

  4. LLM generates answer with weak or missing context

  5. Output quality is inconsistent or generic

Key components affected

  • Document ingestion pipeline

  • Embedding model quality

  • Vector database index

  • Retrieval mechanism (vector / hybrid search)

  • Ranking system

  • Feedback loop (usually not available initially)

How to solve it early on

  1. Bootstrap with pre-existing data
    Use public datasets, FAQs, manuals, or domain documents to initialize the knowledge base.

  2. Use strong pre-trained embedding models
    Helps generalize well even with limited domain-specific data.

  3. Hybrid search (BM25 + vector search)
    Combines keyword matching with semantic search for better early retrieval.

  4. Synthetic data generation
    Use LLMs to create question–answer pairs or document expansions.

  5. Query expansion techniques
    Rewrite or expand queries to improve retrieval coverage.

  6. Good chunking and metadata design
    Proper document splitting improves early retrieval accuracy.

  7. Start with generic models and refine later
    Use general-purpose retrieval first, then fine-tune with usage data.

  8. Logging and feedback collection from day one
    Helps gradually improve ranking and relevance.

Advantages (of addressing cold start properly)

  • Faster system usability from day one

  • Better initial user experience

  • Reduced irrelevant LLM outputs

  • Strong foundation for future improvements

Disadvantages (if not handled well)

  • Poor answer quality initially

  • Irrelevant or hallucinated responses

  • Low user trust

  • Weak retrieval performance

Simple real-world example

Imagine building a customer support chatbot for a new SaaS product.
At launch:

  • No support tickets exist

  • No user queries are logged

  • No optimized embeddings for domain-specific issues

So initially, the bot struggles to answer product-specific questions correctly until manuals, FAQs, and synthetic Q&A pairs are added.

Short interview-ready answer

The cold start problem in RAG occurs when the system has insufficient data, embeddings, or user feedback to retrieve relevant context effectively, leading to poor or generic responses. It is typically solved by bootstrapping with existing documents, using strong pre-trained embedding models, applying hybrid search, generating synthetic training data, and designing good chunking and metadata strategies to improve early retrieval quality.

How do you decide how many retrieved documents to re-rank?

Definition

In a RAG system, re-ranking is the step where a smaller set of initially retrieved documents is reordered using a more accurate but expensive model (like a cross-encoder or LLM reranker) to pick the most relevant context for generation.

The question is essentially about choosing the “top-K” documents to pass into the re-ranker.

Why it is important

Re-ranking improves accuracy but is computationally expensive.
So you must balance:

  • Retrieval quality (more documents = better chance of relevance)

  • Latency and cost (more documents = slower and more expensive processing)

How it works (step-by-step decision process)

  1. Initial retrieval happens
    A fast retriever (like vector search or hybrid search) returns top N candidates (e.g., 50–200 documents).

  2. You choose a subset for re-ranking (top-K)
    Only K documents are passed to the re-ranker.

  3. Re-ranker scores and sorts them
    The best few are selected for final context.

Key factors to decide K (number of documents)

1. Use case complexity

  • Simple factual queries → smaller K (10–20)

  • Complex reasoning or multi-hop queries → larger K (30–100)

2. Retrieval quality

  • High-quality embeddings → lower K needed

  • Noisy or weak retrieval → higher K needed to avoid missing relevant docs

3. Latency constraints

  • Real-time chat → smaller K (fast response required)

  • Offline or analytical systems → larger K allowed

4. Document diversity

If documents are very similar:

  • Smaller K is enough
    If documents are diverse:

  • Larger K helps capture variety

5. Reranker cost

Cross-encoders / LLM rerankers scale linearly with K:

  • More K = more compute cost

6. Empirical tuning

Most systems choose K based on:

  • Recall@K metrics

  • Ground-truth evaluation sets

  • A/B testing

Common practical ranges

  • Initial retrieval: 50–200 documents

  • Re-ranking input (K): 10–50 documents

  • Final context for LLM: 3–10 documents

Advantages of careful selection

  • Better balance of accuracy vs latency

  • Reduced hallucinations

  • Efficient use of compute resources

Disadvantages of poor selection

  • Too small K → relevant documents missed

  • Too large K → slow system and noise in reranking

  • Increased cost without significant gain

Simple real-world example

A customer support RAG system:

  • Initial retrieval returns 100 FAQ documents

  • Re-ranker selects top 20

  • Final LLM uses top 5

If K was too small (e.g., 5), important edge-case answers might be missed.
If K was too large (e.g., 80), response time becomes slow with no meaningful gain.

Short interview-ready answer

The number of documents chosen for re-ranking in RAG is a trade-off between retrieval recall and system efficiency. Typically, a larger initial set is retrieved, and a smaller subset (often 10–50 documents) is passed to the re-ranker based on query complexity, retrieval quality, latency constraints, and cost. The optimal value is usually determined empirically using evaluation metrics like recall and end-to-end answer quality.

What are the biggest cost drivers in RAG systems, and how can they be reduced?

Definition

In a Retrieval-Augmented Generation (RAG) system, costs come from the combination of retrieving documents, processing embeddings, reranking results, and generating responses using large language models. The “biggest cost drivers” are the parts of the pipeline that consume the most compute, storage, or API usage.

Why it is important

RAG systems often scale poorly if not designed carefully because every query may involve multiple expensive steps. Controlling cost is essential for production-grade systems.

How costs are generated (step-by-step pipeline view)

  1. Query embedding generation

  2. Vector database search

  3. Document retrieval (top-N results)

  4. Re-ranking with expensive models

  5. LLM prompt construction (context injection)

  6. LLM inference for final answer

  7. Logging, storage, and indexing overhead

Biggest cost drivers

1. LLM inference (highest cost driver)

This includes:

  • Prompt tokens (input context)

  • Output tokens (generated answer)

Why expensive:

  • Large models (GPT-style or similar) are computationally costly

  • Long contexts increase token usage significantly

2. Embedding generation and storage

Costs come from:

  • Embedding large document corpora

  • Re-embedding when models change

  • Storing high-dimensional vectors

3. Vector database operations

Includes:

  • Index creation

  • Query-time similarity search at scale

  • Storage of large vector indexes

4. Re-ranking models

Cross-encoders or LLM-based rerankers:

  • Applied on many candidate documents

  • Cost increases linearly with number of documents reranked

5. Large retrieval sets (high K values)

Retrieving too many documents increases:

  • Reranking cost

  • Prompt size

  • Latency and LLM token usage

6. Context inflation (poor chunking)

Bad chunking leads to:

  • Too many irrelevant or overlapping chunks

  • Increased token usage in prompts

How to reduce costs

1. Reduce LLM token usage

  • Use smaller context windows (only top relevant docs)

  • Summarize retrieved documents before passing to LLM

  • Use prompt compression techniques

2. Improve retrieval quality

  • Better embeddings reduce need for large top-K retrieval

  • Hybrid search (BM25 + vector) improves precision early

3. Optimize chunking strategy

  • Use meaningful chunk sizes (not too large or too small)

  • Avoid redundant overlap

  • Structure chunks using headings or semantic boundaries

4. Limit re-ranking scope

  • Rerank only top 10–50 instead of hundreds

  • Use lightweight rerankers before heavy ones

5. Cache aggressively

  • Cache embeddings for repeated queries

  • Cache retrieval results for common queries

  • Cache final responses when possible

6. Use tiered models

  • Small model for retrieval and filtering

  • Large model only for final generation

7. Reduce unnecessary retrieval calls

  • Query classification: only use RAG when needed

  • Direct LLM response for simple queries

8. Use approximate search techniques

  • ANN (Approximate Nearest Neighbor) instead of exact search

  • Reduces vector search compute cost

Advantages of cost optimization

  • Lower API and infrastructure expenses

  • Faster response times

  • Better scalability

  • More efficient resource usage

Disadvantages if not optimized

  • High operational cost

  • Slow system performance

  • Poor scalability under load

  • Wasted compute on irrelevant documents

Simple real-world example

A chatbot retrieves 200 documents per query and reranks all of them using a large model. This leads to:

  • High reranking cost

  • Large prompts sent to LLM

  • Slow response time

After optimization:

  • Only top 30 documents are reranked

  • Only top 5 are passed to LLM

  • Cached embeddings are reused

Cost drops significantly while maintaining similar answer quality.

Short interview-ready answer

The biggest cost drivers in RAG systems are LLM inference (token usage), embedding generation and storage, vector database operations, and re-ranking large sets of documents. Costs can be reduced by improving retrieval quality, optimizing chunking, limiting top-K and reranking scope, using caching, applying tiered models, and minimizing unnecessary token usage in prompts.

How do you handle multiple programming languages in a code-focused RAG system?

Definition

A code-focused RAG system retrieves and generates answers over software repositories or documentation written in multiple programming languages (e.g., Python, Java, JavaScript, C++). Handling multiple languages means ensuring the system can correctly understand, index, retrieve, and generate code across different syntaxes and ecosystems.

Why it is important

Different programming languages:

  • Have different syntax and semantics

  • Use different libraries and frameworks

  • Require language-specific context for correct retrieval

Without proper handling, the system may retrieve irrelevant code or generate incorrect solutions.

How it works (step-by-step approach)

1. Language identification during ingestion

Each file or code snippet is detected as a specific language using:

  • File extensions (.py, .java, .js)

  • Language detection models

  • Repository metadata

This metadata is stored with the chunk.

Key components and techniques

2. Language-aware chunking

Code is split based on:

  • Functions, classes, or methods

  • AST (Abstract Syntax Tree) structure instead of plain text

  • Language-specific parsing rules

This preserves semantic meaning across languages.

3. Language-tagged embeddings

Each chunk is stored with metadata like:

  • language: python / java / js

  • framework: django / spring / react

This helps filter retrieval results.

4. Unified embedding space (multi-language models)

Use embedding models trained on code from multiple languages so that:

  • Similar concepts across languages are closer in vector space
    Example: Python list.append() and Java ArrayList.add()

5. Query routing by language

Before retrieval:

  • Detect query language (explicit or inferred)

  • Route to language-specific index or filter

Example:

  • “How to create a REST API in Java?” → Java index only

6. Hybrid retrieval strategy

Combine:

  • Semantic search (vector embeddings)

  • Keyword search (function names, APIs)

  • Language filters (metadata constraints)

7. Language-aware reranking

Reranker considers:

  • Language match (high priority signal)

  • Framework relevance

  • API similarity

8. Context-aware generation

The LLM is guided using:

  • Explicit language tags in prompt

  • System instructions like “respond in Python”

  • Retrieved code snippets from same language only

Advantages

  • Higher accuracy in code retrieval

  • Reduced cross-language confusion

  • Better developer experience

  • More relevant code suggestions

Disadvantages

  • More complex indexing pipeline

  • Higher storage due to metadata

  • Requires language detection accuracy

  • Separate tuning per language may be needed

Simple real-world example

A developer asks:
“How to implement authentication in Spring Boot?”

Without language handling:

  • System might return Node.js or Python examples

With proper handling:

  • Query is detected as Java

  • Retrieval is filtered to Java/Spring Boot code

  • LLM generates correct Spring Boot authentication example

Short interview-ready answer

In a multi-language code RAG system, we handle different programming languages by detecting and tagging language during ingestion, using language-aware chunking and embeddings, and applying metadata-based filtering during retrieval. We also use unified code embedding models, query routing, and language-aware reranking to ensure that retrieved context matches the requested programming language, leading to more accurate and relevant code generation.

How can you switch embedding models in production without downtime?

Definition

Switching embedding models in production means replacing an existing text-to-vector model with a new one (for better quality, lower cost, or improved performance) without breaking the RAG system or interrupting user queries.

The main challenge is that embeddings from different models are not compatible with each other in the same vector space.

Why it is used

You may need to switch embedding models to:

  • Improve retrieval quality

  • Reduce cost

  • Upgrade to a newer model

  • Fix performance issues or bias

  • Support multi-language or domain-specific improvements

Core challenge

Embeddings are not backward compatible:

  • Old model vectors ≠ new model vectors

  • Mixing them in the same index breaks similarity search

So the system must transition carefully.

How it works (step-by-step zero-downtime strategy)

1. Use dual embedding pipelines (shadow indexing)

  • Keep old embedding model active

  • Start generating embeddings using the new model in parallel

  • Do not replace anything yet

Result:

  • Old index continues serving traffic

  • New index is built in background

2. Build a second vector index

  • Create a separate index for the new embeddings

  • Example:

    • Index A → old model

    • Index B → new model

3. Sync data incrementally

  • New documents are embedded using both models

  • Existing documents are backfilled gradually

  • Use batch processing to avoid system overload

4. Run A/B testing or shadow traffic

  • Route a small percentage of queries to the new index

  • Compare:

    • Recall

    • Precision

    • Latency

    • User satisfaction

5. Gradual traffic migration

  • Slowly increase traffic to new model:

    • 5% → 25% → 50% → 100%

  • Monitor system health during each step

6. Hybrid retrieval during transition (optional)

  • Query both indexes in parallel

  • Merge results using a ranking strategy

  • Helps avoid recall drop during migration

7. Final cutover

  • Once performance is validated:

    • Switch fully to new embedding index

    • Retire old index after validation period

Key techniques used

Dual indexing

Maintains two independent vector stores during transition.

Backfill pipelines

Re-embedding historical data without blocking production.

Feature flags

Control routing between old and new models dynamically.

Canary deployment

Gradual exposure of new embeddings to real traffic.

Advantages

  • Zero downtime migration

  • Safe rollback option

  • Performance comparison before full switch

  • Reduced risk of broken retrieval

Disadvantages

  • Higher storage cost during transition

  • Increased system complexity

  • Temporary duplication of compute resources

  • Requires careful synchronization logic

Simple real-world example

A company uses OpenAI embeddings but wants to switch to a newer, cheaper model.

Instead of replacing directly:

  • They build a new index using the new model

  • Run both systems in parallel for a week

  • Gradually shift traffic to the new index

  • Decommission the old index after validation

Short interview-ready answer

To switch embedding models in production without downtime, we use a dual-pipeline strategy where the new model generates embeddings in parallel while the old system continues serving traffic. A separate index is built for the new embeddings, and traffic is gradually migrated using canary releases or A/B testing. Once the new system is validated for quality and stability, we fully cut over and deprecate the old index, ensuring a zero-downtime transition.

How do you route queries between RAG, direct LLM responses, or other systems?

Definition

Query routing in a RAG system is the process of deciding whether a user query should be answered using RAG (retrieval-augmented generation), a direct LLM response, or external tools like APIs or databases.

Why it is used

Routing is used to avoid unnecessary retrieval, reduce cost and latency, and improve accuracy by choosing the most suitable system for each query.

How it works (step-by-step)

1. Query analysis

The system first analyzes the query to detect intent, domain, complexity, and whether external knowledge is needed.

2. Routing decision

A router (rule-based, ML model, or LLM-based classifier) decides the best path for the query.

3. Execution path selection

Based on the decision, the query is sent to RAG, direct LLM, or a tool/API system.

Key routing strategies

Direct LLM

Used for general knowledge, reasoning, and creative tasks that do not need external data.

RAG

Used when domain-specific, private, or up-to-date information from documents is required.

Tool-based routing

Used when external systems like APIs, calculators, or databases are needed.

Hybrid routing

Used when both retrieval and external computation are required.

Key components

  • Intent classifier

  • Embedding similarity check

  • Confidence scoring system

  • Policy/rule engine

  • Fallback mechanism

Advantages

  • Reduces cost and latency

  • Improves accuracy

  • Uses specialized systems effectively

  • Scales better across use cases

Disadvantages

  • Adds system complexity

  • Routing mistakes can reduce quality

  • Requires tuning and monitoring

Simple real-world example

A query like “What is Python?” goes to a direct LLM, “What is our refund policy?” goes to RAG, and “Calculate 10% of 500” goes to a calculator tool. The router automatically selects the correct system.

Short interview-ready answer

Query routing in RAG systems decides whether a query should go to RAG, a direct LLM, or external tools. It uses intent classification, embeddings, and rule or ML-based decision layers to route queries based on complexity, domain, and data requirements, ensuring optimal accuracy, cost, and latency.

When should long-context models replace RAG, and how do you evaluate that decision?

Definition

Long-context models are LLMs that can process very large inputs (entire documents, multiple files, or long conversations) in a single prompt. The question is when this approach is better than using a RAG system, which retrieves only relevant chunks instead of sending everything.

Why this decision matters

Both approaches solve “knowledge grounding,” but with different tradeoffs:

  • RAG reduces input size via retrieval

  • Long-context models reduce need for retrieval entirely

Choosing incorrectly can increase cost, latency, or reduce accuracy.

When long-context models should replace RAG

1. When documents fit comfortably within context limits

If the entire knowledge base (or relevant subset) fits in the model’s context window, RAG becomes unnecessary.

2. When queries require holistic understanding

Use long-context models when:

  • Information is highly interconnected

  • Splitting into chunks breaks meaning

  • Cross-document reasoning is critical

3. When retrieval quality is unreliable

If RAG consistently:

  • Misses relevant chunks

  • Retrieves noisy or irrelevant context
    Then long-context input may be more reliable.

4. When data is relatively static

If documents do not change often, repeatedly retrieving may be unnecessary overhead.

5. When latency simplicity is preferred over retrieval complexity

Long-context models simplify architecture:

  • No vector DB

  • No embeddings

  • No reranking pipeline

When RAG is still better

  • Large-scale corpora (millions of documents)

  • Highly dynamic or frequently updated data

  • Strict cost constraints (long context tokens are expensive)

  • Need for precise, minimal context retrieval

How to evaluate the decision

1. Accuracy comparison

Measure:

  • Answer correctness

  • Faithfulness to source data

  • Hallucination rate

Compare:

  • RAG pipeline vs full-context prompting

2. Context utilization efficiency

Check:

  • How much of the long context is actually used

  • Whether retrieved chunks in RAG overlap with relevant long-context sections

If only a small portion is relevant, RAG is more efficient.

3. Latency and cost analysis

Compare:

  • RAG cost = retrieval + smaller prompt

  • Long-context cost = large token input + inference cost

Long-context becomes expensive quickly as input grows.

4. Retrieval recall vs redundancy

If RAG recall is low (frequently missing important info), long-context may outperform.

5. Scalability testing

Evaluate how both approaches behave when:

  • Document size increases

  • Query complexity increases

  • Number of users increases

6. Maintenance complexity

RAG requires:

  • Indexing pipelines

  • Embedding updates

  • Retrieval tuning

Long-context models reduce this overhead significantly.

Key decision rule of thumb

  • Use long-context models when data is small, stable, and needs global reasoning

  • Use RAG when data is large, dynamic, or requires selective retrieval

Simple real-world example

A legal assistant tool:

  • If analyzing a single 100-page contract → long-context model works well

  • If searching across 1 million legal documents → RAG is required

Short interview-ready answer

Long-context models should replace RAG when the relevant knowledge fits within the context window, retrieval quality is unreliable, or global document reasoning is required. The decision is evaluated using accuracy, retrieval recall, cost, latency, and scalability comparisons. If long-context processing provides higher accuracy with acceptable cost and performance, it can simplify the system by removing the need for a retrieval pipeline.
