What Is GraphRAG and Why Does It Exist?

What is GraphRAG and how is it fundamentally different from standard vector RAG?

Standard RAG retrieves text chunks by vector similarity — it finds paragraphs that are semantically close to your query and stuffs them into the LLM's context. GraphRAG retrieves graph paths — it traverses relationships between entities to answer questions that require multi-hop reasoning.

Think of it this way:

  • Vector RAG: find the 5 most similar paragraphs to "Tom Hanks movies" → send to LLM
  • GraphRAG: traverse the graph → Person node "Tom Hanks" → ACTED_IN edges → all connected Movie nodes → return the subgraph

The difference is structural. Vector similarity returns content that looks like your query. Graph traversal returns content that is directly connected to the entities in your query. For relational questions — "what did X and Y have in common?", "how are A and B connected?", "who worked with Z more than once?" — graph traversal is the only correct approach.

The core claim: no chunk size, no re-ranking, no retrieval-k setting can make vector RAG answer multi-hop graph queries correctly. It's a structural limitation, not a tuning problem.

What specific types of queries can GraphRAG handle that vector RAG cannot?

Three structural operations are impossible for vector search but trivial for graph traversal:

1. Set intersection across relationship chains

"Which actors worked with both Tom Hanks AND Meg Ryan?"

Requires simultaneously traversing two separate ACTED_IN chains and finding the intersection. Vector RAG can retrieve text about each actor separately, but has no mechanism to enforce AND logic across multi-document relationship paths.

2. Shortest-path traversal

"How is Kevin Bacon connected to Tom Hanks?"

This is the "Six Degrees of Kevin Bacon" problem. It requires walking arbitrary relationship hops until a connection is found. In Cypher: shortestPath((a)-[*]-(b)). No amount of text retrieval can compute a graph path.

3. Exclusion reasoning (complement sets)

"Actors in Nolan films who never co-starred together?"

Requires finding entities that do NOT have a specific relationship — the complement of a relationship set. Vector similarity returns similar items; it has no NOT operator across entity relationships.

Summary table:

Query typeVector RAGGraphRAG
Semantic similarity ("movies like Inception")✓ GoodLimited
Multi-entity AND logic✗ Cannot✓ Native
Shortest path between entities✗ Cannot✓ Native
Exclusion / complement sets✗ Cannot✓ Native
Aggregation (who directed the most…)✗ Unreliable✓ Native
What is a graph database and how is data structured in it?

A graph database stores data as nodes (entities) and edges (relationships between entities). Each node and edge can have properties (key-value pairs).

Example: a movie knowledge graph has these components:

# Nodes (entities)
(:Movie  {title: "The Matrix", year: 1999, rating: 8.7})
(:Person {name: "Keanu Reeves", born: 1964})
(:Genre  {name: "Action"})
(:Studio {name: "Warner Bros."})

# Edges (relationships)
(:Person)-[:ACTED_IN {roles: ["Neo"]}]->(:Movie)
(:Person)-[:DIRECTED]->(:Movie)
(:Movie )-[:IN_GENRE]->(:Genre)
(:Movie )-[:PRODUCED_BY]->(:Studio)

The key difference from a relational database: in a graph DB, relationships are first-class citizens — they have their own properties and can be queried directly. Finding all movies connected to a person takes O(1) edge lookups, not a JOIN across tables. For deeply connected data (social networks, knowledge graphs, recommendation engines), this is dramatically faster than SQL JOINs at scale.

When should you choose GraphRAG over vector RAG, and when is vector RAG still better?

Choose GraphRAG when:

  • Your data is inherently relational (people, movies, products, organisations — anything with typed relationships)
  • Users ask multi-hop questions: "who connected to X also worked with Y?"
  • You need aggregation across relationships: "who directed the most films in genre Z?"
  • Exact relationship traversal matters: "find all co-authors of papers citing paper X"
  • You need path-finding: "how many hops separate person A from company B?"

Choose vector RAG when:

  • Your data is unstructured text (articles, documents, PDFs)
  • Queries are semantic: "what does this document say about climate change?"
  • You don't have clear entity/relationship structure in your data
  • You need fuzzy matching: "find content similar to this concept"

Use both (hybrid): many production systems combine graph traversal for structural retrieval + vector search for unstructured content. Example: retrieve related entities via graph, then find similar text chunks for those entities via vector search.

What is Cypher and what are its key syntax patterns?

Cypher is Neo4j's query language. Its visual syntax mirrors the graph structure — parentheses for nodes, arrows for relationships — making it highly readable compared to SQL JOINs.

# Basic patterns:

# Node:           (variable:Label {property: value})
# Relationship:   -[:TYPE {property: value}]->
# Path:           (a)-[:REL]->(b)

# MATCH — find patterns
MATCH (p:Person {name: "Tom Hanks"})-[:ACTED_IN]->(m:Movie)
RETURN p.name, m.title, m.year

# WHERE — filter (use with MATCH for readability)
MATCH (p:Person)-[:ACTED_IN]->(m:Movie)
WHERE p.name = 'Tom Hanks' AND m.year > 2000
RETURN m.title ORDER BY m.year DESC

# WITH — pipe results (like SQL subquery)
MATCH (p:Person)-[:ACTED_IN]->(m:Movie)
WITH p, count(m) AS movie_count
WHERE movie_count > 5
RETURN p.name, movie_count ORDER BY movie_count DESC

# MERGE — upsert (create if not exists, match if it does)
MERGE (p:Person {tmdb_id: 31})
SET p.name = "Tom Hanks"

# LIMIT — always add this for safety
MATCH (m:Movie) RETURN m LIMIT 100

# Case-insensitive matching
WHERE toLower(p.name) = toLower('tom hanks')

Key difference from SQL: Cypher reads like a diagram. (person)-[:ACTED_IN]->(movie) is immediately clear; the equivalent SQL would require a JOIN through an actor_movie bridge table.

How do you write multi-hop Cypher queries, and what does "hop count" mean?

A "hop" is one traversal of a relationship edge. Hop count determines query complexity and is the key metric for understanding what GraphRAG can do that vector RAG cannot.

# 1 hop: direct relationship
MATCH (p:Person)-[:ACTED_IN]->(m:Movie)
WHERE toLower(p.name) = 'tom hanks'
RETURN p, m LIMIT 100

# 2 hops: through an intermediate node (set intersection)
MATCH (tom:Person {name:"Tom Hanks"})-[:ACTED_IN]->(m1:Movie)
       <-[:ACTED_IN]-(shared:Person),
      (meg:Person {name:"Meg Ryan"})-[:ACTED_IN]->(m2:Movie)
       <-[:ACTED_IN]-(shared)
RETURN DISTINCT shared, m1, m2 LIMIT 100

# 3 hops: director + genre + another movie
MATCH (m1:Movie {title:"The Matrix"})<-[:DIRECTED]-(d:Person),
      (m1)-[:IN_GENRE]->(g:Genre),
      (d)-[:DIRECTED]->(m2:Movie)-[:IN_GENRE]->(g)
WHERE m2 <> m1
RETURN d, m2, g LIMIT 100

# 4+ hops: shortest path (variable length)
MATCH path = shortestPath(
  (a:Person {name:"Kevin Bacon"})-[*]-(b:Person {name:"Tom Hanks"})
)
RETURN path LIMIT 100

The [*] in shortestPath means "any number of relationships of any type" — Neo4j finds the minimum-hop path automatically. You can also constrain: [*1..5] means 1 to 5 hops.

How do you connect to Neo4j from Python and run async queries? 
from neo4j import AsyncGraphDatabase
import asyncio

# Singleton driver pattern (create once, reuse for all queries)
_driver = None

def get_driver(uri: str, username: str, password: str):
    global _driver
    if _driver is None:
        _driver = AsyncGraphDatabase.driver(
            uri, auth=(username, password)
        )
    return _driver

async def run_query(cypher: str, params: dict = None) -> list[dict]:
    """Run a Cypher query and return results as list of dicts."""
    driver = get_driver(NEO4J_URI, NEO4J_USERNAME, NEO4J_PASSWORD)
    async with driver.session() as session:
        result = await session.run(cypher, params or {})
        records = await result.data()
        return records

# Synchronous version (for scripts, data loading)
from neo4j import GraphDatabase

def run_sync(cypher: str, params: dict = None):
    with GraphDatabase.driver(NEO4J_URI, auth=(NEO4J_USERNAME, NEO4J_PASSWORD)) as driver:
        with driver.session() as session:
            result = session.run(cypher, params or {})
            return list(result.data())

# Example
async def main():
    results = await run_query(
        "MATCH (p:Person)-[:ACTED_IN]->(m:Movie) "
        "WHERE toLower(p.name) = $name RETURN m.title, m.year",
        {"name": "tom hanks"}
    )
    for r in results:
        print(r["m.title"], r["m.year"])

asyncio.run(main())

Important: always close the driver on shutdown. In FastAPI, use an on_event("shutdown") hook: await driver.close(). Leaked connections will exhaust the AuraDB connection pool.

Building a GraphRAG Pipeline

What is text-to-Cypher (NL-to-Cypher) and how does it work with an LLM?

Text-to-Cypher is the GraphRAG equivalent of text-to-SQL: you send a natural language question to an LLM along with your graph schema, and it generates a valid Cypher query. The LLM never touches the database directly — it produces query strings that your application then executes.

The two-stage pipeline:

  1. Cypher generation (temperature=0, deterministic): LLM receives the question + schema + few-shot examples → outputs a Cypher string
  2. Answer synthesis (temperature=0.3, conversational): LLM receives the original question + the Cypher + the raw query results → outputs a natural language answer
from openai import AsyncOpenAI
client = AsyncOpenAI()

SCHEMA = """
(:Movie  {title, year, tagline, revenue, rating})
(:Person {name, born})
(:Genre  {name})
(:Studio {name})
(:Person)-[:ACTED_IN {roles}]->(:Movie)
(:Person)-[:DIRECTED]->(:Movie)
(:Movie )-[:IN_GENRE]->(:Genre)
(:Movie )-[:PRODUCED_BY]->(:Studio)
"""

CYPHER_SYSTEM = f"""You are a Neo4j Cypher expert. Given a natural-language question,
output ONLY a valid Cypher query — no markdown, no explanation.

Schema:
{SCHEMA}

Rules:
- Always RETURN full node/relationship objects for graph visualization.
- Use shortestPath() for path queries. No APOC procedures.
- Add LIMIT 100 to every query.
- Use toLower() for case-insensitive name matching.
"""

async def generate_cypher(question: str) -> str:
    resp = await client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": CYPHER_SYSTEM},
            {"role": "user",   "content": question},
        ],
        temperature=0,   # deterministic — critical for valid syntax
    )
    return resp.choices[0].message.content.strip()

async def synthesize_answer(question: str, cypher: str, results: list) -> str:
    resp = await client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": "Answer the question using the query results. End with one sentence explaining why this required graph traversal."},
            {"role": "user", "content": f"Question: {question}\nCypher: {cypher}\nResults: {results}"},
        ],
        temperature=0.3,
    )
    return resp.choices[0].message.content.strip()

Why temperature=0 for Cypher generation? Invalid Cypher syntax (wrong brackets, missing colons, incorrect relationship direction) causes hard errors. Deterministic output prevents subtle variations that sometimes produce invalid queries.

How do you write effective few-shot examples for text-to-Cypher?

Few-shot examples are the most important component of your Cypher generation prompt. They teach the LLM your graph's relationship naming conventions, when to use shortestPath(), and when to use WITH ... count() for aggregation.

Cover all the complexity tiers you need:

FEW_SHOT_EXAMPLES = """
# 1 hop — filmography
Q: Which movies did Tom Hanks act in?
MATCH (p:Person)-[r:ACTED_IN]->(m:Movie)
WHERE toLower(p.name) = 'tom hanks'
RETURN p, r, m LIMIT 100

# 2 hops — set intersection
Q: Which actors worked with both Tom Hanks and Meg Ryan?
MATCH (tom:Person {name:"Tom Hanks"})-[:ACTED_IN]->(m1:Movie)<-[:ACTED_IN]-(shared:Person),
      (meg:Person {name:"Meg Ryan"})-[:ACTED_IN]->(m2:Movie)<-[:ACTED_IN]-(shared)
RETURN DISTINCT shared, m1, m2 LIMIT 100

# 2 hops — aggregation
Q: Which directors have worked with Keanu Reeves more than once?
MATCH (k:Person {name:"Keanu Reeves"})-[:ACTED_IN]->(m:Movie)<-[:DIRECTED]-(d:Person)
WITH d, count(m) AS collaborations
WHERE collaborations > 1
MATCH (d)-[r:DIRECTED]->(m2:Movie)<-[a:ACTED_IN]-(k)
RETURN d, r, m2, a, k LIMIT 100

# 4+ hops — shortest path
Q: How is Kevin Bacon connected to Tom Hanks?
MATCH path = shortestPath(
  (a:Person {name:"Kevin Bacon"})-[*]-(b:Person {name:"Tom Hanks"})
)
RETURN path LIMIT 100

# 3 hops — multi-condition join
Q: Which movies share the director and genre of The Matrix?
MATCH (m1:Movie {title:"The Matrix"})<-[:DIRECTED]-(d:Person),
      (m1)-[:IN_GENRE]->(g:Genre),
      (d)-[:DIRECTED]->(m2:Movie)-[:IN_GENRE]->(g)
WHERE m2 <> m1
RETURN d, m2, g LIMIT 100
"""

Key rule: every example must RETURN full node and relationship objects (not just properties) — this enables the graph visualization layer to work correctly without parsing individual properties from text.

How do you parse Neo4j query results that contain Node, Relationship, and Path objects?

When you RETURN p, r, m in Cypher, the Python driver gives you typed objects (neo4j.graph.Node, neo4j.graph.Relationship, neo4j.graph.Path), not plain dicts. You need to walk them recursively to build your graph data structure.

from neo4j.graph import Node, Relationship, Path

def build_graph_data(records: list[dict]) -> dict:
    """
    Extract unique nodes and edges from raw Neo4j records.
    Handles all three return types: Node, Relationship, Path.
    """
    nodes = {}   # element_id -> node dict
    edges = []

    def add_node(node: Node):
        nid = str(node.element_id)
        if nid not in nodes:
            label    = list(node.labels)[0] if node.labels else "Unknown"
            props    = dict(node)
            display  = props.get("title") or props.get("name") or nid
            nodes[nid] = {
                "id": nid, "label": label,
                "properties": {**props, "display": display}
            }

    def add_relationship(rel: Relationship):
        edges.append({
            "source": str(rel.start_node.element_id),
            "target": str(rel.end_node.element_id),
            "type": rel.type,
        })

    def process_value(value):
        if isinstance(value, Node):
            add_node(value)
        elif isinstance(value, Relationship):
            add_node(value.start_node)
            add_node(value.end_node)
            add_relationship(value)
        elif isinstance(value, Path):
            for n in value.nodes:         add_node(n)
            for r in value.relationships:
                add_node(r.start_node);   add_node(r.end_node)
                add_relationship(r)
        elif isinstance(value, list):
            for item in value: process_value(item)

    for record in records:
        for value in record.values():
            process_value(value)

    return {"nodes": list(nodes.values()), "edges": edges}

# Usage
records = await run_cypher("MATCH path = shortestPath(...) RETURN path LIMIT 100")
graph_data = build_graph_data(records)
print(f"{len(graph_data['nodes'])} nodes, {len(graph_data['edges'])} edges")

The deduplication via element_id is critical — the same node often appears in multiple records (e.g. Tom Hanks appears in every record from a filmography query). Without deduplication, you get duplicate visualization nodes.

Loading Graph Data

How do you seed a Neo4j graph database from a REST API (TMDB example)?

The CineGraph project loads ~200 movies from TMDB (The Movie Database) into Neo4j. The pattern: fetch data from REST API → transform → upsert into Neo4j using MERGE (idempotent — safe to re-run).

import httpx
from neo4j import GraphDatabase
from dotenv import load_dotenv
import os

load_dotenv()
TMDB_KEY = os.getenv("TMDB_API_KEY")
NEO4J_URI = os.getenv("NEO4J_URI")
driver = GraphDatabase.driver(NEO4J_URI,
    auth=(os.getenv("NEO4J_USERNAME"), os.getenv("NEO4J_PASSWORD")))

def create_constraints(session):
    """Uniqueness constraints also create indexes automatically."""
    for stmt in [
        "CREATE CONSTRAINT IF NOT EXISTS FOR (m:Movie)  REQUIRE m.tmdb_id IS UNIQUE",
        "CREATE CONSTRAINT IF NOT EXISTS FOR (p:Person) REQUIRE p.tmdb_id IS UNIQUE",
        "CREATE CONSTRAINT IF NOT EXISTS FOR (g:Genre)  REQUIRE g.name    IS UNIQUE",
        "CREATE CONSTRAINT IF NOT EXISTS FOR (s:Studio) REQUIRE s.name    IS UNIQUE",
    ]:
        session.run(stmt)

def load_movie(session, details: dict):
    mid = details["id"]
    # MERGE = insert if not exists, match if it does (idempotent)
    session.run("""
        MERGE (m:Movie {tmdb_id: $id})
        SET m.title=$title, m.year=$year, m.rating=$rating, m.revenue=$revenue
        """,
        id=mid, title=details["title"],
        year=int((details.get("release_date") or "0000")[:4]),
        rating=details.get("vote_average", 0.0),
        revenue=details.get("revenue", 0),
    )
    # Genres
    for g in details.get("genres", []):
        session.run("""MERGE (g:Genre {name:$name})
                       WITH g MATCH (m:Movie {tmdb_id:$id})
                       MERGE (m)-[:IN_GENRE]->(g)""",
                    name=g["name"], id=mid)
    # Cast (top 15 only)
    for actor in details.get("credits", {}).get("cast", [])[:15]:
        session.run("""MERGE (p:Person {tmdb_id:$pid}) SET p.name=$name
                       WITH p MATCH (m:Movie {tmdb_id:$mid})
                       MERGE (p)-[r:ACTED_IN]->(m) SET r.roles=$roles""",
                    pid=actor["id"], name=actor["name"],
                    mid=mid, roles=[actor.get("character","")],
        )
    # Directors and writers
    for crew in details.get("credits", {}).get("crew", []):
        if crew.get("job") not in ("Director", "Writer", "Screenplay"):
            continue
        rel = "DIRECTED" if crew["job"] == "Director" else "WROTE"
        session.run(f"""MERGE (p:Person {{tmdb_id:$pid}}) SET p.name=$name
                        WITH p MATCH (m:Movie {{tmdb_id:$mid}})
                        MERGE (p)-[:{rel}]->(m)""",
                    pid=crew["id"], name=crew["name"], mid=mid)

def seed():
    with driver.session() as session:
        session.run("MATCH (n) DETACH DELETE n")  # clear first
        create_constraints(session)
        # Fetch 10 pages × 20 movies = ~200 movies
        for page in range(1, 11):
            movies = httpx.get(
                "https://api.themoviedb.org/3/movie/popular",
                params={"api_key": TMDB_KEY, "page": page}
            ).json()["results"]
            for movie in movies:
                details = httpx.get(
                    f"https://api.themoviedb.org/3/movie/{movie['id']}",
                    params={"api_key": TMDB_KEY, "append_to_response": "credits"}
                ).json()
                load_movie(session, details)
                print(f"Loaded: {details.get('title', '?')}")

if __name__ == "__main__":
    seed()
    print("Done.")

The MERGE pattern is essential: if you re-run the seed script, it updates existing nodes instead of creating duplicates. Always create uniqueness constraints before loading data — they enforce data integrity and speed up MERGE operations significantly.

What are Neo4j uniqueness constraints and why do you create them before loading data?

Uniqueness constraints enforce that a property value is unique across all nodes of a label — exactly like a UNIQUE constraint in SQL. Creating them before loading data matters for two reasons:

  1. Data integrity: if your seed script runs twice or has duplicate data, constraints prevent duplicate nodes
  2. Index creation: constraints automatically create a B-tree index on the constrained property, making MERGE (n:Movie {tmdb_id: $id}) O(log n) instead of O(n) — critical for large datasets
# Create constraints (safe to run multiple times with IF NOT EXISTS)
CREATE CONSTRAINT IF NOT EXISTS FOR (m:Movie)  REQUIRE m.tmdb_id IS UNIQUE
CREATE CONSTRAINT IF NOT EXISTS FOR (p:Person) REQUIRE p.tmdb_id IS UNIQUE
CREATE CONSTRAINT IF NOT EXISTS FOR (g:Genre)  REQUIRE g.name    IS UNIQUE
CREATE CONSTRAINT IF NOT EXISTS FOR (s:Studio) REQUIRE s.name    IS UNIQUE

# Verify constraints exist
SHOW CONSTRAINTS

# Check index usage in a query (EXPLAIN shows the query plan)
EXPLAIN
MATCH (m:Movie {tmdb_id: 603})
RETURN m

The EXPLAIN output will show NodeIndexSeek (fast, uses index) vs NodeByLabelScan (slow, scans all nodes). Always verify with EXPLAIN that your MATCH patterns use indexes for production workloads.

The CineGraph Architecture (GraphRAG Reference Implementation)

What is CineGraph and how is it architected?

CineGraph is an open-source GraphRAG demo built on top of TMDB movie data. It proves the GraphRAG thesis with a working application: FastAPI backend + Neo4j + GPT-4o + Streamlit frontend with live graph visualisation.

Data flow for every query:

User question
    ↓
Streamlit frontend (app.py)
    ↓ POST /api/query {"question": "..."}
FastAPI backend (routes/query.py)
    ↓
    1. llm_service.generate_cypher(question)
          GPT-4o [temperature=0] + schema + few-shot examples
          → Cypher string
    ↓
    2. neo4j_service.run_cypher(cypher)
          AsyncGraphDatabase → Neo4j AuraDB
          → list[dict] of records
    ↓
    3. neo4j_service.build_graph_data(records)
          Walk Node/Relationship/Path objects
          → {nodes: [...], edges: [...]}
    ↓
    4. llm_service.synthesize_answer(question, cypher, records)
          GPT-4o [temperature=0.3]
          → Natural language answer
    ↓
QueryResponse {answer, cypher, hop_count, graph_data, meta}
    ↓
Streamlit renders:
  - Force-directed graph (streamlit-agraph)
  - Conversational answer
  - Generated Cypher (visible to user)
  - Metrics: node count, edge count, query time (ms)

Repository: github.com/techwithprateek/graphrag

What does the complete FastAPI backend for a GraphRAG system look like? 
# backend/main.py
from fastapi import FastAPI, Request
from fastapi.middleware.cors import CORSMiddleware
from fastapi.responses import JSONResponse

app = FastAPI(title="CineGraph API")

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:8501"],  # Streamlit default port
    allow_methods=["*"], allow_headers=["*"],
)

# Global error handler — return JSON even for unhandled exceptions
@app.exception_handler(Exception)
async def global_error_handler(request: Request, exc: Exception):
    return JSONResponse(
        status_code=200,   # 200 so frontend always gets JSON, not HTML error page
        content={"error": True, "detail": str(exc)},
    )

@app.on_event("shutdown")
async def shutdown():
    await close_driver()   # close Neo4j connection pool cleanly
# backend/routes/query.py
from fastapi import APIRouter
from pydantic import BaseModel
from typing import Any
import time

router = APIRouter()

class QueryRequest(BaseModel):
    question: str

@router.post("/query")
async def query(request: QueryRequest):
    question = request.question

    # Stage 1: Generate Cypher
    cypher = await generate_cypher(question)

    # Stage 2: Execute against Neo4j
    start = time.time()
    records = await run_cypher(cypher)
    elapsed_ms = int((time.time() - start) * 1000)

    # Stage 3: Build visualization data
    graph_data = build_graph_data(records)

    # Stage 4: Synthesize answer
    answer = await synthesize_answer(question, cypher, records)

    # Rough hop count from Cypher syntax
    hop_count = max(cypher.count("->"), cypher.count("<-"), 1)

    return {
        "answer": answer,
        "cypher": cypher,
        "hop_count": hop_count,
        "graph_data": graph_data,
        "meta": {
            "node_count": len(graph_data["nodes"]),
            "edge_count": len(graph_data["edges"]),
            "query_ms": elapsed_ms,
        }
    }

# API endpoints exposed:
# GET  /api/health   → {"status": "ok"}
# GET  /api/schema   → {labels, relationship_types, property_keys}
# GET  /api/examples → {examples: [7 example questions]}
# POST /api/query    → {answer, cypher, hop_count, graph_data, meta}
How do you build a Streamlit frontend with live graph visualisation for GraphRAG?

The CineGraph frontend uses streamlit-agraph — a Streamlit component wrapping vis.js — to render physics-simulated force-directed graphs. Each node type gets a distinct color and size.

import streamlit as st
import httpx
from streamlit_agraph import agraph, Node, Edge, Config

BACKEND = "http://localhost:8000"

# Visual config by node label

How do you introspect a Neo4j database schema at runtime?

Rather than hardcoding the schema in your application, you can query it dynamically — useful when the schema evolves or when building generic GraphRAG tools.

async def get_schema() -> dict:
    """Dynamically discover the graph schema from Neo4j."""
    driver = get_driver()
    async with driver.session() as session:
        # All node labels
        label_result = await session.run("CALL db.labels()")
        labels = [r["label"] async for r in label_result]

        # All relationship types
        rel_result = await session.run("CALL db.relationshipTypes()")
        rel_types = [r["relationshipType"] async for r in rel_result]

        # All property keys
        prop_result = await session.run("CALL db.propertyKeys()")
        prop_keys = [r["propertyKey"] async for r in prop_result]

        # More detailed: node labels with their properties
        sample_query = """
        MATCH (n)
        WITH labels(n)[0] AS label, keys(n) AS props
        RETURN DISTINCT label, props
        LIMIT 50
        """
        sample_result = await session.run(sample_query)
        samples = [{"label": r["label"], "props": r["props"]}
                   async for r in sample_result]

    return {
        "labels": labels,
        "relationship_types": rel_types,
        "property_keys": prop_keys,
        "sample_schema": samples,
    }

# Use this to auto-generate your schema string for the Cypher prompt:
schema = await get_schema()
schema_str = "\n".join(
    f"(:{s['label']} {{{', '.join(s['props'])}}})"
    for s in schema["sample_schema"]
)

GraphRAG Design Patterns & Best Practices

What are the key design decisions when building a production GraphRAG system?

The CineGraph project embeds several production-grade design decisions worth understanding:

1. Temperature=0 for Cypher generation

Invalid Cypher causes hard runtime errors. Deterministic output (temperature=0) is non-negotiable for query generation. Use temperature only for the answer synthesis step.

2. LIMIT on every query

Always add LIMIT 100 in your system prompt. Without it, a path query like shortestPath((a)-[*]-(b)) can return thousands of nodes and exhaust memory.

3. MERGE over CREATE for seeding

MERGE is idempotent — if you re-run the seed script after adding new data, it won't create duplicates. CREATE would duplicate every record.

4. Return full objects, not properties

RETURN p, r, m not RETURN p.name, m.title. Full objects give you the graph structure needed for visualization. Individual properties lose edge information.

5. Singleton driver pattern

The Neo4j driver manages a connection pool internally. Create one driver at application startup and reuse it — don't create a new driver per request.

6. No APOC in system prompt

APOC is a Neo4j plugin not available on all tiers (especially AuraDB free). Use native Cypher functions: shortestPath() instead of APOC equivalents.

7. Explicit few-shot examples for every hop complexity

Cover 1-hop, 2-hop (intersection), 2-hop (aggregation), 3-hop (join), and 4+-hop (path) queries in your few-shot examples. Without examples for each tier, the LLM will default to overly simple 1-hop patterns.

What are the common failure modes in text-to-Cypher systems and how do you handle them?

Text-to-Cypher fails in predictable ways. Knowing them lets you write defensive code:

1. Property name hallucination

LLM uses m.name instead of m.title. The query runs but returns null. Fix: include exact property names in your schema prompt and few-shot examples.

async def run_cypher_safe(cypher: str) -> tuple[list, str | None]:
    """Returns (results, error). Empty results ≠ error."""
    try:
        results = await run_cypher(cypher)
        return results, None
    except Exception as e:
        return [], str(e)

# In the route handler:
results, err = await run_cypher_safe(cypher)
if err:
    # Retry: send the error back to the LLM for correction
    cypher = await fix_cypher(original_question, cypher, err)
    results, _ = await run_cypher_safe(cypher)

2. Case sensitivity errors

LLM generates WHERE p.name = "Tom Hanks" — fails if the DB has "tom hanks". Fix: enforce toLower() in system prompt: "Always use WHERE toLower(p.name) = toLower($name)".

3. Relationship direction errors

(movie)-[:ACTED_IN]->(person) instead of (person)-[:ACTED_IN]->(movie). Fix: show directionality explicitly in the schema and all few-shot examples.

4. Markdown code fences in output

LLM returns ```cypher\nMATCH...``` instead of raw Cypher. Fix: strip fences after generation:

def clean_cypher(raw: str) -> str:
    import re
    cleaned = re.sub(r"```(?:cypher)?\n?", "", raw)
    return cleaned.strip().rstrip("`")

5. Empty results for valid queries

The graph doesn't contain the requested entity. Always check for empty results and return a graceful message: "No results found. The database contains movies from 2020-2024 — try a different actor."

How does GraphRAG differ from Microsoft's GraphRAG (entity extraction approach)?

The term "GraphRAG" is used for two different approaches — it's important not to conflate them:

AspectCineGraph / Neo4j GraphRAGMicrosoft GraphRAG
Data sourcePre-structured relational data (movies, people, companies)Unstructured text documents
Graph constructionManual schema design + data loadingAutomatic: LLM extracts entities + relationships from text
Query mechanismCypher queries (NL → Cypher → execute)Global/local search over entity communities
StrengthsPrecise multi-hop traversal over known entitiesBroad thematic questions over large document corpora
Best forStructured domains: movies, products, organisations, knowledge basesResearch, document analysis, "what are the themes of these reports?"

Microsoft's GraphRAG automatically builds a knowledge graph from raw text using LLM entity extraction, then uses community detection to answer global questions ("what are the main themes?"). The Neo4j/Cypher approach requires a pre-existing structured graph but gives precise, verifiable results via deterministic Cypher queries.

Choose Neo4j GraphRAG when you have structured data and need precise relational queries. Choose Microsoft GraphRAG when you have large unstructured document corpora and want high-level thematic insights.

What is the full Pydantic model structure for a GraphRAG API response?

Cypher Query Reference

What are the seven canonical GraphRAG query patterns with increasing complexity?

These seven patterns cover the full spectrum from simple lookup to complex path-finding — the exact examples in the CineGraph project:

# 1. [1 hop] Filmography — direct relationship
MATCH (p:Person)-[r:ACTED_IN]->(m:Movie)
WHERE toLower(p.name) = 'tom hanks'
RETURN p, r, m LIMIT 100

# 2. [2 hops] Set intersection — actors in common
MATCH (tom:Person {name:"Tom Hanks"})-[:ACTED_IN]->(m1:Movie)
      <-[:ACTED_IN]-(shared:Person),
      (meg:Person {name:"Meg Ryan"})-[:ACTED_IN]->(m2:Movie)
      <-[:ACTED_IN]-(shared)
RETURN DISTINCT shared, m1, m2 LIMIT 100

# 3. [2 hops] Aggregation — directed together 2+ times
MATCH (k:Person {name:"Keanu Reeves"})-[:ACTED_IN]->(m:Movie)
      <-[:DIRECTED]-(d:Person)
WITH d, count(m) AS collab
WHERE collab > 1
MATCH (d)-[r:DIRECTED]->(m2:Movie)<-[a:ACTED_IN]-(k)
RETURN d, r, m2, a, k LIMIT 100

# 4. [3 hops] Most versatile actor (appeared in most genres)
MATCH (p:Person)-[:ACTED_IN]->(m:Movie)-[:IN_GENRE]->(g:Genre)
WITH p, count(DISTINCT g) AS genre_count
ORDER BY genre_count DESC LIMIT 5
MATCH (p)-[r:ACTED_IN]->(m2:Movie)-[ig:IN_GENRE]->(g2:Genre)
RETURN p, r, m2, ig, g2 LIMIT 100

# 5. [4+ hops] Shortest path — Kevin Bacon number
MATCH path = shortestPath(
  (a:Person {name:"Kevin Bacon"})-[*]-(b:Person {name:"Tom Hanks"})
)
RETURN path LIMIT 100

# 6. [3 hops] Shared director AND genre
MATCH (m1:Movie {title:"The Matrix"})<-[:DIRECTED]-(d:Person),
      (m1)-[:IN_GENRE]->(g:Genre),
      (d)-[:DIRECTED]->(m2:Movie)-[:IN_GENRE]->(g)
WHERE m2 <> m1
RETURN d, m2, g LIMIT 100

# 7. [3 hops + exclusion] Nolan actors who never co-starred
MATCH (nolan:Person {name:"Christopher Nolan"})-[:DIRECTED]->(m:Movie)
      <-[:ACTED_IN]-(a:Person)
WITH collect(DISTINCT a) AS nolan_actors, collect(DISTINCT m) AS nolan_films
UNWIND nolan_actors AS actor1
UNWIND nolan_actors AS actor2
WHERE actor1 <> actor2
  AND NOT EXISTS {
    MATCH (actor1)-[:ACTED_IN]->(shared:Movie)<-[:ACTED_IN]-(actor2)
  }
RETURN DISTINCT actor1.name, actor2.name LIMIT 50
How do you use variable-length path queries and the shortestPath() function?

Variable-length paths let you traverse an unknown number of hops — essential for graph distance calculations and path-finding.

# Variable length: [*min..max] — any relationship type
MATCH (a:Person {name:"Kevin Bacon"})-[*1..6]-(b:Person {name:"Tom Hanks"})
RETURN count(*) AS paths_found

# Shortest path between two nodes (built-in, no APOC needed)
MATCH path = shortestPath(
  (start:Person {name:"Kevin Bacon"})-[*]-(end:Person {name:"Tom Hanks"})
)
RETURN
  [node IN nodes(path) | coalesce(node.name, node.title)] AS path_display,
  length(path) AS hops

# All shortest paths (when multiple exist)
MATCH paths = allShortestPaths(
  (a:Person {name:"Meryl Streep"})-[*]-(b:Person {name:"Robert De Niro"})
)
RETURN paths LIMIT 10

# Path with relationship types
MATCH path = shortestPath(
  (a:Person {name:"Kevin Bacon"})-[:ACTED_IN|DIRECTED*]-(b:Person {name:"Spielberg"})
)
RETURN path

# Extract nodes and relationships from a path
MATCH path = shortestPath((a:Person {name:"Kevin Bacon"})-[*]-(b:Person {name:"Tom Hanks"}))
RETURN
  nodes(path)         AS all_nodes,
  relationships(path) AS all_rels,
  length(path)        AS hop_count

Key syntax notes: [*] = any number of hops of any type. [*1..5] = 1 to 5 hops. [:ACTED_IN|DIRECTED*] = variable hops but only those relationship types. Always add a hop limit for production queries — unbounded [*] on large graphs is extremely slow.
