Prompt Engineering Fundamentals

What is prompt engineering?

​Prompt engineering is the practice of designing clear and structured instructions so that a language model produces the desired output.

Instead of changing the model, you change how you ask the question.

Example:

Prompt A:

“Explain APIs.”

Prompt B:

“Explain APIs to a beginner in simple words with one real-world example.”

Prompt B produces a much better and more targeted answer.


What is a zero-shot prompt? Give an example.

What is a Zero-Shot Prompt?

A zero-shot prompt is a type of prompt where you ask a model to perform a task without giving it any examples of how to do that task.

The model relies entirely on:

  • The instructions in the prompt

  • Its pre-trained knowledge

  • General patterns it has learned during training

There are zero examples shown in the prompt — hence the name zero-shot.


Why is it called “Zero-Shot”?

“Shot” refers to an example.

  • Zero-shot → 0 examples

  • One-shot → 1 example

  • Few-shot → A few examples

In zero-shot prompting, the instruction alone must be clear enough for the model to understand what to do.


Basic Structure of a Zero-Shot Prompt

A typical zero-shot prompt contains:

  1. Task instruction

  2. Input data

  3. Expected output format (optional but recommended)

Example structure:

Instruction
Input
Output format requirement (optional)

Example 1: Zero-Shot Sentiment Classification

Prompt:

“Classify the following review as Positive or Negative:

Review: ‘The customer service was extremely helpful and friendly.’”

No examples are provided.

Model Output:

“Positive”

The model understands what “classify” and “Positive or Negative” mean based on its training.


Example 2: Zero-Shot Translation

Prompt:

“Translate the following sentence to French:

‘Good morning, how are you?’”

Model Output:

“Bonjour, comment allez-vous ?”

Again, no translation examples are given.


Example 3: Zero-Shot Structured Extraction

Prompt:

“Extract the name and email address from the following text and return in JSON format:

‘Hi, this is Sarah Johnson. You can reach me at sarah.j@email.com.’”

Model Output:

{
"name": "Sarah Johnson",
"email": "sarah.j@email.com"
}

No sample extraction examples were provided.


Why Does Zero-Shot Work?

Large Language Models are trained on massive datasets containing:

  • Instructions

  • Question-answer pairs

  • Explanations

  • Code

  • Conversations

Because of this, they learn general task patterns.

When you give an instruction like “Classify” or “Translate,” the model recognizes the pattern and performs the task.


When Zero-Shot Prompting Works Well

Zero-shot works best when:

  • The task is common (translation, summarization, classification)

  • Instructions are clear and specific

  • Output format is simple

  • The task is not highly specialized


When Zero-Shot May Fail

Zero-shot prompting may struggle when:

  • The task is very domain-specific

  • The output format is complex

  • The instructions are ambiguous

  • The task requires strict formatting

  • The problem requires complex reasoning

In such cases, few-shot prompting often improves results.


How to Improve Zero-Shot Prompts

To make zero-shot prompts more reliable:

  1. Be explicit about the task
    Instead of “Analyze this,” say
    “Summarize this text in 3 bullet points.”

  2. Define output format
    “Return output as valid JSON.”

  3. Add constraints
    “If unsure, say ‘I don’t know.’”

  4. Specify tone or audience
    “Explain for a beginner.”

Even though it’s zero-shot, clarity improves accuracy.


Comparison with Few-Shot Prompting

Zero-shot:

  • No examples

  • Faster and shorter prompt

  • Less context usage

  • May be less accurate for complex tasks

Few-shot:

  • Includes example demonstrations

  • More reliable

  • Uses more tokens

  • Better for structured or difficult tasks


Summary

A zero-shot prompt is an instruction given to a model without any examples. The model performs the task based only on the instruction and its pre-trained knowledge. It works well for common and clearly defined tasks but may require refinement or few-shot examples for complex scenarios.

What is few-shot prompting and why do we use it?

What is Few-Shot Prompting?

Few-shot prompting is a technique where you provide a small number of examples inside the prompt to demonstrate how the task should be performed.

Instead of just giving instructions, you show the model:

  • What the input looks like

  • What the expected output looks like

  • The pattern between them

The model then uses those examples to generate the correct output for a new input.

“Few” typically means 2–5 examples (sometimes slightly more, depending on context size).


Basic Structure of Few-Shot Prompting

A typical few-shot prompt contains:

  1. Task instruction

  2. Example 1 (Input → Output)

  3. Example 2 (Input → Output)

  4. Example 3 (Input → Output)

  5. New input (Model completes the output)


Example 1: Sentiment Classification

Prompt:

Classify the sentiment as Positive or Negative.

Review: “The movie was amazing and inspiring.”
Sentiment: Positive

Review: “The service was slow and disappointing.”
Sentiment: Negative

Review: “The product quality is outstanding.”
Sentiment:

Model Output:

Positive

The examples help the model understand:

  • Task type

  • Output format

  • Expected style


Example 2: Structured Data Extraction

Prompt:

Extract Name and Age from the sentence and return in JSON.

Sentence: “John is 28 years old.”
Output: { "name": "John", "age": 28 }

Sentence: “Maria turned 34 last week.”
Output: { "name": "Maria", "age": 34 }

Sentence: “David is 41.”
Output:

The model now follows the same structure and generates correct JSON.


Why Do We Use Few-Shot Prompting?

  1. Improves Accuracy

For complex tasks, instructions alone may not be enough.
Examples clarify the expected behavior.

  1. Clarifies Output Format

If you need strict JSON, table, or custom formatting, examples reduce formatting errors.

  1. Reduces Ambiguity

Examples eliminate confusion about:

  • Tone

  • Structure

  • Detail level

  • Scope

  1. Helps in Domain-Specific Tasks

For specialized tasks (legal, medical, technical extraction), few examples guide the model better.

  1. Increases Consistency

The model tends to mimic the pattern shown in examples, producing more stable results.


When Few-Shot Prompting Is Most Useful

  • Custom classification tasks

  • Entity extraction

  • Formatting-heavy outputs

  • Domain-specific transformations

  • Complex reasoning tasks


Trade-Offs

Advantages:

  • Higher reliability

  • Better structure control

  • Better task alignment

Disadvantages:

  • Uses more tokens (cost increases)

  • Reduces available context window

  • Requires careful example selection


Zero-Shot vs Few-Shot

Zero-shot:

  • Only instructions

  • Simpler and cheaper

  • May be less precise

Few-shot:

  • Includes examples

  • More reliable

  • Better for complex or structured tasks


Key Idea

Few-shot prompting works because the model learns patterns from the examples provided in the prompt and imitates that pattern for new inputs.

It turns the prompt into a small demonstration-based learning setup without retraining the model.

What is the difference between a system prompt and a role prompt?

What is a System Prompt?

A system prompt is a high-priority instruction that defines the overall behavior, personality, constraints, and boundaries of the model.

It usually:

  • Sets the identity of the model

  • Defines safety rules

  • Controls tone and style

  • Establishes refusal policies

  • Applies globally across the conversation

It acts as a persistent control layer.

Example:

“You are a professional financial assistant. Provide high-level educational information only. Do not provide personalized investment advice.”

This instruction influences all future responses unless overridden by stronger system rules.

Key characteristics:

  • Highest priority instruction

  • Persistent across conversation

  • Used for safety, compliance, and guardrails

  • Controls overall behavior


What is a Role Prompt?

A role prompt is a user-level instruction that temporarily asks the model to behave in a certain way for a specific task.

It usually:

  • Assigns a perspective or persona

  • Applies to a single query or context

  • Is part of the user’s request

Example:

“Act as a senior software engineer and review this code.”

This influences the response for that specific task but does not permanently redefine the model.

Key characteristics:

  • Task-specific

  • Temporary

  • Lower priority than system instructions

  • Used for stylistic or perspective control


Key Differences

  1. Scope

System Prompt:
Global behavior across the session.

Role Prompt:
Applies to a specific task or request.


  1. Priority

System Prompt:
Higher priority and stronger control.

Role Prompt:
Lower priority; can be overridden by system constraints.


  1. Purpose

System Prompt:
Sets safety, identity, tone, and boundaries.

Role Prompt:
Sets perspective or style for a task.


  1. Example Comparison

System Prompt:
“You are a medical information assistant. Provide general information only and avoid diagnosis.”

Role Prompt:
“Act as a cardiologist and explain heart disease.”

If the cardiologist explanation attempts diagnosis, the system-level rule will restrict it.


Summary

System prompt:
Defines global behavior and safety guardrails.

Role prompt:
Defines temporary task-specific persona or perspective.

System prompts control how the model behaves overall.
Role prompts control how the model behaves for a particular instruction.

What are some best practices for writing good prompts?

1. Be Clear and Specific

Avoid vague instructions.

Vague:
“Explain AI.”

Better:
“Explain artificial intelligence to a beginner in 5 short bullet points using simple language.”

Clarity reduces ambiguity and improves output quality.


2. Clearly Define the Task

Explicitly state what you want the model to do.

Examples:

  • “Summarize the text in 3 sentences.”

  • “Classify the sentiment as Positive, Negative, or Neutral.”

  • “Extract name, email, and phone number.”

Do not assume the model will infer your intention.


3. Specify Output Format

If you need structured output, say so clearly.

Examples:

  • “Return the result in valid JSON.”

  • “Provide the answer as a markdown table.”

  • “Use numbered steps.”

Also add:
“Do not include extra explanation.”

This reduces formatting errors.


4. Provide Constraints

Constraints reduce hallucination and unsafe output.

Examples:

  • “If unsure, say ‘I don’t know.’”

  • “Do not make assumptions.”

  • “Use only the provided text.”

  • “Keep the answer under 150 words.”

Constraints guide boundaries.


5. Break Complex Tasks into Steps

Instead of:

“Design and implement a full AI system.”

Use:

“Step 1: Define the problem.
Step 2: Propose architecture.
Step 3: Suggest evaluation metrics.”

Structured prompting improves reasoning and accuracy.


6. Use Examples When Necessary (Few-Shot)

For complex or structured tasks, give 2–3 examples.

Examples clarify:

  • Output structure

  • Tone

  • Level of detail

Few-shot prompting increases consistency.


7. Define the Target Audience

Specify who the explanation is for.

Examples:

  • “Explain for a 10-year-old.”

  • “Explain for a senior software engineer.”

  • “Explain for non-technical stakeholders.”

This controls depth and complexity.


8. Control Tone and Style

Be explicit about tone.

Examples:

  • “Use a formal tone.”

  • “Write in a conversational style.”

  • “Be neutral and objective.”

This ensures consistent voice.


9. Limit Scope

Prevent unnecessary expansion.

Examples:

  • “Focus only on advantages.”

  • “Do not discuss implementation.”

  • “Provide only three examples.”

This keeps output aligned with your goal.


10. Use Variables for Dynamic Prompts

Design reusable templates.

Example:

“Explain {topic} to a {audience_level} in {tone} style.”

Variables allow scalable systems and personalization.


11. Encourage Step-by-Step Reasoning When Needed

For logic-heavy tasks:

“Think step by step before giving the final answer.”

This improves reasoning accuracy.


12. Avoid Leading or Biased Language

Do not frame questions in a biased way.

Biased:
“Why is this policy bad?”

Neutral:
“Discuss the advantages and disadvantages of this policy.”

Prompt neutrality improves balanced responses.


13. Test and Iterate

Prompt writing is iterative.

  • Test on multiple inputs.

  • Check edge cases.

  • Refine wording.

  • Compare alternative prompts.

Treat prompt design as optimization.


14. Keep Prompts Simple When Possible

Do not overcomplicate.

Short, clear prompts often perform better than long, confusing ones.


15. Define Refusal and Safety Conditions

Especially for production systems:

“If the request involves illegal, harmful, or unsafe actions, refuse politely.”

This adds a safety layer.


Summary

Good prompts are:

  • Clear

  • Specific

  • Structured

  • Constrained

  • Audience-aware

  • Format-controlled

  • Iteratively tested

Effective prompt engineering is about reducing ambiguity and guiding the model toward the exact behavior you want.

How do you measure whether a prompt is effective?

Measuring whether a prompt is effective means evaluating how well it achieves the intended task reliably, accurately, and safely.

You should treat prompts like experiments: define goals, measure performance, compare alternatives, and iterate.


1. Define Clear Success Criteria First

Before measuring effectiveness, define:

  • What is the task?

  • What does a correct answer look like?

  • What metric will determine success?

Examples:

  • Classification → Accuracy, F1-score

  • Summarization → ROUGE or human rating

  • Code generation → Does it compile? Pass tests?

  • Extraction → Exact match / schema validation

Without a clear metric, you cannot evaluate prompt quality objectively.


2. Measure Accuracy Against Ground Truth

For tasks with correct answers:

  • Prepare a test dataset.

  • Run the prompt on all examples.

  • Compare outputs to ground truth.

  • Calculate metrics such as:

    • Accuracy

    • Precision / Recall

    • F1-score

    • Exact match rate

Example:
If 90 out of 100 outputs are correct, prompt accuracy = 90%.


3. Check Format Compliance

If structured output is required:

  • Does the output follow JSON schema?

  • Is it valid and parsable?

  • Does it match required fields?

You can measure:

  • Schema validation success rate

  • Percentage of correctly formatted responses

A prompt that produces correct content but wrong format may still fail in production.


4. Evaluate Consistency

Run the same prompt multiple times (especially with non-zero temperature).

Measure:

  • Do results vary significantly?

  • Is the core answer stable?

High variability may indicate weak prompt clarity.


5. Measure Hallucination Rate

For factual tasks:

  • Count incorrect or fabricated statements.

  • Track frequency of “made-up” information.

Lower hallucination rate = better prompt reliability.


6. Test Robustness on Edge Cases

Evaluate:

  • Ambiguous inputs

  • Missing data

  • Adversarial inputs

  • Slightly reworded inputs

An effective prompt should perform consistently across variations.


7. Human Evaluation

For open-ended tasks (creative writing, explanations):

Use human scoring on:

  • Relevance

  • Clarity

  • Coherence

  • Completeness

  • Tone alignment

Often scored on a 1–5 scale.

Human evaluation is critical for subjective tasks.


8. A/B Testing

Compare two or more prompt versions:

Prompt A vs Prompt B

Evaluate them on:

  • Accuracy

  • Format compliance

  • User satisfaction

  • Task success rate

Choose the one that performs better consistently.


9. Measure Latency and Cost

In production systems:

  • Does the prompt increase token usage?

  • Does it increase response time?

  • Is it cost-efficient?

A prompt that works but is expensive may not be optimal.


10. Measure User Satisfaction (For Chatbots)

Track:

  • Resolution rate

  • Escalation rate

  • User ratings

  • Conversation success rate

Real-world usage metrics often matter more than lab metrics.


Summary

A prompt is effective if it:

  • Produces accurate outputs

  • Follows required format

  • Minimizes hallucinations

  • Is consistent across runs

  • Handles edge cases well

  • Meets performance and cost constraints

  • Satisfies end users

Prompt evaluation should be systematic, measurable, and iterative rather than subjective.

What does the temperature setting do in an LLM?

Temperature is a parameter used during text generation in Large Language Models (LLMs). It controls the randomness of the model’s output.

LLMs generate text by predicting the next word based on probabilities. For each possible next token, the model assigns a probability score. Temperature modifies how these probabilities are used when sampling the next token.

How temperature works

Before selecting the next word, the model computes probabilities using softmax. Temperature adjusts these probabilities:

  • Low temperature (e.g., 0.1–0.3)

    • Makes high-probability tokens even more likely.

    • Output becomes more deterministic and focused.

    • Less creativity, more predictable responses.

  • Medium temperature (e.g., 0.7–1.0)

    • Balanced randomness.

    • Good mix of coherence and creativity.

  • High temperature (e.g., 1.2 and above)

    • Flattens the probability distribution.

    • Low-probability tokens become more likely.

    • Output becomes more diverse but less stable.

    • May produce surprising or less coherent responses.

Intuition

Think of temperature like risk tolerance.

Low temperature:
The model plays safe and picks the most likely words.

High temperature:
The model explores more possibilities and takes creative risks.

Example

Prompt: “The sky is”

Low temperature output:
“The sky is blue.”

High temperature output:
“The sky is a canvas of shifting silver dreams.”

Effect on probability distribution

Low temperature sharpens the distribution:
The highest probability token dominates.

High temperature smooths the distribution:
Differences between token probabilities become smaller.

When to use what

Low temperature:

  • Factual answers

  • Code generation

  • Technical explanations

  • Mathematical reasoning

Higher temperature:

  • Creative writing

  • Brainstorming

  • Story generation

  • Idea exploration

Important note

Temperature does not change what the model knows.
It only changes how confidently or creatively the model selects from its predicted probabilities.


What is top-K and top-P sampling?

When an LLM generates text, it assigns probabilities to all possible next tokens. Instead of always picking the highest-probability token, we can sample from the distribution.

Top-K and Top-P are sampling strategies that control how many candidate tokens the model is allowed to choose from.


Top-K Sampling

Top-K sampling limits the model to the top K most probable tokens.

Steps:

  1. Sort tokens by probability.

  2. Keep only the top K tokens.

  3. Set probabilities of all other tokens to zero.

  4. Renormalize the remaining probabilities.

  5. Sample from these K tokens.

Example:

If K = 5:

  • Only the 5 highest-probability tokens are considered.

  • The rest are completely ignored.

Effect:

  • Smaller K → more focused and deterministic.

  • Larger K → more diversity.

If K = 1:

  • This becomes greedy decoding (always pick highest probability).

Advantages:

  • Simple and easy to control.

  • Prevents very unlikely tokens from being chosen.

Limitation:

  • Fixed number of tokens, even if probability distribution is very uneven.


Top-P Sampling (Nucleus Sampling)

Top-P sampling selects the smallest set of tokens whose cumulative probability is at least P.

Steps:

  1. Sort tokens by probability.

  2. Add tokens one by one until cumulative probability ≥ P.

  3. Keep only those tokens.

  4. Renormalize.

  5. Sample from this subset.

Example:

If P = 0.9:

  • Select the smallest number of tokens whose probabilities sum to 90%.

  • In one situation, it may include 3 tokens.

  • In another, it may include 10 tokens.

Effect:

  • Adapts dynamically to the shape of the probability distribution.

  • If distribution is sharp → fewer tokens selected.

  • If distribution is flat → more tokens selected.

Advantages:

  • More flexible than Top-K.

  • Often produces more natural text.


Key Difference

Top-K:

  • Keeps a fixed number of tokens.

  • Does not consider total probability mass.

Top-P:

  • Keeps a variable number of tokens.

  • Based on cumulative probability threshold.


Practical Usage

Top-K:

  • Useful when you want strict control over randomness.

Top-P:

  • More commonly used in modern LLMs.

  • Produces better balance between diversity and coherence.

Both are often combined with temperature to control creativity and randomness during text generation.

What should you do if the model repeats words or generates loops?

If a language model repeats words, phrases, or generates loops, it usually means the decoding strategy is allowing high-probability tokens to be selected repeatedly. This is a generation-time issue, not a training issue.

Ways to reduce repetition and loops:

  1. Lower the temperature (if too high)

Very high temperature can make the model unstable and repetitive.

  • Try reducing temperature to 0.7 or lower.

  • If it becomes too deterministic, adjust gradually.

  1. Adjust Top-K or Top-P

If Top-K is too small:

  • The model may keep selecting the same few tokens.

  • Increase K slightly.

If Top-P is too high (e.g., 0.98 or 1.0):

  • Too many tokens are allowed.

  • Reduce it to 0.8–0.9.

  1. Use repetition penalty

Repetition penalty reduces the probability of tokens that have already appeared in the generated text.

  • If a token appears again, its probability is decreased.

  • Helps prevent loops like:
    “The result is good. The result is good. The result is good.”

This is one of the most effective solutions.

  1. Use frequency penalty and presence penalty

Frequency penalty:

  • Penalizes tokens based on how many times they have appeared.

Presence penalty:

  • Penalizes tokens if they have appeared at least once.

These help encourage diversity in output.

  1. Use beam search with length penalty

If using beam search:

  • Add a length penalty to prevent very short repetitive outputs.

  • Use diverse beam search to increase variety.

  1. Limit maximum tokens

If loops continue indefinitely:

  • Set a max_tokens limit.

  • Ensure stop sequences are defined properly.

  1. Improve prompt design

Repetition can also be prompt-induced.

  • Be explicit in instructions.

  • Add constraints like:
    “Do not repeat sentences.”
    “Provide concise and non-repetitive output.”

  1. Fine-tuning or better training data (advanced case)

If repetition is persistent:

  • Model may be poorly fine-tuned.

  • Training data may contain repetitive patterns.

In most practical systems, repetition penalty combined with controlled temperature and Top-P sampling is enough to prevent looping behavior.

How can you fix hallucinations or inaccurate answers using prompts?

Hallucinations happen when the model generates confident but incorrect or fabricated information. Since LLMs predict text based on patterns rather than verifying facts, prompt design plays a major role in reducing hallucinations.

Ways to reduce hallucinations using prompts:

  1. Ask the model to admit uncertainty

Explicitly instruct the model not to guess.

Example:

  • “If you do not know the answer, say ‘I don’t know.’”

  • “Do not make up facts. If unsure, say that you are unsure.”

This reduces fabricated answers significantly.

  1. Ask for step-by-step reasoning

Encourage structured reasoning.

Example:

  • “Explain your reasoning step by step before giving the final answer.”

  • “Show the logic used to reach the conclusion.”

This forces the model to think through the problem instead of generating a direct guess.

  1. Request sources or justification

Ask the model to justify claims.

Example:

  • “Cite the source of this information.”

  • “Explain why this answer is correct.”

If the model cannot justify properly, hallucinations are more likely to be detected.

  1. Constrain the output format

Force structured responses.

Example:

  • “Answer only using the information provided below.”

  • “Provide answer in JSON format with fields: fact, source.”

Structured prompts reduce free-form fabrication.

  1. Provide context (Grounding)

Give relevant information in the prompt.

Example:

  • Provide a paragraph of verified data.

  • Ask: “Answer only based on the context provided.”

Grounded prompts significantly reduce hallucination.

  1. Use negative instructions carefully

Explicit constraints help.

Example:

  • “Do not assume missing information.”

  • “Do not create fictional names, dates, or references.”

Clear boundaries reduce creative guessing.

  1. Break complex tasks into smaller steps

Instead of asking one large question:

Bad:

  • “Explain everything about quantum computing.”

Better:

  • “Define quantum computing.”

  • “List three real-world applications.”

  • “Explain one application in detail.”

Stepwise prompting reduces speculative generation.

  1. Use verification prompts

Ask the model to verify its own answer.

Example:

  • “Check the above answer for factual inaccuracies.”

  • “List possible errors in your response.”

Self-check prompting can catch some hallucinations.

  1. Encourage concise answers

Long outputs increase chance of fabricated details.

Example:

  • “Provide a short, factual answer in 3 sentences.”

Concise responses reduce speculative content.

Important limitation

Prompting can reduce hallucinations, but cannot eliminate them completely. For high-stakes use cases, retrieval-augmented generation (RAG) or external verification systems are more reliable than prompting alone.

How do you handle long or multi-step tasks using prompts?

Handling long or multi-step tasks requires structuring the prompt so the model does not try to solve everything in one uncontrolled generation. The goal is to break complexity into manageable steps and guide reasoning clearly.

  1. Break the task into explicit steps

Instead of asking:

“Write a business plan for a startup.”

Structure it like:

Step 1: Define the problem.
Step 2: Define the target audience.
Step 3: Describe the solution.
Step 4: Outline revenue model.
Step 5: Provide a marketing strategy.

This reduces hallucination and improves logical flow.

  1. Use step-by-step reasoning

Explicitly instruct the model:

  • “Think step by step.”

  • “First analyze the problem, then propose the solution.”

  • “Explain reasoning before final answer.”

This improves logical consistency in multi-step problems.

  1. Use staged prompting (multi-turn approach)

Instead of one large prompt, use multiple prompts:

Prompt 1: Generate outline.
Prompt 2: Expand section 1.
Prompt 3: Expand section 2.

This gives better control and reduces drift.

  1. Ask for intermediate outputs

Force the model to show structure before continuing.

Example:

“First provide a numbered outline only. Do not write full content yet.”

Then:

“Now expand section 1.”

This prevents the model from losing track.

  1. Use role and constraints

Define the role and format clearly:

“You are a technical architect.
Provide solution in the following format:

  1. Problem

  2. Assumptions

  3. Architecture

  4. Risks”

Clear structure reduces confusion.

  1. Use memory or summarization for long contexts

For very long tasks:

  • Ask the model to summarize previous output.

  • Then continue using the summary.

Example:

“Summarize the above in 5 bullet points.
Now build a strategy based on that summary.”

This helps manage context window limits.

  1. Use check-and-revise prompting

After completing a task:

“Review the above solution for logical gaps.
List improvements.”

Then refine based on feedback.

  1. Limit scope per step

Avoid:

“Design, implement, optimize, document, and test this system.”

Instead split:

  • Design

  • Implementation plan

  • Optimization strategy

  • Testing strategy

  1. Use structured output formats

JSON, bullet points, or numbered lists reduce ambiguity and help maintain organization in multi-step workflows.

Example:

“Return output in JSON with keys: analysis, solution, risks, conclusion.”

  1. Control generation length

For complex reasoning tasks:

  • Use smaller max tokens per stage.

  • Generate progressively rather than all at once.

For multi-step tasks, structured decomposition, staged prompting, and controlled reasoning are more reliable than a single large prompt.

What is Chain of Thought (CoT) prompting?

Chain of Thought prompting is a technique where the model is encouraged to generate intermediate reasoning steps before giving the final answer.

Instead of directly answering:

“Think step by step and then give the final answer.”

This helps in problems that require logical reasoning, math, or multi-step thinking.

Example:

Question: If a train travels 60 km in 1 hour, how far in 3 hours?

With CoT:
Step 1: Speed is 60 km per hour.
Step 2: Time is 3 hours.
Step 3: Distance = Speed × Time = 60 × 3 = 180 km.
Final answer: 180 km.

Why it works:
Breaking reasoning into steps reduces logical errors and improves performance on reasoning-heavy tasks.

What is self-consistency prompting?

Self-consistency is an extension of Chain of Thought.

Instead of generating one reasoning path, the model generates multiple reasoning chains and selects the most consistent final answer.

Process:

  1. Generate multiple reasoning paths (with randomness).

  2. Extract final answers from each.

  3. Choose the most frequent answer.

Why it helps:

Sometimes one reasoning chain may contain an error. By sampling multiple chains, the model reduces the probability of relying on a single flawed reasoning path.

It improves accuracy in math and logical problems.

What is Tree of Thoughts (ToT)?

Tree of Thoughts is an advanced reasoning framework where the model explores multiple possible reasoning paths in a tree-like structure.

Instead of:

Single linear reasoning path (like CoT),

It does:

  1. Generate multiple possible next reasoning steps.

  2. Evaluate each step.

  3. Expand the promising ones.

  4. Prune weak reasoning paths.

  5. Continue exploration until reaching a solution.

It is similar to search algorithms in AI (like breadth-first or depth-first search).

Used for:

  • Complex planning

  • Strategy games

  • Multi-step decision making

  • Optimization problems

It treats reasoning as a search problem.

What is the difference between Chain of Thought and ReAct prompting?

Chain of Thought:

  • Focuses on step-by-step reasoning.

  • Pure reasoning within the model.

  • No external interaction.

ReAct (Reason + Act):

  • Combines reasoning and external actions.

  • The model reasons, then takes an action (like calling a tool or searching).

  • Observes the result.

  • Continues reasoning.

Structure:

Reason → Act → Observe → Reason → Act → Observe → Final Answer

Key difference:

CoT:
Internal reasoning only.

ReAct:
Reasoning + tool usage + environment interaction.

ReAct is commonly used in agent systems and retrieval-based tasks.


[Image/Media]

How does Tree of Thoughts improve on Chain of Thought?

Chain of Thought:

  • Follows a single linear reasoning path.

  • If that path is wrong, final answer is likely wrong.

Tree of Thoughts:

  • Explores multiple reasoning branches.

  • Evaluates intermediate steps.

  • Prunes bad paths.

  • Selects better solution paths.

Improvements over CoT:

  1. Not limited to one reasoning chain.

  2. Can backtrack if a reasoning path fails.

  3. Better for planning and combinatorial problems.

  4. More robust in complex decision-making tasks.

In short:

CoT = Linear reasoning
ToT = Search-based reasoning over multiple possibilities

What is ReAct prompting?

ReAct stands for Reason + Act.

It is a prompting framework where the model alternates between:

  • Reasoning (thinking step by step)

  • Taking actions (calling tools, searching, querying databases)

  • Observing results

  • Continuing reasoning

Structure:

Thought → Action → Observation → Thought → Action → Observation → Final Answer

Example flow:

Thought: I need to check the latest stock price.
Action: Call stock API.
Observation: Stock price is $150.
Thought: Now calculate growth percentage.
Final Answer: The stock increased by 5%.

Key idea:
ReAct combines reasoning with external tool usage.
It is widely used in agent-based systems and retrieval-augmented systems.

What is Automatic Prompt Engineering (APE)?

Automatic Prompt Engineering (APE) is a method where prompts are automatically generated, evaluated, and optimized instead of being manually written.

Instead of a human writing prompts, the system:

  1. Generates multiple candidate prompts.

  2. Tests them on a task.

  3. Evaluates performance.

  4. Selects or refines the best prompt.

It treats prompt design as an optimization problem.


[Image/Media]

What is Automatic Prompt Engineering (APE) used for, and how do you judge the best prompt?

APE is used for:

  • Improving task accuracy.

  • Finding better instruction phrasing.

  • Automating prompt optimization.

  • Reducing manual trial-and-error in prompt writing.

How to judge the best prompt:

  1. Define evaluation metric

    • Accuracy (for classification)

    • BLEU/ROUGE (for summarization)

    • F1-score

    • Human evaluation

    • Task-specific KPIs

  2. Test prompts on validation dataset

    • Run same inputs through different prompts.

    • Compare outputs.

  3. Measure:

    • Correctness

    • Consistency

    • Clarity

    • Robustness to edge cases

The best prompt is the one that consistently performs highest according to defined metrics.

How can variables be used in prompts to handle dynamic user input?

Variables allow prompts to be reusable templates.

Instead of writing a static prompt:

“Explain machine learning to a beginner.”

You create a template:

“Explain {topic} to a {audience_level} in {tone}.”

Then dynamically replace:

{topic} = machine learning
{audience_level} = beginner
{tone} = simple and clear

Result:
“Explain machine learning to a beginner in simple and clear language.”

Benefits:

  • Personalization

  • Reusability

  • Scalability

  • Dynamic content generation

In production systems, variables are filled using:

  • User input

  • Database values

  • API responses

  • Session memory

This is essential in chatbot systems.

How can you design a prompt to extract structured data from messy text?

To extract structured data from messy or unstructured text, the prompt must:

  1. Clearly define the task
    Example:
    “Extract the following fields from the text: Name, Email, Phone, Company.”

  2. Specify output format
    Example:
    “Return the output strictly in JSON format.”

  3. Define handling of missing values
    Example:
    “If a field is not present, return null.”

  4. Provide an example (few-shot prompting)

Example prompt:

“Extract the following fields from the text:

  • Full Name

  • Email

  • Phone Number

  • Company

Return the result in valid JSON format.
If any field is missing, set it to null.
Text:
{input_text}”

Good practices:

  • Be explicit about field names.

  • Enforce strict formatting.

  • Instruct the model not to add extra commentary.

  • Use validation phrases like “Return only JSON.”

How can you design prompts that reduce bias and generate safe outputs?

  1. Define safety constraints explicitly
    Example:
    “Avoid stereotypes, discriminatory language, or assumptions about gender, race, religion, or nationality.”

  2. Ask for evidence-based reasoning
    “Base the response on widely accepted research or neutral facts.”

  3. Avoid leading or biased phrasing
    Instead of:
    “Why are group X bad at Y?”
    Use:
    “Discuss factors influencing performance in Y without attributing it to any specific demographic group.”

  4. Instruct balanced perspectives
    “Provide multiple viewpoints without favoring one side.”

  5. Add refusal instructions
    “If the request involves harmful, illegal, or unsafe actions, decline and explain why.”

  6. Limit speculation
    “Do not make assumptions about personal characteristics unless explicitly stated.”

Clear constraints reduce biased or unsafe outputs.

How can prompt constraints help avoid unsafe or incorrect outputs?

Prompt constraints limit what the model is allowed to do.

Examples:

  1. Scope limitation
    “Answer only using the information provided.”

  2. Refusal rule
    “If the request involves illegal or harmful activity, refuse.”

  3. Format restriction
    “Return only verified factual statements.”

  4. Uncertainty handling
    “If unsure, say ‘I don’t know.’ Do not guess.”

  5. Domain restriction
    “Do not provide medical or legal advice.”

Constraints reduce hallucination, speculation, and unsafe content generation.

How do system-level instructions help control model behavior in sensitive tasks?

System-level instructions define global behavior before user input is processed.

They:

  1. Set role and identity
    “You are a licensed medical information assistant providing general educational information only.”

  2. Enforce safety boundaries
    “Never provide instructions for harmful activities.”

  3. Define tone and professionalism
    “Maintain calm, respectful, and non-judgmental language.”

  4. Establish refusal policy
    “Decline requests involving violence, hate speech, or illegal guidance.”

  5. Control scope
    “Provide high-level information only, not personalized medical diagnosis.”

System instructions act as persistent guardrails.
They ensure consistent behavior across all user interactions, especially in domains like healthcare, finance, law, or mental health.
