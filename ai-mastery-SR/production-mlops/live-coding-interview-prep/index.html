RAG & Retrieval Systems

Build a basic RAG pipeline using embeddings and a vector database?

Definition

A RAG (Retrieval-Augmented Generation) pipeline combines a vector database (for retrieval) with an LLM (for generation). It retrieves relevant context using embeddings and uses that context to generate accurate answers.

Why it is used

  • Reduces hallucinations

  • Uses external knowledge (docs, PDFs, DBs)

  • Improves answer relevance

  • Keeps LLM responses grounded

How it works (step-by-step)

  1. Collect documents (PDFs, text, etc.)

  2. Split into chunks

  3. Convert chunks into embeddings

  4. Store embeddings in a vector database

  5. Convert user query into embedding

  6. Retrieve top-k similar chunks

  7. Pass retrieved context + query to LLM

  8. Generate final answer

Key components

  • Embedding model: Converts text → vectors

  • Vector database: Stores and retrieves embeddings

  • Retriever: Finds relevant chunks

  • LLM: Generates answer using context

  • Chunking strategy: Splits documents effectively

Basic implementation (Python example)

from sentence_transformers import SentenceTransformer
import faiss
import numpy as np
from openai import OpenAI

# 1. Load embedding model
embed_model = SentenceTransformer('all-MiniLM-L6-v2')

# 2. Sample documents
documents = [
    "LLMs are large language models used for NLP tasks.",
    "RAG combines retrieval with generation.",
    "Vector databases store embeddings for semantic search."
]

# 3. Create embeddings
doc_embeddings = embed_model.encode(documents)

# 4. Store in FAISS index
index = faiss.IndexFlatL2(doc_embeddings.shape[1])
index.add(np.array(doc_embeddings))

# 5. Query
query = "What is RAG?"
query_embedding = embed_model.encode([query])

# 6. Retrieve top-k documents
k = 2
distances, indices = index.search(np.array(query_embedding), k)
retrieved_docs = [documents[i] for i in indices[0]]

# 7. Build prompt
context = "\n".join(retrieved_docs)
prompt = f"Answer based on context:\n{context}\n\nQuestion: {query}"

# 8. Generate response (example with OpenAI)
client = OpenAI()

response = client.chat.completions.create(
    model="gpt-4.1-mini",
    messages=[{"role": "user", "content": prompt}]
)

print(response.choices[0].message.content)

Advantages

  • Improves accuracy using external knowledge

  • Reduces hallucinations

  • Scales to large datasets

  • Keeps models lightweight (no retraining needed)

Disadvantages

  • Retrieval quality affects output

  • Adds latency (retrieval + generation)

  • Requires tuning (chunking, top-k, embeddings)

Real-world example

A chatbot answering questions from company documents:

  • Stores docs in vector DB

  • Retrieves relevant sections per query

  • LLM generates answer using that context

Short interview-ready answer

A basic RAG pipeline involves chunking documents, converting them into embeddings, and storing them in a vector database. At query time, the user query is embedded, relevant chunks are retrieved using similarity search, and the LLM generates an answer using this context. This improves accuracy and reduces hallucinations by grounding responses in real data.

Implement semantic search using cosine similarity.

Definition

Semantic search using cosine similarity finds relevant results based on meaning by comparing vector embeddings of text instead of exact keyword matches.

Why it is used

  • Captures meaning, not just keywords

  • Improves search relevance

  • Works well with unstructured text

  • Core technique in RAG systems

How it works (step-by-step)

  1. Convert documents into embeddings (vectors)

  2. Store embeddings

  3. Convert user query into embedding

  4. Compute cosine similarity between query and documents

  5. Rank results by similarity score

  6. Return top-k most relevant documents

Key components

  • Embedding model: Converts text → vector

  • Vector store: Stores embeddings

  • Cosine similarity: Measures similarity between vectors

  • Top-k retrieval: Returns best matches

Cosine similarity formula

\cos(\theta) = \frac{A \cdot B}{|A| |B|}

Basic implementation (Python)

from sentence_transformers import SentenceTransformer
import numpy as np

# 1. Load embedding model
model = SentenceTransformer('all-MiniLM-L6-v2')

# 2. Documents
documents = [
    "LLMs are used for natural language processing",
    "Semantic search finds meaning not keywords",
    "Vector databases store embeddings"
]

# 3. Create embeddings
doc_embeddings = model.encode(documents)

# 4. Define cosine similarity
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# 5. Query
query = "What is semantic search?"
query_embedding = model.encode(query)

# 6. Compute similarities
scores = [cosine_similarity(query_embedding, doc_emb) for doc_emb in doc_embeddings]

# 7. Rank results
top_k = 2
top_indices = np.argsort(scores)[::-1][:top_k]

# 8. Retrieve results
results = [documents[i] for i in top_indices]

print("Top results:", results)

Advantages

  • Better relevance than keyword search

  • Handles synonyms and context

  • Simple and effective approach

Disadvantages

  • Requires embedding model

  • Slightly more compute than keyword search

  • Depends on embedding quality

Real-world example

A search system for support articles:

  • User asks: “How to improve search results?”

  • System retrieves articles about “semantic search” even without exact keyword match

Short interview-ready answer

Semantic search uses embeddings to represent text and cosine similarity to measure how similar two vectors are. By comparing the query embedding with document embeddings, we can rank and retrieve the most relevant results based on meaning rather than exact keywords.

Write code for a hybrid search system (keyword + vector search).

Definition

Hybrid search combines keyword-based search (e.g., BM25) with semantic vector search (embeddings) to get the best of both worlds: exact matches + meaning-based relevance.

Why it is used

  • Improves search accuracy

  • Handles both exact terms and semantic intent

  • Reduces missed results from pure vector search

  • Common in production RAG systems

How it works (step-by-step)

  1. Store documents

  2. Build a keyword index (BM25 or TF-IDF)

  3. Generate embeddings for documents

  4. On query:

    • Run keyword search → get scores

    • Run vector search → get similarity scores

  5. Normalize and combine both scores

  6. Rank results and return top-k

Key components

  • Keyword search: BM25 / TF-IDF

  • Embedding model: Converts text → vectors

  • Vector similarity: Cosine similarity

  • Score fusion: Combines keyword + semantic scores

Basic implementation (Python)

from sentence_transformers import SentenceTransformer
from sklearn.feature_extraction.text import TfidfVectorizer
import numpy as np

# 1. Documents
documents = [
    "LLMs are used for natural language processing",
    "Semantic search improves search relevance",
    "Vector databases store embeddings for retrieval"
]

# 2. Keyword search (TF-IDF)
vectorizer = TfidfVectorizer()
tfidf_matrix = vectorizer.fit_transform(documents)

# 3. Embedding model
embed_model = SentenceTransformer('all-MiniLM-L6-v2')
doc_embeddings = embed_model.encode(documents)

# 4. Cosine similarity
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# 5. Query
query = "How does semantic search work?"

# Keyword score
query_tfidf = vectorizer.transform([query])
keyword_scores = (tfidf_matrix @ query_tfidf.T).toarray().flatten()

# Vector score
query_embedding = embed_model.encode(query)
vector_scores = np.array([
    cosine_similarity(query_embedding, emb) for emb in doc_embeddings
])

# 6. Normalize scores
keyword_scores = keyword_scores / (keyword_scores.max() + 1e-8)
vector_scores = vector_scores / (vector_scores.max() + 1e-8)

# 7. Combine scores (weighted)
alpha = 0.5  # balance factor
final_scores = alpha * keyword_scores + (1 - alpha) * vector_scores

# 8. Rank results
top_k = 2
top_indices = np.argsort(final_scores)[::-1][:top_k]
results = [documents[i] for i in top_indices]

print("Top results:", results)

Advantages

  • Better recall and precision

  • Handles exact keywords + semantic meaning

  • More robust than single approach

Disadvantages

  • Slightly more complex

  • Requires tuning weights (alpha)

  • Higher compute cost

Real-world example

A product search system:

  • Keyword search matches exact product names

  • Vector search finds similar items (synonyms, intent)

  • Hybrid ranking gives best results

Short interview-ready answer

Hybrid search combines keyword-based methods like TF-IDF/BM25 with vector similarity search. It computes scores from both approaches, normalizes and merges them, and ranks results. This improves relevance by capturing both exact matches and semantic meaning.

Implement a simple re-ranking model for retrieved results.

Definition

Re-ranking is a step after retrieval where an initial list of results is reordered using a more accurate model (usually slower but better) to improve final relevance.

Why it is used

  • Improves precision of top results

  • Fixes errors from initial retrieval (BM25/vector search)

  • Enhances answer quality in RAG systems

How it works (step-by-step)

  1. Retrieve top-k candidates using fast search (vector/BM25)

  2. Pair each candidate with the query

  3. Pass (query, document) pairs to a re-ranking model

  4. Get relevance scores

  5. Sort results based on new scores

  6. Return top-ranked results

Key components

  • Retriever: Fast, high-recall (vector/BM25)

  • Re-ranker model: Cross-encoder or scoring model

  • Scoring function: Assigns relevance score

  • Top-k selection: Final ranked output

Basic implementation (Python)

from sentence_transformers import CrossEncoder

# 1. Query and retrieved documents (from retriever)
query = "What is semantic search?"
retrieved_docs = [
    "Semantic search finds meaning rather than exact keywords.",
    "Vector databases store embeddings.",
    "LLMs are used for NLP tasks."
]

# 2. Load cross-encoder model (re-ranker)
reranker = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2')

# 3. Prepare (query, doc) pairs
pairs = [[query, doc] for doc in retrieved_docs]

# 4. Get relevance scores
scores = reranker.predict(pairs)

# 5. Rank results
ranked = sorted(zip(retrieved_docs, scores), key=lambda x: x[1], reverse=True)

# 6. Output
for doc, score in ranked:
    print(f"Score: {score:.4f} | Doc: {doc}")

Advantages

  • Higher accuracy than retrieval alone

  • Better ranking of top results

  • Improves final LLM response quality

Disadvantages

  • Slower than retrieval

  • Adds extra compute cost

  • Usually applied only on top-k results

Real-world example

In a RAG chatbot:

  • Retriever fetches top 10 documents

  • Re-ranker selects best 3 based on query relevance

  • LLM uses only top 3 → better answers

Short interview-ready answer

A re-ranking model improves retrieval results by scoring query-document pairs using a more accurate model like a cross-encoder. After retrieving top-k candidates, it reorders them based on relevance scores, ensuring the most relevant results are used for final output.

Build a vector similarity search engine from scratch.

Definition

A vector similarity search engine finds the most similar items by comparing vector embeddings using a similarity metric (like cosine similarity).

Why it is used

  • Enables semantic search (meaning-based)

  • Core building block for RAG systems

  • Works with unstructured data (text, images)

  • Fast retrieval of relevant information

How it works (step-by-step)

  1. Convert data into embeddings (vectors)

  2. Store vectors in an index

  3. Convert query into embedding

  4. Compute similarity between query and stored vectors

  5. Rank results by similarity

  6. Return top-k matches

Key components

  • Embedding model: Text → vector

  • Vector store: Stores embeddings

  • Similarity function: Cosine similarity

  • Search function: Retrieves top-k results

Basic implementation (Python from scratch)

import numpy as np
from sentence_transformers import SentenceTransformer

class VectorSearchEngine:
    def __init__(self, model_name='all-MiniLM-L6-v2'):
        self.model = SentenceTransformer(model_name)
        self.documents = []
        self.embeddings = None

    def add_documents(self, docs):
        self.documents.extend(docs)
        new_embeddings = self.model.encode(docs)
        
        if self.embeddings is None:
            self.embeddings = np.array(new_embeddings)
        else:
            self.embeddings = np.vstack([self.embeddings, new_embeddings])

    def cosine_similarity(self, a, b):
        return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

    def search(self, query, top_k=3):
        query_embedding = self.model.encode(query)
        
        scores = [
            self.cosine_similarity(query_embedding, doc_emb)
            for doc_emb in self.embeddings
        ]
        
        top_indices = np.argsort(scores)[::-1][:top_k]
        
        results = [(self.documents[i], scores[i]) for i in top_indices]
        return results

# Example usage
engine = VectorSearchEngine()

docs = [
    "LLMs are used for natural language processing",
    "Semantic search improves search relevance",
    "Vector databases store embeddings",
    "RAG combines retrieval and generation"
]

engine.add_documents(docs)

results = engine.search("What is semantic search?", top_k=2)

for doc, score in results:
    print(f"Score: {score:.4f} | Doc: {doc}")

Advantages

  • Simple and easy to implement

  • Works well for small to medium datasets

  • Fully customizable

Disadvantages

  • Not scalable for large datasets (O(n) search)

  • Slower than optimized vector DBs (FAISS, HNSW)

  • No indexing or optimization

Real-world example

A document search system:

  • Converts all documents into embeddings

  • Stores them in memory

  • Finds most relevant docs for a query using cosine similarity

Short interview-ready answer

A vector similarity search engine stores embeddings of documents and compares them with a query embedding using a similarity metric like cosine similarity. It ranks results based on similarity scores and returns the most relevant matches, forming the foundation of semantic search systems.

Write cosine similarity, dot product, and Euclidean distance functions manually.

Definition

These are basic vector operations used in similarity search:

  • Dot product: Measures alignment between two vectors

  • Cosine similarity: Measures angle-based similarity

  • Euclidean distance: Measures straight-line distance

Why it is used

  • Core math behind vector search and embeddings

  • Helps compare similarity between texts/images

  • Used in semantic search, clustering, recommendation systems

How it works (step-by-step)

  1. Take two vectors (lists of numbers)

  2. Apply mathematical formula

  3. Return a similarity or distance score

  4. Use score to rank or compare results

Key components

  • Vectors: Numeric representations (same length)

  • Norm (magnitude): Length of a vector

  • Element-wise operations: Multiply, subtract, sum

Implementations (from scratch in Python)

import math

# Dot Product
def dot_product(a, b):
    return sum(x * y for x, y in zip(a, b))


# Euclidean Distance
def euclidean_distance(a, b):
    return math.sqrt(sum((x - y) ** 2 for x, y in zip(a, b)))


# Cosine Similarity
def cosine_similarity(a, b):
    dot = dot_product(a, b)
    
    norm_a = math.sqrt(sum(x ** 2 for x in a))
    norm_b = math.sqrt(sum(y ** 2 for y in b))
    
    if norm_a == 0 or norm_b == 0:
        return 0.0
    
    return dot / (norm_a * norm_b)


# Example usage
vec1 = [1, 2, 3]
vec2 = [4, 5, 6]

print("Dot Product:", dot_product(vec1, vec2))
print("Euclidean Distance:", euclidean_distance(vec1, vec2))
print("Cosine Similarity:", cosine_similarity(vec1, vec2))

Advantages

  • Simple and efficient

  • Foundation for many ML algorithms

  • Easy to implement and understand

Disadvantages

  • Requires vectors of same dimension

  • Performance issues for very large datasets (without optimization)

Real-world example

In semantic search:

  • Convert text to embeddings

  • Use cosine similarity to find most similar documents

Short interview-ready answer

Dot product multiplies and sums corresponding vector elements, Euclidean distance measures straight-line distance between vectors, and cosine similarity measures the angle between them. These functions are fundamental for comparing embeddings in tasks like semantic search and recommendation systems.

Implement query expansion for better retrieval results.

Definition

Query expansion improves retrieval by rewriting or extending the user query with related terms (synonyms, phrases, or semantic variants) to fetch more relevant results.

Why it is used

  • Handles vague or short queries

  • Improves recall (finds more relevant docs)

  • Captures synonyms and variations

  • Boosts performance of search/RAG systems

How it works (step-by-step)

  1. Take the original query

  2. Generate related terms (synonyms, paraphrases, keywords)

  3. Create expanded queries

  4. Run search for each query

  5. Merge and rank results

  6. Return top-k results

Key components

  • Expansion method: Synonyms, embeddings, or LLM

  • Retriever: Search system (vector/BM25)

  • Fusion logic: Combine results from multiple queries

  • Ranking: Final scoring

Common techniques

Synonym-based expansion

  • Use predefined dictionary or WordNet

Embedding-based expansion

  • Find similar terms using vector similarity

LLM-based expansion

  • Generate multiple query variations

Multi-query retrieval

  • Run multiple queries and merge results

Basic implementation (Python)

from sentence_transformers import SentenceTransformer
import numpy as np

# Simple dataset
documents = [
    "Semantic search improves retrieval quality",
    "Keyword search matches exact terms",
    "Vector databases enable similarity search",
    "LLMs are used for natural language understanding"
]

# Embedding model
model = SentenceTransformer('all-MiniLM-L6-v2')
doc_embeddings = model.encode(documents)

# Cosine similarity
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Simple query expansion (manual synonyms)
def expand_query(query):
    expansions = {
        "semantic": ["meaning", "contextual"],
        "search": ["retrieval", "lookup"]
    }
    
    expanded_queries = [query]
    
    for word, synonyms in expansions.items():
        if word in query.lower():
            for syn in synonyms:
                expanded_queries.append(query.replace(word, syn))
    
    return expanded_queries

# Search function
def search(query, top_k=2):
    query_embedding = model.encode(query)
    scores = [
        cosine_similarity(query_embedding, emb) for emb in doc_embeddings
    ]
    top_indices = np.argsort(scores)[::-1][:top_k]
    return [documents[i] for i in top_indices]

# Hybrid search with query expansion
def expanded_search(query, top_k=2):
    expanded_queries = expand_query(query)
    
    all_results = []
    
    for q in expanded_queries:
        results = search(q, top_k)
        all_results.extend(results)
    
    # Remove duplicates and return top_k
    unique_results = list(dict.fromkeys(all_results))
    return unique_results[:top_k]

# Example usage
query = "semantic search"
results = expanded_search(query)

print("Expanded results:", results)

Advantages

  • Improves recall and coverage

  • Handles synonyms and variations

  • Enhances search quality

Disadvantages

  • May introduce noise (irrelevant results)

  • Increases compute (multiple queries)

  • Needs tuning

Real-world example

User searches: “cheap flights”

  • Expanded to: “low-cost flights”, “budget airlines”

  • Retrieves more relevant travel results

Short interview-ready answer

Query expansion improves retrieval by generating variations of the original query using synonyms or semantic techniques. These expanded queries are searched separately, and results are merged and ranked, increasing recall and improving overall search quality.

Build a multi-query RAG system that merges results.

A multi-query RAG system generates multiple variations of a user query, retrieves results for each, and merges them to improve coverage and relevance before passing context to the LLM.

from sentence_transformers import SentenceTransformer
import numpy as np

# Sample documents
documents = [
    "Semantic search improves retrieval quality",
    "Vector databases store embeddings for search",
    "RAG combines retrieval and generation",
    "Keyword search matches exact terms",
    "LLMs are used for natural language understanding"

# Cosine similarity
def cosine_similarity(a, b):

def generate_queries(query):
    return [
        query,
        f"What is {query}?",
        f"Explain {query}",
        f"How does {query} work?"

def retrieve(query, top_k=2):

Implement context compression before passing data to the LLM.

Definition

Context compression reduces the size of retrieved documents before sending them to the LLM, keeping only the most relevant information to fit within token limits and improve efficiency.

Why it is used

  • Fits large context into limited token window

  • Reduces cost and latency

  • Removes irrelevant information

  • Improves answer quality

How it works (step-by-step)

  1. Retrieve top-k documents

  2. Split documents into smaller chunks/sentences

  3. Score each chunk based on relevance to query

  4. Select top relevant chunks

  5. Optionally summarize or trim content

  6. Combine compressed context

  7. Send to LLM

Key components

  • Chunking: Break documents into smaller units

  • Scoring function: Measures relevance (cosine similarity)

  • Selection strategy: Top-k or threshold-based

  • Optional summarization: Further compress content

Basic implementation (Python)

from sentence_transformers import SentenceTransformer
import numpy as np

# Sample retrieved documents
documents = [
    "Semantic search improves retrieval by understanding meaning instead of keywords.",
    "It uses embeddings and vector similarity to find relevant results.",
    "Traditional keyword search relies on exact matches.",
    "Vector databases store embeddings efficiently for fast retrieval."
]

# Load embedding model
model = SentenceTransformer('all-MiniLM-L6-v2')

# Cosine similarity
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Context compression
def compress_context(query, docs, top_k=2):
    query_embedding = model.encode(query)
    
    # Split into smaller chunks (simple split)
    chunks = []
    for doc in docs:
        chunks.extend(doc.split(". "))
    
    # Compute scores
    chunk_embeddings = model.encode(chunks)
    scores = [
        cosine_similarity(query_embedding, emb)
        for emb in chunk_embeddings
    ]
    
    # Select top-k chunks
    top_indices = np.argsort(scores)[::-1][:top_k]
    selected_chunks = [chunks[i] for i in top_indices]
    
    return " ".join(selected_chunks)

# Example usage
query = "How does semantic search work?"
compressed_context = compress_context(query, documents)

print("Compressed context:", compressed_context)

Advantages

  • Reduces token usage and cost

  • Improves latency

  • Focuses LLM on relevant information

  • Works well with RAG pipelines

Disadvantages

  • Risk of losing important context

  • Requires good scoring strategy

  • Adds preprocessing step

Real-world example

A chatbot retrieves 10 long documents:

  • Compression selects only top relevant sentences

  • LLM receives concise context

  • Produces faster and more accurate response

Short interview-ready answer

Context compression reduces retrieved data by selecting only the most relevant chunks using techniques like similarity scoring or summarization. This minimizes token usage, improves latency, and ensures the LLM focuses on the most important information for generating accurate responses.

def retry_with_backoff(func, max_retries=5, base_delay=1):
    for attempt in range(max_retries):
        try:
            return func()
        except Exception as e:
            print(f"Attempt {attempt+1} failed: {e}")

            if attempt == max_retries - 1:
                return "Final failure"

            # Exponential backoff with jitter
            delay = base_delay * (2 ** attempt) + random.uniform(0, 0.5)
            print(f"Retrying in {delay:.2f} seconds...")
            time.sleep(delay)

import time
import hashlib

# -------- Simple Cache -------- #

class Cache:
    def __init__(self, ttl=60):
        self.store = {}
        self.ttl = ttl

    def _is_expired(self, timestamp):
        return (time.time() - timestamp) > self.ttl

    def get(self, key):
        if key in self.store:
            value, timestamp = self.store[key]
            if not self._is_expired(timestamp):
                return value
            else:
                del self.store[key]
        return None

    def set(self, key, value):
        self.store[key] = (value, time.time())

# -------- LLM Mock -------- #

def call_llm(prompt):
    return f"LLM response for: {prompt}"

# -------- Cache Layer -------- #

cache = Cache(ttl=120)

def generate_cache_key(query):
    return hashlib.md5(query.encode()).hexdigest()

def get_response(query):
    key = generate_cache_key(query)

    # Check cache
    cached = cache.get(key)
    if cached:
        print("Cache hit")
        return cached

    # Call LLM
    print("Cache miss → calling LLM")
    response = call_llm(query)

    # Store in cache
    cache.set(key, response)

    return response

Implement semantic caching for similar queries.

Definition

Semantic caching stores past queries and responses using embeddings and retrieves a cached response if a new query is semantically similar, not just an exact match.

Why it is used

  • Handles similar queries (not just identical ones)

  • Reduces LLM calls → lowers cost

  • Improves response time

  • Enhances user experience

How it works (step-by-step)

  1. Convert incoming query into embedding

  2. Compare with stored query embeddings

  3. Compute similarity (cosine similarity)

  4. If similarity > threshold → return cached response

  5. Else → call LLM and store result

Key components

  • Embedding model: Converts queries → vectors

  • Vector store: Stores past query embeddings

  • Similarity function: Cosine similarity

  • Threshold: Determines cache hit

Basic implementation (Python)

from sentence_transformers import SentenceTransformer
import numpy as np

# -------- Semantic Cache -------- #

class SemanticCache:
    def __init__(self, threshold=0.85):
        self.model = SentenceTransformer('all-MiniLM-L6-v2')
        self.queries = []
        self.responses = []
        self.embeddings = []
        self.threshold = threshold

    def cosine_similarity(self, a, b):
        return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

    def get(self, query):
        if not self.embeddings:
            return None

        query_emb = self.model.encode(query)

        scores = [
            self.cosine_similarity(query_emb, emb)
            for emb in self.embeddings
        ]

        best_idx = np.argmax(scores)
        best_score = scores[best_idx]

        if best_score >= self.threshold:
            print(f"Semantic cache hit (score={best_score:.2f})")
            return self.responses[best_idx]

        return None

    def set(self, query, response):
        emb = self.model.encode(query)
        self.queries.append(query)
        self.responses.append(response)
        self.embeddings.append(emb)

# -------- LLM Mock -------- #

def call_llm(query):
    return f"LLM response for: {query}"

# -------- Cache Layer -------- #

cache = SemanticCache(threshold=0.8)

def get_response(query):
    cached = cache.get(query)
    if cached:
        return cached

    print("Cache miss → calling LLM")
    response = call_llm(query)
    cache.set(query, response)
    return response

print(get_response("Explain semantic search"))  # Similar → cache hit

import tiktoken

# -------- Config -------- #

MODEL_NAME = "gpt-4.1-mini"
MAX_TOKENS = 8000
RESERVED_OUTPUT_TOKENS = 500

# -------- Tokenizer -------- #

encoding = tiktoken.encoding_for_model(MODEL_NAME)

def count_tokens(text):
    return len(encoding.encode(text))

# -------- Context Management -------- #

def trim_context(messages):
    total_tokens = 0
    trimmed_messages = []

    # Iterate from latest messages (keep most recent)
    for msg in reversed(messages):
        msg_tokens = count_tokens(msg["content"])

        if total_tokens + msg_tokens > (MAX_TOKENS - RESERVED_OUTPUT_TOKENS):
            break

        trimmed_messages.insert(0, msg)
        total_tokens += msg_tokens

    return trimmed_messages

Basic implementation (Python - Token Bucket)

import time
import threading

class TokenBucket:
    def __init__(self, rate, capacity):
        self.rate = rate              # tokens per second
        self.capacity = capacity      # max tokens
        self.tokens = capacity
        self.last_refill = time.time()
        self.lock = threading.Lock()

    def allow_request(self):
        with self.lock:
            now = time.time()
            elapsed = now - self.last_refill

            # Refill tokens
            self.tokens = min(
                self.capacity,
                self.tokens + elapsed * self.rate
            )
            self.last_refill = now

            if self.tokens >= 1:
                self.tokens -= 1
                return True
            return False

Write code for fixed-size and recursive text chunking.

Definition

Text chunking splits large text into smaller pieces so LLMs can process it efficiently.

  • Fixed-size chunking: Splits text into equal-sized chunks

  • Recursive chunking: Splits text hierarchically (paragraph → sentence → words) to preserve meaning

  • Fits text into context window limits

How it works (step-by-step)

Fixed-size

  1. Take input text

  2. Split into chunks of size N

  3. Optionally add overlap

  4. Return chunks

Recursive

  1. Try splitting by large units (paragraphs)

  2. If chunk too large → split into sentences

  3. If still large → split into words

  4. Continue until size constraint satisfied

  • Chunk size: Max tokens/characters

  • Overlap: Preserve context between chunks

  • Split hierarchy: Paragraph → sentence → word

Fixed-size chunking (Python)

def fixed_chunk(text, chunk_size=100, overlap=20):
    chunks = []
    start = 0

    while start < len(text):
        end = start + chunk_size
        chunk = text[start:end]
        chunks.append(chunk)

        start += chunk_size - overlap

    return chunks


# Example

Recursive chunking (Python)

def recursive_chunk(text, chunk_size=100):
    separators = ["\n\n", ". ", " "]

    def split_text(text, level=0):
        if len(text) <= chunk_size:
            return [text]

        if level >= len(separators):
            # fallback: hard split
            return [text[i:i+chunk_size] for i in range(0, len(text), chunk_size)]

        sep = separators[level]
        parts = text.split(sep)

        chunks = []
        current = ""

        for part in parts:
            if len(current) + len(part) + len(sep) <= chunk_size:
                current += (sep if current else "") + part
            else:
                if current:
                    chunks.append(current)
                current = part

        if current:
            chunks.append(current)

        # recursively split large chunks
        final_chunks = []
        for chunk in chunks:
            if len(chunk) > chunk_size:
                final_chunks.extend(split_text(chunk, level + 1))
            else:
                final_chunks.append(chunk)

        return final_chunks

    return split_text(text)


# Example
text = """Semantic search improves retrieval.
It uses embeddings.
Traditional search relies on keywords."""

chunks = recursive_chunk(text, chunk_size=50)

for c in chunks:
    print(c)

Implement semantic chunking based on meaning.

Definition

Semantic chunking splits text into chunks based on meaning and context, instead of fixed size. It groups sentences that are semantically similar.

Why it is used

  • Preserves context and meaning

  • Improves retrieval quality in RAG

  • Avoids breaking important information

  • Produces more relevant chunks

How it works (step-by-step)

  1. Split text into sentences

  2. Convert each sentence into embeddings

  3. Compute similarity between consecutive sentences

  4. Group sentences with high similarity

  5. Start a new chunk when similarity drops below threshold

Key components

  • Sentence splitter

  • Embedding model

  • Similarity function (cosine)

  • Threshold: Controls chunk boundaries

Basic implementation (Python)

from sentence_transformers import SentenceTransformer
import numpy as np
import re

# Load embedding model
model = SentenceTransformer('all-MiniLM-L6-v2')

# Cosine similarity
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Sentence splitter
def split_sentences(text):
    return re.split(r'(?<=[.!?]) +', text)

# Semantic chunking
def semantic_chunk(text, threshold=0.7):
    sentences = split_sentences(text)
    embeddings = model.encode(sentences)

    chunks = []
    current_chunk = [sentences[0]]

    for i in range(1, len(sentences)):
        sim = cosine_similarity(embeddings[i-1], embeddings[i])

        if sim >= threshold:
            current_chunk.append(sentences[i])
        else:
            chunks.append(" ".join(current_chunk))
            current_chunk = [sentences[i]]

    # Add last chunk
    if current_chunk:
        chunks.append(" ".join(current_chunk))

    return chunks

Semantic search improves retrieval by understanding meaning.
It uses embeddings and vector similarity.
Traditional keyword search relies on exact matches.
Databases store structured data.
"""

chunks = semantic_chunk(text)

for i, c in enumerate(chunks, 1):
    print(f"Chunk {i}:\n{c}\n")

Handle "lost in the middle" by reordering context chunks.

Definition

The “lost in the middle” problem occurs when LLMs pay less attention to information placed in the middle of a long context.
Reordering context chunks mitigates this by placing the most relevant chunks at the beginning and end of the prompt.

Why it is used

  • Improves answer accuracy

  • Ensures critical information is noticed

  • Optimizes long-context prompts

  • Enhances RAG performance

How it works (step-by-step)

  1. Retrieve top-k relevant chunks

  2. Rank chunks by relevance score

  3. Place most important chunks at:

    • Beginning (high attention)

    • End (recency bias)

  4. Put less important chunks in the middle

  5. Build final prompt

Key components

  • Retriever: Gets relevant chunks

  • Scoring: Determines importance

  • Reordering strategy: Controls placement

  • Prompt builder: Formats final context

Reordering strategy

  • Top 1 → beginning

  • Top 2 → end

  • Remaining → middle

Basic implementation (Python)

def reorder_chunks(chunks, scores):
    """
    chunks: list of text chunks
    scores: corresponding relevance scores
    """

    # Sort by score (descending)
    ranked = sorted(zip(chunks, scores), key=lambda x: x[1], reverse=True)
    ranked_chunks = [c for c, _ in ranked]

    if len(ranked_chunks) <= 2:
        return ranked_chunks

    # Place most important at start and end
    reordered = []
    reordered.append(ranked_chunks[0])      # best → start
    reordered.extend(ranked_chunks[2:])     # middle chunks
    reordered.append(ranked_chunks[1])      # second best → end

    return reordered

    "Chunk B: highly relevant",

]

Improved strategy (balanced placement)

def balanced_reorder(chunks):
    result = []
    left, right = 0, len(chunks) - 1

    toggle = True
    while left <= right:
        if toggle:
            result.append(chunks[left])
            left += 1
        else:
            result.append(chunks[right])
            right -= 1
        toggle = not toggle

    return result

from openai import OpenAI
from sentence_transformers import SentenceTransformer
import numpy as np

client = OpenAI()
embed_model = SentenceTransformer('all-MiniLM-L6-v2')

# -------- Generate Multiple Responses -------- #

def generate_responses(query, n=3):
    responses = []

    for _ in range(n):
        res = client.chat.completions.create(
            model="gpt-4.1-mini",
            messages=[{"role": "user", "content": query}],
            temperature=0.7  # introduce variation
        )
        responses.append(res.choices[0].message.content)

    return responses

# -------- Similarity -------- #

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# -------- Consistency Check -------- #

def check_consistency(responses, threshold=0.75):
    embeddings = embed_model.encode(responses)

    scores = []
    for i in range(len(embeddings)):
        for j in range(i + 1, len(embeddings)):
            scores.append(cosine_similarity(embeddings[i], embeddings[j]))

    avg_score = sum(scores) / len(scores)

    return avg_score, avg_score >= threshold

    responses = generate_responses(query)
    score, is_consistent = check_consistency(responses)

    return {
        "responses": responses,
        "consistency_score": score,
        "is_reliable": is_consistent
    }

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Zero-shot classifier
class ZeroShotClassifier:
    def __init__(self, labels):
        self.labels = labels

Write code to compute confusion matrix and evaluation metrics.

Definition

A confusion matrix shows how many predictions were correct vs incorrect per class, and evaluation metrics (accuracy, precision, recall, F1) quantify model performance.

Why it is used

  • Understand model errors

  • Evaluate classification performance

  • Compare models

  • Identify class imbalance issues

How it works (step-by-step)

  1. Collect true labels and predicted labels

  2. Build confusion matrix

  3. Compute metrics from matrix

  4. Analyze results

Key components

  • TP (True Positive)

  • FP (False Positive)

  • FN (False Negative)

  • TN (True Negative)

Basic implementation (Python from scratch)

from collections import defaultdict

def confusion_matrix(y_true, y_pred):
    labels = list(set(y_true))
    matrix = {label: {l: 0 for l in labels} for label in labels}

    for t, p in zip(y_true, y_pred):
        matrix[t][p] += 1

    return matrix


def compute_metrics(y_true, y_pred):
    cm = confusion_matrix(y_true, y_pred)
    labels = cm.keys()

    metrics = {}

    for label in labels:
        TP = cm[label][label]
        FP = sum(cm[l][label] for l in labels if l != label)
        FN = sum(cm[label][l] for l in labels if l != label)
        TN = sum(cm[l1][l2] for l1 in labels for l2 in labels) - (TP + FP + FN)

        precision = TP / (TP + FP) if (TP + FP) else 0
        recall = TP / (TP + FN) if (TP + FN) else 0
        f1 = (2 * precision * recall / (precision + recall)) if (precision + recall) else 0

        metrics[label] = {
            "precision": precision,
            "recall": recall,
            "f1": f1
        }

    accuracy = sum(1 for t, p in zip(y_true, y_pred) if t == p) / len(y_true)

    return cm, metrics, accuracy


# -------- Example -------- #

y_true = ["tech", "sports", "tech", "finance", "sports"]
y_pred = ["tech", "tech", "tech", "finance", "sports"]

cm, metrics, accuracy = compute_metrics(y_true, y_pred)

print("Confusion Matrix:", cm)
print("Metrics:", metrics)

Build a simple topic modeling pipeline (BERTopic-style).

Definition

A BERTopic-style topic modeling pipeline extracts topics by combining:

  • Embeddings (semantic meaning)

  • Finds topics without labeled data

  • Works better than traditional LDA for semantic text

  • Handles noise and dynamic topics

  • Useful for analytics, search, and insights

  1. Convert documents → embeddings

  2. Reduce dimensions (UMAP)

  3. Cluster documents (HDBSCAN)

  4. Group documents per cluster

  5. Extract keywords per cluster (TF-IDF)

  6. Label topics

  • Embedding model: SentenceTransformers

  • Reducer: UMAP

  • Clusterer: HDBSCAN

Basic implementation (Python)

from sentence_transformers import SentenceTransformer
import umap
import hdbscan
from sklearn.feature_extraction.text import TfidfVectorizer
import numpy as np

# -------- Sample Data -------- #

docs = [
    "AI is transforming technology",
    "Machine learning improves AI systems",
    "Football is a popular sport",
    "Basketball players are tall",
    "Stock markets are volatile",
    "Investing in stocks is risky"
]

# -------- Step 1: Embeddings -------- #

model = SentenceTransformer('all-MiniLM-L6-v2')
embeddings = model.encode(docs)

# -------- Step 2: UMAP -------- #

reducer = umap.UMAP(n_components=5, metric='cosine')
reduced = reducer.fit_transform(embeddings)

# -------- Step 3: HDBSCAN -------- #

clusterer = hdbscan.HDBSCAN(min_cluster_size=2)
labels = clusterer.fit_predict(reduced)

# -------- Step 4: Group Docs -------- #

clusters = {}
for doc, label in zip(docs, labels):
    if label == -1:
        continue  # skip noise
    clusters.setdefault(label, []).append(doc)

# -------- Step 5: Extract Keywords -------- #

def extract_keywords(cluster_docs, top_k=3):
    vectorizer = TfidfVectorizer(stop_words='english')
    X = vectorizer.fit_transform(cluster_docs)
    scores = np.asarray(X.mean(axis=0)).flatten()
    terms = vectorizer.get_feature_names_out()
    top_indices = scores.argsort()[::-1][:top_k]
    return [terms[i] for i in top_indices]

# -------- Step 6: Topics -------- #

topics = {}
for label, cluster_docs in clusters.items():
    topics[label] = extract_keywords(cluster_docs)

# -------- Output -------- #

for label, keywords in topics.items():
    print(f"Topic {label}: {keywords}")

Build an end-to-end chatbot with memory, RAG, and evaluation pipeline.

Definition

An end-to-end chatbot with memory, RAG, and evaluation combines:

  • Memory → retains conversation context

  • RAG (Retrieval-Augmented Generation) → fetches relevant knowledge

  • Evaluation pipeline → measures response quality

Why it is used

  • Produces grounded, accurate answers

  • Maintains conversational context

  • Continuously improves via evaluation

  • Production-ready architecture

How it works (step-by-step)

  1. User sends query

  2. Retrieve relevant documents (RAG)

  3. Fetch memory (past conversation)

  4. Build prompt (query + context + memory)

  5. Call LLM

  6. Store interaction in memory

  7. Evaluate response quality

  8. Return final answer

Key components

  • Memory: Sliding window / summary

  • Retriever: Vector search

  • LLM: Response generation

  • Evaluator: LLM-as-judge or rules

  • Orchestrator: Controls flow

Basic implementation (Python)

from sentence_transformers import SentenceTransformer
import numpy as np
from openai import OpenAI

client = OpenAI()

# -------- Embedding + Vector Store -------- #

model = SentenceTransformer('all-MiniLM-L6-v2')

documents = [
    "Semantic search uses embeddings to find meaning.",
    "RAG combines retrieval with generation.",
    "Vector databases store embeddings."
]

doc_embeddings = model.encode(documents)

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

def retrieve(query, top_k=2):
    query_emb = model.encode(query)
    scores = [cosine_similarity(query_emb, emb) for emb in doc_embeddings]
    top_idx = np.argsort(scores)[::-1][:top_k]
    return [documents[i] for i in top_idx]

# -------- Memory -------- #

class Memory:
    def __init__(self, max_turns=3):
        self.history = []
        self.max_turns = max_turns

    def add(self, user, bot):
        self.history.append((user, bot))
        if len(self.history) > self.max_turns:
            self.history.pop(0)

    def get_context(self):
        return "\n".join([f"User: {u}\nBot: {b}" for u, b in self.history])

memory = Memory()

# -------- LLM -------- #

def generate_response(query, context, memory_context):
    prompt = f"""
    Context:
    {context}

    Conversation:
    {memory_context}

    Question: {query}
    Answer:
    """

    response = client.chat.completions.create(
        model="gpt-4.1-mini",
        messages=[{"role": "user", "content": prompt}]
    )

    return response.choices[0].message.content

# -------- Evaluation -------- #

def evaluate(query, answer):
    prompt = f"""
    Evaluate the answer for relevance (1-5):

    Question: {query}
    Answer: {answer}

    Return only a score.
    """

    res = client.chat.completions.create(
        model="gpt-4.1-mini",
        messages=[{"role": "user", "content": prompt}]
    )

    return res.choices[0].message.content

# -------- Chat Pipeline -------- #

def chatbot(query):
    # Step 1: Retrieve
    docs = retrieve(query)
    context = "\n".join(docs)

    # Step 2: Memory
    mem_context = memory.get_context()

    # Step 3: Generate
    answer = generate_response(query, context, mem_context)

    # Step 4: Store memory
    memory.add(query, answer)

    # Step 5: Evaluate
    score = evaluate(query, answer)

    return {
        "answer": answer,
        "evaluation_score": score
    }

# -------- Example -------- #

print(chatbot("What is semantic search?"))
print(chatbot("Explain more"))
